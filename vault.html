What is Vault?

  • A tool for securely managing secrets, encryption, and access
  • Built by HashiCorp — open-source with Enterprise and HCP Cloud tiers
  • Central place to store and tightly control access to tokens, passwords, certificates, API keys
  • Provides dynamic secrets — generate credentials on-demand with automatic revocation
  • Industry standard for zero-trust secret management in cloud-native environments

Why Vault?

Without Vault

  • Secrets in env vars & config files
  • Hard-coded credentials in code
  • Shared passwords across teams
  • No audit trail
  • Manual credential rotation

With Vault

  • Centralized secret storage
  • Dynamic, short-lived credentials
  • Fine-grained access policies
  • Full audit logging
  • Automatic rotation & revocation

Getting Started

# Start Vault in dev mode (NOT for production)
vault server -dev

# Set address and token
export VAULT_ADDR='http://127.0.0.1:8200'
export VAULT_TOKEN='hvs.roottoken...'

# Check status
vault status

# Docker (dev mode)
docker run -d --name vault   -p 8200:8200   -e 'VAULT_DEV_ROOT_TOKEN_ID=myroot'   hashicorp/vault

# Production: use Raft or Consul storage backend
# with TLS, auto-unseal, and HA clustering

KV Secret Engine (v2)

# Enable KV v2 secret engine
vault secrets enable -path=secret kv-v2

# Write a secret
vault kv put secret/myapp/config   db_host="postgres.prod.internal"   db_user="app"   db_pass="s3cur3P@ss!"

# Read a secret
vault kv get secret/myapp/config
vault kv get -field=db_pass secret/myapp/config

# List secrets
vault kv list secret/myapp/

# Version history (KV v2)
vault kv get -version=2 secret/myapp/config

# Delete / undelete / destroy
vault kv delete secret/myapp/config
vault kv undelete -versions=3 secret/myapp/config
vault kv destroy -versions=1 secret/myapp/config

Dynamic Secrets: Databases

# Enable database secret engine
vault secrets enable database

# Configure PostgreSQL connection
vault write database/config/mydb   plugin_name=postgresql-database-plugin   allowed_roles="app-role"   connection_url="postgresql://{{username}}:{{password}}@db:5432/mydb"   username="vault_admin"   password="admin_pass"

# Create a role (generates dynamic credentials)
vault write database/roles/app-role   db_name=mydb   creation_statements="CREATE ROLE "{{name}}" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; GRANT SELECT ON ALL TABLES IN SCHEMA public TO "{{name}}";"   default_ttl="1h"   max_ttl="24h"

# Get dynamic credentials (unique per request!)
vault read database/creds/app-role
# username: v-app-role-abc123
# password: A1b2C3d4E5-random
# ttl: 1h (auto-revoked after expiry)

Dynamic Secrets: AWS

# Enable AWS secret engine
vault secrets enable aws

# Configure root credentials

AppRole Authentication

# Enable AppRole auth
vault auth enable approle

# Create a role
vault write auth/approle/role/my-app   secret_id_ttl=10m   token_ttl=20m   token_max_ttl=30m   token_policies="my-app-policy"

# Get RoleID (embedded in app config)
vault read auth/approle/role/my-app/role-id

# Generate SecretID (delivered securely at deploy time)
vault write -f auth/approle/role/my-app/secret-id

# App authenticates with both
vault write auth/approle/login   role_id="abc-123-..."   secret_id="def-456-..."
# Returns a Vault token the app uses for secret access

Policies (ACL)

# my-app-policy.hcl
path "database/creds/app-role" {
  capabilities = ["read"]
}

path "secret/data/myapp/*" {
  capabilities = ["read", "list"]
}

path "secret/data/other-app/*" {
  capabilities = ["deny"]
}

# Apply: vault policy write my-app-policy my-app-policy.hcl
# List:  vault policy list
# Read:  vault policy read my-app-policy
# Capabilities: create, read, update, delete, list, sudo, deny

PKI — Certificate Authority

# Enable PKI secret engine
vault secrets enable pki
vault secrets tune -max-lease-ttl=87600h pki

# Generate root CA
vault write pki/root/generate/internal   common_name="My Company Root CA"   ttl=87600h

# Create a role for issuing certs
vault write pki/roles/web-server   allowed_domains="example.com"   allow_subdomains=true   max_ttl=720h

# Issue a certificate
vault write pki/issue/web-server   common_name="api.example.com"   ttl=24h
# Returns: certificate, private_key, ca_chain

Automate TLS certificate lifecycle — no more manual renewals

Transit — Encryption as a Service

# Enable transit secret engine
vault secrets enable transit

# Create an encryption key
vault write -f transit/keys/my-app-key

# Encrypt data (base64 input)
vault write transit/encrypt/my-app-key   plaintext=$(echo "secret data" | base64)
# ciphertext: vault:v1:abc123...

# Decrypt data
vault write transit/decrypt/my-app-key   ciphertext="vault:v1:abc123..."
# plaintext: c2VjcmV0IGRhdGE=  (base64 decode -> "secret data")

# Key rotation (old data still decryptable)
vault write -f transit/keys/my-app-key/rotate
  • App never sees the encryption key — Vault handles all crypto
  • Supports AES-GCM, ChaCha20, RSA, ECDSA, Ed25519

Seal / Unseal & Auto-Unseal

  • Seal — Vault encrypts its master key; cannot read data when sealed
  • Shamir's Secret Sharing — master key split into N shares, requires threshold K to unseal
  • Auto-Unseal — use a cloud KMS to automatically unseal on restart
# Initialize Vault (first time)
vault operator init -key-shares=5 -key-threshold=3

# Manual unseal (need 3 of 5 keys)
vault operator unseal 
vault operator unseal 
vault operator unseal 

# Auto-unseal config (e.g., AWS KMS)
seal "awskms" {
  region     = "us-east-1"
  kms_key_id = "alias/vault-unseal"
}

Vault in CI/CD

# GitHub Actions with Vault
- name: Import Secrets from Vault
  uses: hashicorp/vault-action@v3
  with:
    url: https://vault.mycompany.com
    method: approle
    roleId: ${{ secrets.VAULT_ROLE_ID }}
    secretId: ${{ secrets.VAULT_SECRET_ID }}
    secrets: |
      secret/data/myapp/prod db_password | DB_PASSWORD ;
      secret/data/myapp/prod api_key | API_KEY

- name: Deploy
  env:
    DB_PASSWORD: ${{ steps.vault.outputs.DB_PASSWORD }}
  run: ./deploy.sh

Never store secrets in CI/CD variables — fetch them at runtime from Vault

Best Practices

1 Use dynamic secrets over static — short TTLs, auto-revoked
2 Least privilege policies — apps get only what they need
3 Enable audit logging on all clusters
4 Use auto-unseal with cloud KMS for production
5 Namespace secrets by team/app: secret/team/app/env
6 Rotate root tokens and store unseal keys securely
7 Use AppRole or K8s auth for machines, OIDC for humans
8 Run a 3 or 5 node Raft cluster for HA in production
