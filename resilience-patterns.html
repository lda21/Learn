Resilience Patterns

Design for failure, not just prevention

ResilienceCircuit BreakerFault Tolerance

Why Resilience?

Distributed Systems Fail

  • Networks are unreliable — packets drop, latency spikes
  • Services crash, restart, deploy mid-request
  • Databases hit connection limits under load
  • Third-party APIs have outages (exchange downtime)
  • Hardware fails — disks, NICs, entire AZs

Design for Failure

  • Accept failure as normal, not exceptional
  • Contain blast radius — one failure should not cascade
  • Degrade gracefully — partial function beats total outage
  • Recover automatically — self-healing over manual intervention
  • Fail fast — don't waste resources on doomed requests

In a crypto trading agent, a 5-second exchange outage should not take down the entire system.

Circuit Breaker Implementation

// Per-exchange circuit breaker in the AI Trading Agent
// Each exchange (Binance, Kraken, Coinbase) has its own breaker

public class ExchangeCircuitBreaker {
    private CircuitState _state = CircuitState.Closed;
    private int _failureCount = 0;
    private DateTime _lastFailure;
    private readonly int _threshold = 5;        // failures before opening
    private readonly TimeSpan _openDuration = TimeSpan.FromSeconds(60);

    public async Task<T> ExecuteAsync<T>(Func<Task<T>> action) {
        if (_state == CircuitState.Open) {
            if (DateTime.UtcNow - _lastFailure > _openDuration)
                _state = CircuitState.HalfOpen;  // allow test request
            else
                throw new CircuitOpenException("Exchange circuit open");
        }
        try {
            var result = await action();
            OnSuccess();
            return result;
        } catch (Exception ex) {
            OnFailure();
            throw;
        }
    }
}

// Global breaker wraps all exchanges — trips if ALL exchanges fail
// Per-exchange breaker isolates individual exchange failures

Retry Pattern

Exponential Backoff + Jitter

  • Base delay doubles each attempt: 1s, 2s, 4s, 8s
  • Jitter adds randomness to prevent thundering herd
  • Max retries caps total attempts (e.g. 3-5)
  • Max delay caps backoff ceiling (e.g. 30s)
  • Only retry transient errors (5xx, timeout, network)
  • Never retry permanent errors (400, 401, 403)

Idempotency Requirement

// Exponential backoff formula
delay = min(base * 2^attempt + random(0, jitter), maxDelay)
// Attempt 1: min(1s * 2^0 + rand(0, 500ms), 30s) = ~1.3s
// Attempt 2: min(1s * 2^1 + rand(0, 500ms), 30s) = ~2.4s
// Attempt 3: min(1s * 2^2 + rand(0, 500ms), 30s) = ~4.2s

Timeout Pattern

Timeout Types

  • Connection timeout — how long to establish connection (1-5s)
  • Read timeout — how long to wait for response (5-30s)
  • gRPC deadline — end-to-end call budget propagated through chain
  • Cancellation tokens — cooperative cancellation in .NET/Go

Bulkhead Pattern

Isolation Strategy

  • Named after ship bulkheads — compartmentalize failures
  • Separate thread/connection pools per dependency
  • One slow service cannot exhaust all resources
  • Limit concurrent calls to each downstream service
  • Reject excess requests immediately (fail fast)
// Bulkhead: max 10 concurrent calls per exchange
var binanceBulkhead = Policy
  .BulkheadAsync(maxParallelization: 10,
                  maxQueuingActions: 20);

var krakenBulkhead = Policy
  .BulkheadAsync(maxParallelization: 10,
                  maxQueuingActions: 20);

Without vs With Bulkhead

Without: Shared connection pool

  • Binance slow → exhausts all connections
  • Kraken requests queue behind Binance
  • Coinbase also blocked — cascade failure

With: Isolated pools

  • Binance slow → only its pool saturated
  • Kraken and Coinbase unaffected
  • Trading continues on healthy exchanges

Per-Exchange Breakers

Each exchange has its own circuit breaker. Binance down? Kraken and Coinbase continue trading normally.

Global Breaker

If all exchanges trip, global breaker halts all new orders and triggers emergency position flattening.

Graceful Degradation

Single exchange down → redistribute to others. All feeds stale → progressive position reduction.

Data Freshness Signal

Market data age is a resilience signal. Stale feed (>2min) triggers automatic position reduction.

Fail-Safe Defaults

When in doubt, choose the safest possible action. In trading, safety means reducing risk exposure.

Feed Stale → Flatten

If market data is stale (no updates for >2min), begin flattening positions. Stale prices mean unknown risk.

Service Down → No New Orders

If risk service, pricing engine, or order router is down, reject all new orders. Never trade blind.

Unknown State → Halt

If position state is uncertain (reconciliation failure), halt trading and alert. Don't guess positions.

Error → Reject Order

Any unhandled exception in the order pipeline defaults to rejection. Better to miss a trade than to place an unvalidated one.

// Fail-safe default: reject order if ANY validation step fails or times out
public async Task<OrderResult> ProcessOrder(Order order) {
    try {
        await ValidateRisk(order);       // 50ms timeout
        await CheckExchangeHealth();     // circuit breaker check
        await VerifyDataFreshness();     // stale data check
        return await SubmitToExchange(order);
    } catch (Exception) {
        return OrderResult.Rejected("Fail-safe: validation incomplete");
    }
}

Graceful Degradation

Partial Failure Handling

  • 1 of 3 exchanges down → continue on remaining 2
  • Risk service slow → use cached risk limits (with tighter bounds)
  • Analytics pipeline down → trade without ML signals, use rule-based only
  • Monitoring down → continue but reduce position sizes

var staleness = DateTime.UtcNow - lastMarketDataUpdate;

if (staleness > TimeSpan.FromMinutes(5))
    await FlattenAllPositions();              // 100% reduction
else if (staleness > TimeSpan.FromMinutes(2))
    await ReducePositions(targetPercent: 50); // 50% reduction
else if (staleness > TimeSpan.FromSeconds(30))
    await HaltExchange(staleExchangeId);      // isolate stale exchange
// else: normal operations

Rate Limiting & Throttling

Exchange API Rate Limits

  • Binance: 1200 req/min (weight-based)
  • Kraken: 15 req/s with call counter decay
  • Coinbase: 10 req/s per endpoint
  • Exceeding limits → IP ban (temporary or permanent)
  • Must track rate limit headers in every response

Token Bucket Algorithm

// Token bucket rate limiter
class TokenBucket {
  capacity: 1200,    // max tokens
  refillRate: 20,    // tokens per second
  tokens: 1200,      // current tokens

  async acquire(weight = 1) {
    this.refill();
    if (this.tokens >= weight) {
      this.tokens -= weight;
      return true;  // request allowed
    }
    // Queue or reject
    await this.waitForTokens(weight);
  }
}

// Each exchange has its own bucket
const binanceLimiter = new TokenBucket(1200, 20);
const krakenLimiter  = new TokenBucket(15, 15);

Critical: Rate limiting is both a resilience pattern (protect yourself) and a compliance requirement (respect exchange limits).

Fallback Strategies

Cache Fallback

Primary data source down? Serve from cache. Mark response as stale. In trading: use last known price with wider spreads.

Default Values

Config service unreachable? Use hardcoded safe defaults. Risk limits default to most conservative settings.

Degraded Mode

ML model service down? Fall back to rule-based signals. Less optimal but still functional trading.

Alternative Provider

Primary exchange API down? Route orders to backup exchange. Different liquidity but maintains availability.

// Fallback chain with Polly
var policy = Policy<Price>
    .Handle<Exception>()
    .FallbackAsync(async (ct) => {
        // Fallback 1: Try cached price
        var cached = await cache.GetAsync<Price>(symbol);
        if (cached != null && cached.Age < TimeSpan.FromMinutes(1))
            return cached.WithFlag("stale");

        // Fallback 2: Use last known price with safety margin
        var lastKnown = positionStore.GetLastPrice(symbol);
        return lastKnown.WithSpread(multiplier: 2.0).WithFlag("degraded");
    });

Saga Pattern

Manage distributed transactions across services with compensating actions instead of distributed locks.

Order Lifecycle Saga

  • Step 1: Reserve margin in risk service
  • Step 2: Submit order to exchange
  • Step 3: Update position tracking
  • Step 4: Record in trade ledger

Compensation (Rollback)

  • Step 3 fails: Cancel order on exchange, release margin
  • Step 2 fails: Release reserved margin
  • Each step has a defined compensating action
  • Compensation must be idempotent

Outbox Pattern

The Problem: Dual Writes

  • Save trade to DB and publish event to message bus
  • DB write succeeds, event publish fails → data inconsistency
  • Event published, DB write fails → phantom events
  • Two-phase commit is slow and fragile

The Solution: Outbox

  • Write trade and event to same DB transaction
  • Background worker polls outbox table, publishes events
  • Guarantees at-least-once delivery
  • Consumers must handle duplicates (idempotency)
// Single transaction: trade + outbox event
await using var tx = await db.BeginTransactionAsync();

// 1. Write trade record
await db.Trades.AddAsync(new Trade { Symbol = "BTC/USDT", Qty = 0.5, Price = 42000 });

// 2. Write event to outbox (same transaction!)
await db.Outbox.AddAsync(new OutboxEvent {
    Type = "TradeExecuted",
    Payload = JsonSerializer.Serialize(tradeEvent),
    CreatedAt = DateTime.UtcNow,
    Published = false
});

await tx.CommitAsync();  // atomic: both succeed or both fail

// Background worker: poll outbox, publish to Kafka/RabbitMQ, mark published

Idempotency

Why Idempotency Matters

  • Retries can deliver the same request multiple times
  • Network timeout ≠ failure (request may have succeeded)
  • Message queues deliver at-least-once, not exactly-once
  • Without idempotency: duplicate orders, double charges
  • Idempotent: same request N times = same result as 1 time

Implementation Strategies

  • Idempotency keys — client-generated unique ID per operation
  • Database constraints — unique index on (order_id, exchange)
  • Conditional writes — "update WHERE version = X"
  • Natural idempotency — GET, DELETE (by ID) are inherently idempotent
// Idempotency key for exchange orders
public async Task<OrderResult> PlaceOrder(Order order) {
    var idempotencyKey = $"{order.ClientOrderId}_{order.Exchange}_{order.Symbol}";

    // Check if already processed
    var existing = await db.Orders.FindByKeyAsync(idempotencyKey);
    if (existing != null)
        return existing.Result;  // return cached result, don't re-submit

    // First time: submit to exchange
    var result = await exchange.SubmitAsync(order, idempotencyKey);
    await db.Orders.SaveWithKeyAsync(idempotencyKey, result);
    return result;
}

Chaos Engineering

Fault Injection

  • Network faults — inject latency, packet loss, DNS failures
  • Service faults — kill pods, return 500s, corrupt responses
  • Resource faults — exhaust CPU, memory, disk, file descriptors
  • Clock skew — shift system time to test time-dependent logic
# Chaos Mesh: inject 3s delay to exchange API
apiVersion: chaos-mesh.org/v1alpha1
kind: NetworkChaos
spec:
  action: delay
  delay:
    latency: "3000ms"
  selector:
    labelSelectors:
      app: exchange-gateway

Trading Agent Drills

  • Circuit breaker test: force-open a per-exchange breaker, verify graceful degradation
  • Kill switch drill: trigger emergency halt, verify all positions flatten within SLA
  • Stale feed test: freeze market data, verify progressive position reduction kicks in
  • Exchange outage: block traffic to one exchange, verify orders route to alternatives
  • Full chaos day: multiple simultaneous faults, verify system degrades (not crashes)

Tools: Chaos Mesh, Litmus, Gremlin, Toxiproxy

Best Practices

1. Apply circuit breakers to every external dependency, not just HTTP calls
2. Always add jitter to retry backoff to prevent thundering herd
3. Set timeouts on every outbound call — no call should wait forever
4. Use bulkheads to isolate failure domains (per-exchange, per-service)
5. Design fail-safe defaults: when uncertain, choose the safest action
6. Make all retryable operations idempotent with unique keys
7. Monitor resilience metrics (breaker state, retry rate, timeout %) in dashboards
8. Practice chaos engineering regularly — test failure modes before production does
9. Use data freshness as a first-class resilience signal, not just an afterthought
10. Combine patterns: bulkhead + circuit breaker + retry + timeout as a resilience pipeline
