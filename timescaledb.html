What Is TimescaleDB?

PostgreSQL Extension

  • Runs as an extension inside PostgreSQL
  • Full SQL compatibility — joins, CTEs, subqueries
  • All PostgreSQL tools work: psql, pg_dump, ORMs
  • ACID-compliant, row-level security, roles
  • No new query language to learn

Time-Series Optimized

  • Automatic time-based partitioning (hypertables)
  • 10-100x faster inserts vs vanilla PostgreSQL
  • Columnar compression (90%+ space savings)
  • Continuous aggregates for real-time rollups
  • Built-in retention & data lifecycle policies

Hypertables

Regular Table
➡
create_hypertable()
➡
Auto-Partitioned Chunks
-- Convert a regular table into a hypertable
-- Automatically partitions by time into "chunks"
CREATE TABLE ticks (
    time        TIMESTAMPTZ NOT NULL,
    symbol      TEXT        NOT NULL,
    price       DOUBLE PRECISION,
    volume      DOUBLE PRECISION
);

SELECT create_hypertable('ticks', 'time');
-- Chunks: 7-day intervals by default

-- Each chunk is a separate PostgreSQL table under the hood
-- Queries automatically route to relevant chunks
-- Old chunks can be compressed, moved, or dropped independently

-- Custom chunk interval (1 day for high-frequency data)
SELECT create_hypertable('ticks', 'time',
    chunk_time_interval => INTERVAL '1 day');

Streaming Pipeline

# Python: Kafka consumer -> TimescaleDB
import psycopg2
from kafka import KafkaConsumer

consumer = KafkaConsumer('trades')
conn = psycopg2.connect(DSN)
cur = conn.cursor()

batch = []
for msg in consumer:
    trade = json.loads(msg.value)
    batch.append((
        trade['time'],
        trade['exchange'],
        trade['symbol'],
        trade['price'],
        trade['volume'],
        trade['side']
    ))
    if len(batch) >= 1000:
        cur.executemany(INSERT_SQL, batch)
        conn.commit()
        batch.clear()

Time-Series Queries

-- time_bucket: group data into fixed intervals
SELECT
    time_bucket('5 minutes', time) AS bucket,
    symbol,
    first(price, time)  AS open,
    max(price)           AS high,
    min(price)           AS low,
    last(price, time)    AS close,
    sum(volume)          AS volume
FROM ticks
WHERE symbol = 'BTC/USDT'
  AND time > NOW() - INTERVAL '24 hours'
GROUP BY bucket, symbol
ORDER BY bucket DESC;

-- Gap filling: fill missing intervals with previous values
SELECT
    time_bucket_gapfill('1 hour', time) AS bucket,
    symbol,
    locf(avg(price)) AS avg_price  -- Last Observation Carried Forward
FROM ticks
WHERE time BETWEEN '2025-01-01' AND '2025-01-02'
  AND symbol = 'BTC/USDT'
GROUP BY bucket, symbol;

-- Interpolation: linear fill between known values
SELECT
    time_bucket_gapfill('1 hour', time) AS bucket,
    interpolate(avg(price)) AS interpolated_price
FROM ticks
WHERE time BETWEEN '2025-01-01' AND '2025-01-02'
GROUP BY bucket;

Continuous Aggregates

Materialized views that auto-refresh as new data arrives. Build candle hierarchy from raw ticks.

1m Ticks
➡
5m Candles
➡
1h Candles
➡
1d Candles
-- 5-minute candles from raw ticks (continuous aggregate)
CREATE MATERIALIZED VIEW candles_5m
WITH (timescaledb.continuous) AS
SELECT
    time_bucket('5 minutes', time) AS bucket,
    symbol,
    first(price, time)  AS open,
    max(price)           AS high,
    min(price)           AS low,
    last(price, time)    AS close,
    sum(volume)          AS volume
FROM ticks
GROUP BY bucket, symbol
WITH NO DATA;

-- 1-hour candles from 5-minute candles (cascading)
CREATE MATERIALIZED VIEW candles_1h
WITH (timescaledb.continuous) AS
SELECT
    time_bucket('1 hour', bucket) AS bucket,
    symbol,
    first(open, bucket)  AS open,
    max(high)             AS high,
    min(low)              AS low,
    last(close, bucket)   AS close,
    sum(volume)           AS volume
FROM candles_5m
GROUP BY time_bucket('1 hour', bucket), symbol
WITH NO DATA;

Real-Time Aggregation

  • Real-time mode merges materialized data with live un-materialized data
  • Queries always return up-to-date results
  • Small performance cost for latest bucket
  • Enabled by default in TimescaleDB 2.0+
-- Check real-time status
SELECT view_name, materialized_only
FROM timescaledb_information
    .continuous_aggregates;

-- Toggle real-time off for faster reads
ALTER MATERIALIZED VIEW candles_5m
SET (timescaledb.materialized_only = true);

Compression

Columnar compression converts row-based chunks into compressed columnar format. 90-95% space savings typical.

-- Enable compression on ticks table
ALTER TABLE ticks SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'symbol',       -- group by symbol
    timescaledb.compress_orderby = 'time DESC'        -- order within segment
);

-- Compress chunks older than 7 days automatically
SELECT add_compression_policy('ticks', INTERVAL '7 days');

-- Compress chunks older than 30 days for candles
ALTER TABLE candles SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'symbol, interval',
    timescaledb.compress_orderby = 'time DESC'
);
SELECT add_compression_policy('candles', INTERVAL '30 days');

-- Manual compression of specific chunk
SELECT compress_chunk('_timescaledb_internal._hyper_1_42_chunk');

-- Check compression stats
SELECT
    hypertable_name,
    pg_size_pretty(before_compression_total_bytes) AS before,
    pg_size_pretty(after_compression_total_bytes)  AS after,
    round((1 - after_compression_total_bytes::numeric /
           before_compression_total_bytes) * 100, 1) AS savings_pct
FROM hypertable_compression_stats('ticks');

Data Lifecycle

Hot: Raw ticks (0-7 days) — Uncompressed
➡
Warm: Compressed ticks (7d-2y) — 90% smaller
➡
Cold: Dropped (>2y) — Aggregates retained

Strategy: Keep raw data short-term, aggregates long-term. Raw ticks compress from 500GB to 50GB.

Indexing Strategies

-- TimescaleDB auto-creates a time index on the hypertable column
-- But composite indexes are essential for trading queries

-- Composite index: most queries filter by symbol + time range
CREATE INDEX idx_ticks_symbol_time ON ticks (symbol, time DESC);
CREATE INDEX idx_candles_symbol_interval_time
    ON candles (symbol, interval, time DESC);
CREATE INDEX idx_features_symbol_time ON features (symbol, time DESC);

-- BRIN index: excellent for time-series (tiny size, fast range scans)
-- TimescaleDB chunks are inherently time-ordered, making BRIN ideal
CREATE INDEX idx_ticks_time_brin ON ticks
    USING BRIN (time) WITH (pages_per_range = 32);

-- Funding rates: query by exchange + symbol
CREATE INDEX idx_funding_exchange_symbol_time
    ON funding_rates (exchange, symbol, time DESC);

B-tree (Default)

  • Best for equality + range queries
  • Larger index size
  • Use for: WHERE symbol = 'X'

BRIN (Block Range)

  • 1000x smaller than B-tree
  • Perfect for time columns
  • Use for: WHERE time > X

Advanced Queries

-- ASOF JOIN: match each tick to the nearest funding rate
SELECT t.time, t.symbol, t.price,
       f.rate AS funding_rate
FROM ticks t
ASOF JOIN funding_rates f
    ON t.symbol = f.symbol AND t.time >= f.time
WHERE t.symbol = 'BTC/USDT'
  AND t.time > NOW() - INTERVAL '1 day';

-- Window functions: rolling VWAP (Volume-Weighted Avg Price)
SELECT time, symbol, price,
    SUM(price * volume) OVER w / SUM(volume) OVER w AS vwap_20
FROM ticks
WHERE symbol = 'BTC/USDT'
WINDOW w AS (
    PARTITION BY symbol ORDER BY time
    ROWS BETWEEN 19 PRECEDING AND CURRENT ROW
);

-- Percentile: P50/P95/P99 price distribution per hour
SELECT
    time_bucket('1 hour', time) AS bucket,
    symbol,
    percentile_cont(0.50) WITHIN GROUP (ORDER BY price) AS p50,
    percentile_cont(0.95) WITHIN GROUP (ORDER BY price) AS p95,
    percentile_cont(0.99) WITHIN GROUP (ORDER BY price) AS p99
FROM ticks
WHERE time > NOW() - INTERVAL '7 days'
GROUP BY bucket, symbol
ORDER BY bucket DESC;

Performance Tuning

chunk_time_interval

Match to your query patterns. 1 day for high-frequency ticks, 7 days for candles.

chunk_time_interval => '1 day'

Memory Settings

Set shared_buffers to 25% of RAM. work_mem to 256MB+ for aggregations.

shared_buffers = 8GB

Parallel Queries

TimescaleDB parallelizes across chunks. Set max_parallel_workers_per_gather to CPU cores / 2.

max_parallel_workers_per_gather = 8

Chunk Exclusion

Always include WHERE time > X in queries. TimescaleDB skips irrelevant chunks entirely.

WHERE time > NOW() - '7 days'

TimescaleDB + PostgreSQL Features

JSONB & CTEs

-- Store metadata as JSONB
ALTER TABLE ticks ADD COLUMN
    meta JSONB DEFAULT '{}';

-- Query JSON fields
SELECT * FROM ticks
WHERE meta->>'source' = 'websocket';

-- CTE: compute features in stages
WITH recent AS (
    SELECT * FROM candles_1h
    WHERE symbol = 'BTC/USDT'
      AND time > NOW() - '30 days'
),
with_sma AS (
    SELECT *,
      AVG(close) OVER (ORDER BY time
        ROWS 19 PRECEDING) AS sma_20
    FROM recent
)
SELECT * FROM with_sma
WHERE close > sma_20;

Triggers & Foreign Keys

-- Regular table for instruments
CREATE TABLE instruments (
    symbol TEXT PRIMARY KEY,
    name   TEXT,
    type   TEXT  -- 'spot', 'perp'
);

-- Foreign key from hypertable
-- (on segmentby column is efficient)
ALTER TABLE ticks
ADD CONSTRAINT fk_symbol
FOREIGN KEY (symbol)
REFERENCES instruments(symbol);

-- Trigger: auto-compute features
CREATE OR REPLACE FUNCTION
  compute_features() RETURNS trigger AS $$
BEGIN
  -- Insert computed features
  INSERT INTO features (time, symbol)
  VALUES (NEW.time, NEW.symbol);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

Monitoring & Maintenance

-- Hypertable disk usage
SELECT hypertable_name,
       pg_size_pretty(hypertable_size(format('%I.%I',
           hypertable_schema, hypertable_name)::regclass)) AS total_size,
       num_chunks
FROM timescaledb_information.hypertables;

-- Compression ratios per hypertable
SELECT hypertable_name,
       pg_size_pretty(before_compression_total_bytes) AS before,
       pg_size_pretty(after_compression_total_bytes)  AS after,

-- Continuous aggregate job status
SELECT application_name, schedule_interval,
       last_run_status, last_run_duration, next_start

SELECT chunk_name,
       pg_size_pretty(total_bytes) AS size,

Best Practices

1. Always include a time predicate in queries for chunk exclusion
2. Set chunk_time_interval based on query range (1d for ticks, 7d for candles)
3. Use continuous aggregates instead of cron jobs for rollups
4. Enable compression on all hypertables after hot window passes
5. Set retention policies to auto-drop old raw data while keeping aggregates
6. Use COPY or multi-row INSERT for bulk loading, never single-row loops
7. Create composite indexes on (segmentby_col, time DESC) for filtered queries
8. Monitor chunk sizes and compression ratios weekly
9. Use real-time aggregation for dashboards, materialized-only for batch jobs
10. Test query plans with EXPLAIN ANALYZE to verify chunk exclusion works
