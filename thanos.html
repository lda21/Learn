The Problem: Prometheus at Scale

Local Retention Limits

  • Prometheus stores data on local disk only
  • Typical retention: 2 weeks (disk cost grows fast)
  • No native long-term storage support
  • Losing months of historical data for capacity planning
  • Disk failure = total metric loss

Single Instance Pain

  • No global query view across clusters
  • HA pairs produce duplicate data
  • Each Prometheus is an isolated silo
  • Federation is fragile and limited
  • Cannot correlate metrics across regions

What Is Thanos?

  • Highly Available Prometheus with unlimited metric retention
  • CNCF Incubating project — production-grade, battle-tested
  • Seamlessly extends Prometheus without replacing it
  • Uses cheap object storage (S3, GCS, Azure Blob) as backend
  • Global query view across all Prometheus instances
  • Automatic deduplication of HA replica data
  • Downsampling for cost-efficient long-term queries
  • 100% PromQL compatible — Grafana works with zero changes

Key idea: Prometheus handles the last 2 weeks. Thanos handles the rest — months or years.

Thanos Sidecar

What It Does

  • Runs alongside each Prometheus instance
  • Uploads completed TSDB blocks to object storage
  • Proxies real-time queries from Thanos Query
  • Exposes StoreAPI gRPC for Thanos Query to discover
  • Zero impact on Prometheus performance

How Blocks Flow

  • Prometheus writes 2-hour TSDB blocks to disk
  • Sidecar detects completed blocks
  • Uploads to S3 bucket with external labels
  • Labels identify cluster, replica, environment
  • Prometheus can now safely delete old blocks
# Sidecar container in Prometheus Pod
- name: thanos-sidecar
  image: quay.io/thanos/thanos:v0.37.2
  args:
    - sidecar
    - --tsdb.path=/prometheus
    - --prometheus.url=http://localhost:9090
    - --objstore.config-file=/etc/thanos/bucket.yaml
    - --grpc-address=0.0.0.0:10901

Thanos Store Gateway

Purpose

  • Serves historical metric data from object storage
  • Acts as a "virtual Prometheus" for old data
  • Implements the same StoreAPI gRPC interface
  • Thanos Query treats it like any other data source
  • Stateless — can be scaled horizontally

Performance

  • Caches block metadata and index in memory
  • Lazy-loads only requested chunks from S3
  • Supports bucket caching with memcached/Redis
  • Time-based partitioning for sharding across replicas
  • Queries 1-year data as fast as querying Prometheus
# Thanos Store Gateway deployment
args:
  - store
  - --data-dir=/var/thanos/store
  - --objstore.config-file=/etc/thanos/bucket.yaml
  - --index-cache-size=1GB
  - --chunk-pool-size=4GB
  - --grpc-address=0.0.0.0:10901

Thanos Query

  • Global PromQL endpoint — single pane of glass for all metrics
  • Fans out queries to all Sidecars and Store Gateways
  • Automatic deduplication of HA Prometheus replicas
  • Merges results transparently — Grafana sees one unified source
  • Supports partial response mode for degraded environments
  • Horizontal scaling: add more Query replicas behind a load balancer

Real-World: Trading Agent Metrics

The AI Crypto Trading Agent uses Thanos for 1-year metric retention.

Prometheus (Hot)

Retains last 2 weeks on local SSD. Sub-second queries for live dashboards, alerts, and trading signals.

--storage.tsdb.retention.time=14d

Thanos S3 (Warm)

Extends retention to 1 year on S3. Historical analysis of PnL trends, model accuracy, market patterns.

--retention.resolution-raw=365d

Key Metrics Stored

Trade execution latency, order fill rates, portfolio value, strategy signals, exchange API errors, model inference time.

Why 1 Year?

Crypto markets are cyclical. Comparing current strategy performance against last year's bull/bear data is essential for backtesting.

Object Storage Configuration

Supported Backends

  • AWS S3 — most common, cheapest at scale
  • Google GCS — native for GKE clusters
  • Azure Blob — for Azure environments
  • MinIO — S3-compatible self-hosted
  • OpenStack Swift, Tencent COS, Alibaba OSS

Trading Agent Setup

  • S3 bucket: trading-agent-thanos
  • Region: us-east-1 (same as EKS)
  • IRSA for pod-level IAM authentication
  • Server-side encryption (SSE-S3)
  • Versioning disabled (Thanos manages blocks)

Thanos Compactor

What It Does

  • Merges small 2h blocks into larger ones
  • Deduplicates HA replica data
  • Downsamples old data (5m, 1h aggregates)
  • Applies retention policies per resolution
  • Runs as a singleton — only one instance allowed

Compaction Levels

  • Level 1: 2h blocks (raw from Prometheus)
  • Level 2: merged into larger blocks
  • Level 3: further compacted for efficiency
  • Reduces S3 object count by 10-50x
  • Faster queries on compacted data
# Thanos Compactor configuration
args:
  - compact
  - --data-dir=/var/thanos/compact
  - --objstore.config-file=/etc/thanos/bucket.yaml
  - --retention.resolution-raw=365d       # Keep raw data 1 year
  - --retention.resolution-5m=365d        # Keep 5m downsampled 1 year
  - --retention.resolution-1h=365d        # Keep 1h downsampled 1 year
  - --compact.concurrency=4
  - --downsample.concurrency=4
  - --wait                                # Run continuously

Downsampling

Thanos automatically creates lower-resolution aggregates for older data, dramatically reducing storage and query time.

ResolutionCreated AfterAggregates StoredUse Case
RawImmediatelyAll original samplesLast 2 weeks, real-time alerting
5 minutes40 hoursmin, max, sum, count, counterLast few months, daily dashboards
1 hour10 daysmin, max, sum, count, counter6-12 month trends, capacity planning

Storage Savings

  • Raw: ~15 second scrape interval
  • 5m: 20x fewer samples than raw
  • 1h: 240x fewer samples than raw
  • 1 year of trading metrics: ~50GB (vs ~2TB raw)

Auto-Downsampling in Queries

  • Thanos Query selects resolution automatically
  • Short ranges use raw data
  • Long ranges use 5m or 1h data
  • --query.auto-downsampling enables this

Thanos Ruler

Purpose

  • Evaluates recording rules against global view
  • Alerts based on cross-cluster aggregated data
  • Uploads rule results as TSDB blocks to S3
  • Works like Prometheus ruler but queries Thanos Query
  • Essential for multi-cluster alerting

Trading Agent Rules

  • Alert: portfolio drawdown exceeds 5% in 24h
  • Alert: trade execution latency p99 > 500ms
  • Record: daily exchange error rates

PromQL Across Time

Thanos Query is a drop-in Prometheus replacement. All PromQL works seamlessly across local and historical data.

# Query last 30 days of trade execution latency (uses raw + 5m downsampled)
histogram_quantile(0.99,
  sum(rate(trade_execution_duration_seconds_bucket[30d])) by (le, strategy)
)

# Compare this week's PnL vs same week last year (uses 1h downsampled)
sum(increase(portfolio_pnl_total[7d]))
  -
sum(increase(portfolio_pnl_total[7d] offset 365d))

# Average order fill rate over 6 months (1h resolution, fast query)
avg_over_time(order_fill_rate[180d])

# Model accuracy trend over 1 year (auto-downsampled to 1h)
avg_over_time(model_prediction_accuracy[365d])
  • Grafana datasource: just point to Thanos Query :9090 instead of Prometheus
  • Auto-downsampling selects the right resolution for the time range
  • offset 365d queries seamlessly pull from S3-backed historical data

How It Works

  • Each cluster runs Prometheus + Thanos Sidecar
  • All sidecars upload to the same S3 bucket
  • External labels distinguish clusters: cluster="prod-us"
  • One central Thanos Query fans out to all sidecars
  • Store Gateway serves historical data from any cluster

Trading Agent Example

  • Prod cluster: live trading, order execution metrics
  • Staging cluster: backtesting, model training metrics
  • Single dashboard compares prod vs staging performance
  • Alerts fire on aggregated cross-cluster data
  • One Grafana instance, one Thanos Query endpoint

Monitoring Thanos Itself

Block Upload Success

Sidecar must upload every 2h block. Alert on missing uploads.

thanos_shipper_upload_failures_total

Compaction Lag

If compactor falls behind, queries slow down and storage bloats.

thanos_compact_group_compactions_failures_total

Query Latency

Track p99 query duration. Should stay under 10s for dashboards.

http_request_duration_seconds_bucket{handler="query"}

Store Gateway Cache Hit

Low cache hit ratio means excessive S3 API calls and higher cost.

thanos_store_index_cache_hits_total 
# Alert: Sidecar upload failures
- alert: ThanosSidecarUploadFailures
  expr: rate(thanos_shipper_upload_failures_total[5m]) > 0
  for: 15m
  labels:
    severity: critical
  annotations:
    summary: "Thanos Sidecar failing to upload blocks to S3"

# Alert: Compactor not running
- alert: ThanosCompactHalted
  expr: thanos_compact_halted == 1
  for: 5m
  labels:
    severity: critical

Best Practices

1. Always set external labels on Prometheus (cluster, replica, env)
2. Use IRSA/Workload Identity for S3 auth — never static credentials
3. Run Compactor as a singleton with persistent volume (50GB+)
4. Enable auto-downsampling on Thanos Query for long-range queries
5. Set Prometheus retention to 2 weeks — let Thanos handle the rest
6. Monitor Thanos with Thanos — use dedicated alerts for upload/compaction failures
7. Use memcached for Store Gateway index and chunk caching
8. Separate S3 buckets per environment (prod, staging, dev)
9. Enable S3 server-side encryption and bucket versioning policies
10. Test disaster recovery: delete Store Gateway cache and verify it rebuilds
