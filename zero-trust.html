Zero Trust Security

Never Trust, Always Verify

Use ← → arrow keys or buttons to navigate

What is Zero Trust?

  • A security model that assumes no implicit trust — inside or outside the network
  • Every request must be authenticated, authorized, and encrypted before access is granted
  • Coined by Forrester Research (2010), popularized by Google's BeyondCorp (2014)
  • Replaces the traditional castle-and-moat (perimeter-based) security model
  • Now mandated by NIST SP 800-207 and U.S. Executive Order 14028

Traditional vs Zero Trust

Perimeter Security (Old)

  • Trust everything inside the firewall
  • VPN = full network access
  • Flat internal network
  • Once inside, move laterally freely
  • Hard exterior, soft interior

Zero Trust (Modern)

  • Trust nothing, verify everything
  • Identity-based access per resource
  • Micro-segmented network
  • Continuous verification
  • Assume breach at all times

Core Principles

Verify Explicitly

Authenticate and authorize every request based on all available data points

Least Privilege Access

Grant minimum permissions needed, just-in-time, just-enough-access

Assume Breach

Minimize blast radius with segmentation, encryption, and continuous monitoring

Continuous Validation

Don't trust a session forever — re-evaluate risk signals continuously

Identity & Access Management

  • MFA everywhere — passwords alone are not enough
  • SSO + OIDC/SAML — centralized identity via Okta, Azure AD, Google Workspace
  • Conditional access — evaluate risk signals before granting access
  • Passwordless — FIDO2 / passkeys / biometrics
# Conditional access decision inputs:
User Identity      -> Who are you? (verified via MFA)
Device Posture     -> Is your device compliant? (patched, encrypted)
Location / IP      -> Where are you connecting from?
Risk Score         -> Any anomalous behavior detected?
Resource Sensitivity -> How sensitive is this resource?

# Decision: Allow / Deny / Step-up MFA / Read-only

Device Trust

  • Device inventory — know every device on your network
  • Health attestation — OS patched? Disk encrypted? Firewall on?
  • MDM / EDR — Mobile Device Management + Endpoint Detection & Response
  • Device certificates — mTLS to prove device identity
  • Posture checks at every access request, not just enrollment

Network: Micro-Segmentation

  • Micro-segmentation — isolate workloads so a breach in one doesn't spread
  • East-west traffic inspection — don't just guard the perimeter
  • Software-defined perimeter (SDP) — resources invisible until authenticated
  • mTLS — mutual TLS between all services (service mesh)
# Traditional flat network:
[App A] <---> [App B] <---> [Database] <---> [Admin Tools]
# Attacker compromises App A -> can reach everything

# Micro-segmented:
[App A] --x-- [App B]     (blocked)
[App A] ----> [Database]   (only port 5432, read-only)
[App B] ----> [Database]   (only port 5432, read-write)
[Admin] ----> [All]        (after MFA + device check)
# Attacker compromises App A -> can only reach its own DB

Service-to-Service Zero Trust

  • Service mesh (Istio, Linkerd) — mTLS between all microservices
  • SPIFFE / SPIRE — cryptographic service identity standard
  • API gateways — authenticate and authorize every API call
  • Workload identity — no shared secrets, identity tied to workload
# SPIFFE ID format:
spiffe://trust-domain/workload-identifier
spiffe://mycompany.com/ns/prod/sa/payment-service

# Istio PeerAuthentication (enforce mTLS)
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: production
spec:
  mtls:
    mode: STRICT    # All traffic must be mTLS

Data Protection

  • Classification — label data: public, internal, confidential, restricted
  • Encryption at rest — AES-256 for storage (S3 SSE, EBS, RDS)
  • Encryption in transit — TLS 1.3 everywhere, mTLS between services
  • DLP (Data Loss Prevention) — detect and block sensitive data exfiltration
  • Tokenization — replace sensitive data with non-sensitive tokens
  • Key management — AWS KMS, HashiCorp Vault, GCP Cloud KMS

Continuous Monitoring & Analytics

  • SIEM — aggregate and correlate security events (Splunk, Sentinel, Elastic)
  • UEBA — User and Entity Behavior Analytics — detect anomalies
  • SOAR — automate incident response playbooks
  • NDR — Network Detection and Response for east-west traffic
# Signals to monitor continuously:
- Login from new device or impossible travel
- Privilege escalation attempts
- Unusual data access patterns or volume
- Failed authentication spikes
- Service-to-service traffic anomalies
- Configuration changes to critical systems
- DNS queries to known malicious domains

# Response: auto-revoke session, require step-up MFA,
#           isolate workload, alert SOC team

Privileged Access Management

  • JIT (Just-In-Time) access — elevate privileges only when needed, time-limited
  • JEA (Just-Enough-Access) — minimum scope for the task at hand
  • PAM solutions — CyberArk, HashiCorp Vault, AWS SSM Session Manager
  • Break-glass procedures — emergency access with full audit trail
# Example: JIT access flow
1. Engineer requests prod DB access via Slack/portal
2. Manager approves (or auto-approved for on-call)
3. Vault generates temporary credentials (TTL: 1 hour)
4. Engineer accesses DB with short-lived creds
5. Credentials auto-revoked after TTL
6. Full session recorded in audit log

# No standing privileges = smaller blast radius

Zero Trust for Cloud (AWS Example)

# Identity: IAM + SSO
- AWS IAM Identity Center (SSO) with MFA
- IAM roles with least privilege policies
- No long-lived access keys; use IAM roles + STS

# Network: VPC + Security Groups
- Private subnets for workloads (no public IPs)
- Security groups: deny all, allow specific ports
- VPC endpoints for AWS services (no internet transit)

# Data: Encryption everywhere
- S3: SSE-KMS, bucket policies, Block Public Access
- RDS: encrypted at rest + in transit + IAM auth
- KMS: customer-managed keys, key rotation

# Monitoring: CloudTrail + GuardDuty
- CloudTrail: audit every API call
- GuardDuty: threat detection
- Config: compliance rules
- SecurityHub: unified security dashboard

Most organizations are between stages 1-2. Full zero trust is a journey, not a project.

Implementation Roadmap

1 Identify your crown jewels — what data/systems matter most?
2 Map transaction flows — who/what accesses what?
3 Deploy MFA — fastest, highest-impact step
4 Implement SSO — centralize identity with OIDC/SAML
5 Micro-segment — start with critical workloads
6 Replace VPN with ZTNA for remote access
7 Encrypt all data at rest and in transit
8 Monitor continuously with SIEM + UEBA

Key Takeaways

  • Zero Trust means never trust, always verify
  • Three principles: Verify Explicitly Least Privilege Assume Breach
  • Identity is the new perimeter — not the firewall
  • MFA + SSO is the fastest, highest-impact first step
  • Micro-segmentation limits blast radius of breaches
  • It's a journey — start with identity, expand to network, data, and workloads
