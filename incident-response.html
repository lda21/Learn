What Is SRE?

Site Reliability Engineering

  • Coined by Google in 2003 — "what happens when you treat operations as a software problem"
  • Bridge between development and operations
  • Quantify reliability with SLOs (Service Level Objectives)
  • Automate toil — anything manual, repetitive, and automatable
  • Blameless culture focused on learning from failure

Error Budgets

  • Error budget = 100% − SLO target
  • 99.9% SLO = 0.1% budget = ~43 min/month downtime
  • Budget remaining? Ship features faster
  • Budget exhausted? Freeze deploys, focus on reliability
  • Aligns incentives between product and ops teams
  • For a 24/7 trading agent: 99.95% = ~22 min/month

Real-World: Trading Agent Incidents

Exchange Down (SEV-2)

Binance WebSocket disconnects. Agent loses price feed. Stale quotes lead to mispriced orders on other venues.

Impact: Cross-exchange arb exposure

Drawdown Breach (SEV-1)

Portfolio hits -5% daily drawdown limit. Kill switch must fire within 30 seconds to flatten all positions.

Impact: Direct financial loss

Risk Gate Failure (SEV-2)

Risk evaluation service crashes. Orders bypass position limits. Concentration risk builds undetected.

Impact: Unbounded position sizing

Position Mismatch (SEV-2)

Internal state diverges from exchange. Agent thinks it is flat but holds 10 BTC. Reconciliation fails.

Impact: Phantom P&L, hidden risk

Runbooks

What Is a Runbook?

  • Step-by-step playbook for known failure scenarios
  • Written before the incident — not during
  • Reduces MTTR (Mean Time To Recover)
  • Enables junior engineers to handle SEV-2 incidents
  • Stored in version control, reviewed quarterly
  • Links to dashboards, logs, and escalation contacts

Runbook Structure

## Runbook: [Failure Name]
**Severity:** SEV-2
**Last Updated:** 2026-03-15
**Owner:** @on-call-sre

### Symptoms

### Diagnosis

### Escalation

Runbook Examples

## Runbook: Exchange WebSocket Disconnect
Severity: SEV-2 | Response: < 15 min

1. DETECT: Alert "ws_disconnect_{exchange}" fires
2. CHECK:  Grafana dashboard → Exchange Connectivity panel
3. VERIFY: curl -s https://api.binance.com/api/v3/ping  (is exchange up?)
4. IF exchange up:
   - Restart WS reconnect:  kubectl rollout restart deploy/feed-handler-binance
   - Monitor reconnect in Loki: {app="feed-handler"} |= "reconnected"
5. IF exchange down:
   - Disable venue:  curl -X POST http://agent:8080/venue/binance/disable
   - Reduce positions 50%: curl -X POST http://agent:8080/risk/reduce?factor=0.5
   - Monitor exchange status page, re-enable when stable

## Runbook: Risk Gate Failure
Severity: SEV-2 | Response: < 15 min

1. DETECT: Alert "risk_gate_error_rate > 1%" fires
2. IMMEDIATE: Enable fail-closed mode (reject all new orders)
   curl -X POST http://agent:8080/risk/mode/fail-closed
3. CHECK: Loki logs {app="risk-engine"} |= "error"
4. FIX: Restart risk engine, verify evaluation latency < 10ms
5. RE-ENABLE: curl -X POST http://agent:8080/risk/mode/normal

## Runbook: Position Mismatch
Severity: SEV-2 | Response: < 15 min

1. DETECT: Alert "position_mismatch_detected" fires
2. HALT: Pause all new orders for affected exchange
3. RECONCILE: curl -X POST http://agent:8080/positions/reconcile?exchange=binance
4. COMPARE: Internal state vs exchange API response
5. RESOLVE: Accept exchange state as source of truth, adjust internal ledger

Per-Exchange Circuit Breaker

  • Closed: Orders flow normally to exchange
  • Error threshold: 5 failures in 30 seconds
  • Open: All orders to that exchange rejected
  • Cooldown: 60 seconds before probing
  • Half-Open: Send 1 test order, if OK → close
  • Other exchanges remain unaffected

Global Circuit Breaker

  • Triggers when 2+ exchange breakers open simultaneously
  • Reduces total portfolio exposure by 75%
  • Halts all new position entries
  • Allows exits and risk-reducing trades only
  • Auto-recovers when all per-exchange breakers close
  • Manual override via kill switch if needed

Graceful Degradation

ConditionDetectionDegradation Action
Price feed stale (> 5s)feed_staleness_seconds > 5Reduce position sizes by 50%, widen spreads
Single exchange downCircuit breaker opensRoute orders to remaining venues, reduce exposure
Risk service downrisk_gate_health == 0Fail-closed: reject all new orders, allow exits
Model inference slowmodel_latency_p99 > 1sFall back to simple momentum signals
Database unreachableConnection pool exhaustedTrade from in-memory state, queue writes
Funding rate spikefunding_rate > 0.1%Close affected perpetual positions, switch to spot

Principle: Every component must have a degraded mode. Never fail silently — always reduce risk.

Kill Switch Design

Emergency Halt Requirements

  • Flatten all positions across all exchanges in < 30 seconds
  • Cancel all open orders immediately
  • Disable all new order submission
  • Triggered by: drawdown breach, operator command, or system anomaly
  • Multiple trigger paths: API, CLI, Slack command, physical button
  • Cannot be accidentally triggered (2-step confirmation)

Kill Switch Sequence

# Kill switch activation sequence
1. TRIGGER received (drawdown / manual)
2. SET global_halt = true          # 0ms
3. CANCEL all open orders          # parallel, per-exchange
   - Binance: cancel_all()         # ~200ms
   - OKX: cancel_all()             # ~200ms
   - Bybit: cancel_all()           # ~200ms
4. FLATTEN all positions            # market orders
   - Send market sells/buys        # ~500ms
   - Verify fills received         # ~2s
5. VERIFY net exposure == 0         # ~5s
6. LOCK system (require manual restart)
7. NOTIFY: Slack + SMS + phone
8. LOG: Full state snapshot to S3

Total time budget: < 30 seconds

Chaos Engineering

Fault Injection Tests

  • Exchange disconnect: Terminate WS connections randomly — verify circuit breaker opens within 30s
  • Risk gate crash: Kill risk engine pod — verify fail-closed mode activates
  • Network partition: Block traffic between agent and exchange — verify graceful degradation
  • Clock skew: Offset system time by 5s — verify timestamp validation catches it
  • Disk full: Fill log volume — verify log rotation and alerts fire

Kill Switch Drills

  • Run monthly on staging with simulated positions
  • Measure: time to flatten, order cancellation completeness
  • Target: full flatten in < 30 seconds
  • Record results, compare against previous drills
  • Rotate drill leader each month (trains entire team)

Game Days

  • Quarterly full-scale incident simulation
  • Inject SEV-1 scenario into staging
  • Team responds as if production incident
  • Score: detection time, MTTR, communication quality

Best Practices

1. Define SLOs before writing alerts — alert on SLO burn rate, not raw metrics
2. Every alert must have a runbook link — no alert without a documented response
3. Practice blameless postmortems — focus on systems, not individuals
4. Test your kill switch monthly — an untested safety net is not a safety net
5. Design for graceful degradation — every component needs a failure mode
6. Use circuit breakers per dependency — isolate blast radius of failures
7. Correlate metrics, logs, and traces — the LGTM stack with shared trace IDs
8. Automate toil ruthlessly — if you do it twice, automate it the third time
9. Maintain on-call health — compensate, rotate fairly, limit alert fatigue
10. Treat incidents as learning opportunities — every failure makes the system stronger
