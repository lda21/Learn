Why Kubernetes?

The Problem

  • You have 10s–1000s of containers to manage
  • Need to deploy, scale, and heal automatically
  • Docker runs containers — K8s orchestrates them
  • Handles networking, storage, secrets, config
  • Declarative: describe desired state, K8s makes it happen

What K8s Gives You

  • Self-healing — restarts crashed containers
  • Auto-scaling — HPA scales on CPU/memory
  • Rolling updates — zero-downtime deploys
  • Service discovery — built-in DNS
  • Secret management — secure config injection
  • Storage orchestration — attach volumes dynamically

Control Plane

  • API Server — front door for all operations (REST)
  • etcd — distributed KV store for all cluster state
  • Scheduler — assigns pods to nodes based on constraints
  • Controller Manager — runs reconciliation loops (Deployment, ReplicaSet, Job controllers)
  • Cloud Controller — manages cloud-specific resources (load balancers, volumes)

Worker Nodes

  • kubelet — agent that runs pods on each node
  • kube-proxy — maintains network rules, load balances traffic
  • Container Runtime — containerd or CRI-O (not Docker since K8s 1.24)
  • Nodes can be VMs, bare metal, or cloud instances
  • Each node reports status, capacity, and conditions to control plane

Nodes

What is a Node?

  • A physical or virtual machine that runs pods
  • Has allocatable CPU, memory, and storage
  • Conditions — Ready, MemoryPressure, DiskPressure, PIDPressure
  • If NotReady for 5 min → pods get evicted
  • Dashboard shows 3 nodes with Ready/NotReady status
kubectl get nodes -o wide
kubectl describe node <name>
kubectl top node          # CPU/memory usage
kubectl cordon <node>    # mark unschedulable
kubectl drain <node>     # evict pods safely
kubectl uncordon <node>  # allow scheduling again

Labels, Taints & Tolerations

Labels — key/value metadata for selection:

kubectl label node node-1 \
  disktype=ssd tier=production

# Schedule to specific nodes:
nodeSelector:
  disktype: ssd

Taints & Tolerations — repel pods unless they tolerate the taint:

# Taint a node (only GPU pods)
kubectl taint node gpu-1 \
  gpu=true:NoSchedule

# Pod toleration:
tolerations:
  - key: "gpu"
    operator: "Equal"
    value: "true"
    effect: "NoSchedule"

  • Resources within a namespace have unique names
  • Cross-namespace access via FQDN: svc.namespace.svc.cluster.local
  • Apply resource quotas and network policies per namespace

  • Created by controllers, scheduler, kubelet
  • Types: Normal (informational) and Warning (problems)
  • Retained for 1 hour by default (configurable)
  • First place to look when debugging

Pods

The Smallest Unit

  • Smallest deployable unit in K8s
  • One or more containers sharing network and storage
  • Each pod gets its own IP address
  • Containers in a pod communicate via localhost
  • Pods are ephemeral — they can be killed and recreated
  • Never create pods directly — use Deployments

Multi-Container Patterns

  • Sidecar — log collector, service mesh proxy (Envoy)
  • Init container — runs before main, sets up config/DB migrations
  • Ambassador — proxy for outgoing connections

Pod YAML

apiVersion: v1
kind: Pod
metadata:
  name: myapp
  labels:
    app: myapp
    version: v1
spec:
  initContainers:
    - name: db-migrate
      image: myapp:1.0
      command: ["./migrate", "up"]
  containers:
    - name: app
      image: myapp:1.0
      ports:
        - containerPort: 3000
      resources:
        requests:        # scheduler guarantee
          cpu: "100m"
          memory: "128Mi"
        limits:          # hard ceiling
          cpu: "500m"
          memory: "256Mi"
    - name: log-shipper  # sidecar
      image: fluent-bit:latest
      volumeMounts:
        - name: logs
          mountPath: /var/log/app
  volumes:
    - name: logs
      emptyDir: {}

Pod Lifecycle & Probes

Pod Phases

Pending — waiting for scheduling or image pull
→
Running — at least one container is running
→
Succeeded / Failed — all containers terminated

Container States

  • Waiting — pulling image, creating container
  • Running — executing with no issues
  • Terminated — finished or crashed (check exit code)
  • CrashLoopBackOff — keeps crashing, K8s backs off restart interval
  • ImagePullBackOff — can’t pull the image (wrong tag, no auth)

Health Probes

containers:
  - name: app
    image: myapp:1.0

    # Wait for slow startups (checked first)
    startupProbe:
      httpGet:
        path: /healthz
        port: 3000
      failureThreshold: 30
      periodSeconds: 10

    # Is the container alive? (restart if not)
    livenessProbe:
      httpGet:
        path: /healthz
        port: 3000
      periodSeconds: 10
      failureThreshold: 3

    # Can it accept traffic? (remove from Service)
    readinessProbe:
      httpGet:
        path: /ready
        port: 3000
      periodSeconds: 5
      failureThreshold: 3

Probe types: httpGet, tcpSocket, exec, grpc.

ReplicaSets

Ensuring Desired Replicas

  • Guarantees a specified number of identical pods are running
  • If a pod dies, ReplicaSet creates a replacement
  • If too many pods exist, it terminates the excess
  • Uses label selectors to identify its pods
  • You almost never create ReplicaSets directly — Deployments manage them

kind: ReplicaSet
metadata:
  name: myapp-rs
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp        # must match template
  template:

      containers:
        - name: app
          image: myapp:1.0
          ports:
            - containerPort: 3000

How Deployments Use Them

During a rolling update, the Deployment creates a new ReplicaSet (v2) and scales it up while scaling the old ReplicaSet (v1) down. Old ReplicaSets are kept for rollback history.

Deployments

The most common workload. Manages a set of identical pods via ReplicaSets. Handles rolling updates and rollbacks declaratively.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3                    # run 3 identical pods
  selector:
    matchLabels:
      app: myapp                 # manage pods with this label
  strategy:
    type: RollingUpdate          # zero-downtime deploys
    rollingUpdate:
      maxSurge: 1                # 1 extra pod during update
      maxUnavailable: 0          # never drop below desired count
  template:                      # pod template
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: app
          image: myapp:1.0
          ports:
            - containerPort: 3000
          readinessProbe:        # only route traffic when ready
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 5
          livenessProbe:         # restart if unhealthy
            httpGet:
              path: /health
              port: 3000
            periodSeconds: 10

Deployment Strategies & Rollbacks

Strategy Types

StrategyBehavior
RollingUpdateGradually replace old pods with new. Zero downtime. Default.
RecreateKill all old pods first, then create new. Brief downtime.

Advanced Patterns

  • Blue-Green — run 2 Deployments, switch Service selector
  • Canary — route % of traffic to new version via Ingress weights or service mesh
  • A/B Testing — route by header/cookie via Istio or Nginx annotations

Rollback Commands

# Watch a rollout
kubectl rollout status deploy/myapp

# View rollout history
kubectl rollout history deploy/myapp

# See a specific revision
kubectl rollout history deploy/myapp \
  --revision=3

# Rollback to previous version
kubectl rollout undo deploy/myapp

# Rollback to specific revision
kubectl rollout undo deploy/myapp \
  --to-revision=2

# Pause rollout (for canary testing)
kubectl rollout pause deploy/myapp

# Resume rollout
kubectl rollout resume deploy/myapp

StatefulSets

When Order & Identity Matter

  • Pods get stable, unique names: postgres-0, postgres-1, postgres-2
  • Created and deleted in order (0 → 1 → 2)
  • Each pod gets its own PersistentVolume that survives restarts
  • Stable DNS: postgres-0.postgres-headless.ns.svc.cluster.local
  • Use for: databases, Kafka, ZooKeeper, Redis Cluster, Elasticsearch
  • Dashboard shows 7 running StatefulSets

vs Deployment

FeatureDeploymentStatefulSet
Pod namesRandom hashOrdered index
StorageShared/nonePer-pod PVC
ScalingParallelSequential
NetworkInterchangeableStable DNS

StatefulSet YAML

apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
spec:
  serviceName: postgres-headless  # required
  replicas: 3
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:16
          ports:
            - containerPort: 5432
          volumeMounts:
            - name: data
              mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:    # per-pod PVC
    - metadata:
        name: data
      spec:
        accessModes: [ReadWriteOnce]
        storageClassName: gp3
        resources:
          requests:
            storage: 20Gi

DaemonSets

One Pod Per Node

  • Ensures a copy of a pod runs on every node (or a subset)
  • When a new node joins → DaemonSet pod is automatically added
  • When a node is removed → pod is garbage collected
  • Dashboard shows 7 DaemonSets running (matches 3 nodes × typical agents)

Common Use Cases

  • Log collectors — Fluent Bit, Fluentd, Filebeat
  • Monitoring agents — Prometheus Node Exporter, Datadog
  • Network plugins — Calico, Cilium, AWS VPC CNI
  • Storage daemons — CSI node drivers, Longhorn
  • Security agents — Falco, Sysdig
  • kube-proxy itself runs as a DaemonSet

DaemonSet YAML

apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluent-bit
  namespace: kube-system
spec:
  selector:
    matchLabels:
      app: fluent-bit
  template:
    metadata:
      labels:
        app: fluent-bit
    spec:
      # Tolerate control plane taints
      tolerations:
        - effect: NoSchedule
          operator: Exists
      containers:
        - name: fluent-bit
          image: fluent/fluent-bit:latest
          resources:
            requests:
              cpu: 50m
              memory: 64Mi
            limits:
              cpu: 200m
              memory: 128Mi
          volumeMounts:
            - name: varlog
              mountPath: /var/log
              readOnly: true
      volumes:
        - name: varlog
          hostPath:
            path: /var/log

Jobs

Run to Completion

  • Creates one or more pods and ensures they succeed
  • Unlike Deployments, Jobs terminate when done
  • Retries on failure up to backoffLimit
  • Dashboard shows Complete, Pending, Failed status

Use Cases

  • Database migrations
  • Batch data processing
  • Sending bulk emails/notifications
  • ETL pipelines
  • One-time setup tasks

Parallelism

ModeSetting
Sequentialcompletions: 5, parallelism: 1
Parallelcompletions: 10, parallelism: 3
Work queueparallelism: 3 (no completions)

Job YAML

apiVersion: batch/v1
kind: Job
metadata:
  name: db-migrate
spec:
  completions: 1         # how many must succeed
  parallelism: 1         # how many run at once
  backoffLimit: 3        # retries before failure
  activeDeadlineSeconds: 600  # timeout (10 min)
  ttlSecondsAfterFinished: 86400  # cleanup after 24h
  template:
    spec:
      restartPolicy: Never   # or OnFailure
      containers:
        - name: migrate
          image: myapp:1.0
          command: ["./migrate", "up"]
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: db-credentials
                  key: url

# Check job status
kubectl get jobs
kubectl describe job db-migrate
kubectl logs job/db-migrate

CronJobs

Scheduled Jobs

  • Creates Jobs on a cron schedule
  • Like Unix cron but for Kubernetes pods
  • Each trigger creates a new Job → which creates Pods
  • Dashboard shows 4 scheduled CronJobs

Cron Syntax

ScheduleMeaning
*/5 * * * *Every 5 minutes
0 * * * *Every hour
0 2 * * *Daily at 2:00 AM
0 0 * * 0Weekly on Sunday
0 0 1 * *First of each month

Concurrency Policies

  • Allow — multiple Jobs can run simultaneously (default)
  • Forbid — skip if previous Job still running
  • Replace — kill running Job, start new one

CronJob YAML

apiVersion: batch/v1
kind: CronJob
metadata:
  name: cleanup-old-data
spec:
  schedule: "0 3 * * *"       # 3 AM daily
  concurrencyPolicy: Forbid   # skip if still running
  startingDeadlineSeconds: 200 # miss window? skip
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 1
  jobTemplate:
    spec:
      backoffLimit: 2
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: cleanup
              image: myapp:1.0
              command:
                - /bin/sh
                - -c
                - |
                  echo "Cleaning data older than 90d"
                  ./cleanup --days=90
              resources:
                requests:
                  cpu: 100m
                  memory: 128Mi

kubectl get cronjobs
kubectl get jobs --watch  # see triggered Jobs

ConfigMaps

Non-Sensitive Configuration

  • Decouple config from container images
  • Store key-value pairs or entire config files
  • Inject as environment variables or mounted files
  • Changes to mounted ConfigMaps auto-update (within ~30s)
  • Changes to env vars require pod restart
  • Max size: 1 MiB

ConfigMap YAML

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: "info"
  API_URL: "https://api.example.com"
  config.json: |
    {
      "feature_flags": {"v2": true},
      "cache_ttl": 300
    }
---
# Use as env vars:
envFrom:
  - configMapRef:
      name: app-config

# Or mount as files:
volumes:
  - name: config
    configMap:
      name: app-config
containers:
  - volumeMounts:
      - name: config
        mountPath: /etc/app
        # creates /etc/app/LOG_LEVEL
        # creates /etc/app/config.json

Secrets

Sensitive Data

  • Store passwords, tokens, TLS certs, SSH keys
  • Base64-encoded (not encrypted by default!)
  • Enable encryption at rest in etcd for real security
  • Mounted as tmpfs (in-memory) — never written to node disk
  • RBAC controls who can read Secrets

Secret Types

TypeUse
OpaqueGeneric key-value (default)
kubernetes.io/tlsTLS cert + private key

kubernetes.io/basic-authUsername + password
kubernetes.io/ssh-authSSH private key

Resource Quotas

Namespace-Level Limits

  • Cap total CPU, memory, and object count per namespace
  • Prevents one team from consuming all cluster resources
  • Pod creation is rejected if it would exceed quota
  • Works with LimitRange for per-pod defaults
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
  namespace: team-alpha
spec:
  hard:
    requests.cpu: "10"       # 10 cores total
    requests.memory: 20Gi
    limits.cpu: "20"
    limits.memory: 40Gi
    pods: "50"               # max 50 pods
    services: "10"
    persistentvolumeclaims: "20"
    configmaps: "30"
    secrets: "30"

Scoped Quotas

# Quota only for BestEffort pods
# (pods without resource requests/limits)
apiVersion: v1
kind: ResourceQuota
metadata:
  name: besteffort-quota
spec:
  hard:
    pods: "5"
  scopes:
    - BestEffort

---
# Quota for non-terminating workloads
apiVersion: v1
kind: ResourceQuota
metadata:
  name: compute-quota
spec:
  hard:
    pods: "20"
  scopes:
    - NotTerminating

# Check quota usage
kubectl get resourcequota -n team-alpha
kubectl describe resourcequota team-quota \
  -n team-alpha

Limit Ranges

Per-Pod Defaults & Bounds

  • Set default resource requests/limits for containers
  • Enforce min/max per container or pod
  • Applied per namespace
  • If a pod has no resources set → LimitRange injects defaults
  • If resources exceed max → pod creation is rejected

ResourceQuota = total for the namespace.
LimitRange = per individual pod/container.

LimitRange YAML

apiVersion: v1
kind: LimitRange
metadata:
  name: pod-limits
  namespace: staging
spec:
  limits:
    - type: Container
      default:           # applied if not specified
        cpu: "200m"
        memory: "256Mi"
      defaultRequest:    # request defaults
        cpu: "100m"
        memory: "128Mi"
      min:               # minimum allowed
        cpu: "50m"
        memory: "64Mi"
      max:               # maximum allowed
        cpu: "2"
        memory: "2Gi"
    - type: Pod
      max:
        cpu: "4"
        memory: "4Gi"
    - type: PersistentVolumeClaim
      min:
        storage: 1Gi
      max:
        storage: 50Gi

Horizontal Pod Autoscaler (HPA)

Auto-Scale Pod Count

  • Watches CPU, memory, or custom metrics
  • Scales Deployment/StatefulSet replicas up or down
  • Checks metrics every 15 seconds by default
  • Requires metrics-server installed in cluster
  • Pods must have resource requests set
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70

Advanced HPA

# Multiple metrics + scaling behavior
spec:
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
    - type: Pods      # custom metric
      pods:
        metric:
          name: requests_per_second
        target:
          type: AverageValue
          averageValue: 1000
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
        - type: Percent
          value: 10
          periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
        - type: Percent
          value: 50
          periodSeconds: 60

kubectl get hpa
kubectl describe hpa myapp-hpa

Pod Disruption Budgets (PDB)

Protecting Availability

  • Limits how many pods can be voluntarily disrupted at once
  • Voluntary: node drain, cluster upgrade, scaling down
  • Involuntary (node crash) bypasses PDBs
  • Ensures enough replicas stay running during maintenance
  • Blocks kubectl drain if PDB would be violated

When to Use

  • Production services that need high availability
  • Databases and stateful workloads
  • During cluster upgrades (node-by-node drain)
  • Any service where downtime = SLO violation

PDB YAML

# Option 1: Minimum available
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: myapp-pdb
spec:
  minAvailable: 2          # at least 2 must stay up
  selector:
    matchLabels:
      app: myapp

---
# Option 2: Maximum unavailable
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-pdb
spec:
  maxUnavailable: 1        # at most 1 can be down
  selector:
    matchLabels:
      app: api

---
# Percentage-based
spec:
  minAvailable: "50%"      # at least half stay up

kubectl get pdb
kubectl describe pdb myapp-pdb

Priority Classes

Pod Scheduling Priority

  • Assigns a priority value to pods (higher = more important)
  • Scheduler prefers scheduling higher-priority pods first
  • Preemption — lower-priority pods can be evicted to make room
  • Critical for mixed workloads (production vs batch jobs)
PriorityValueUse
system-cluster-critical2BCoreDNS, kube-proxy
system-node-critical2B+kubelet, CNI
Production1MUser-facing APIs
Batch100KBackground jobs
Best-effort0Dev/test workloads

PriorityClass YAML

apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: production-critical
value: 1000000
globalDefault: false
preemptionPolicy: PreemptLowerPriority
description: "Production workloads"

---
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: batch-low
value: 100000
preemptionPolicy: Never  # don't evict others
description: "Batch jobs, can wait"

# Use in pod spec:
spec:
  priorityClassName: production-critical
  containers:
    - name: app
      image: myapp:1.0

Runtime Classes

Container Runtime Selection

  • Choose which container runtime runs your pods
  • Different runtimes offer different security/performance tradeoffs
  • Configured per node via CRI (Container Runtime Interface)
RuntimeHandlerUse Case
containerdruncDefault, standard containers
gVisorrunscKernel isolation (security sandbox)
Kata ContainerskataLightweight VMs per pod
Spin (Wasm)spinWebAssembly workloads

When to use: Multi-tenant clusters, untrusted code, compliance requirements that need stronger isolation than standard containers.

RuntimeClass YAML

apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: gvisor
handler: runsc              # CRI handler name
overhead:                   # extra resources needed
  podFixed:
    cpu: "50m"
    memory: "64Mi"
scheduling:                 # only on nodes with gVisor
  nodeSelector:
    runtime: gvisor

# Use in pod spec:
apiVersion: v1
kind: Pod
metadata:
  name: sandboxed-app
spec:
  runtimeClassName: gvisor
  containers:
    - name: app
      image: untrusted-app:1.0

Most clusters only use the default runtime. RuntimeClasses are advanced.

Leases

Distributed Coordination

  • Lightweight objects for leader election and heartbeats
  • K8s uses them internally for node heartbeats (replaces Node status updates)
  • Each node has a Lease in kube-node-lease namespace
  • kubelet renews its Lease every 10 seconds
  • If Lease isn’t renewed → node considered unhealthy

Use Cases

  • Node heartbeats — more efficient than updating Node status
  • Leader election — only one controller replica is active
  • API server identity — tracks which API servers are alive
  • Custom controllers can use Leases for coordination

Lease YAML

# Node heartbeat lease (auto-managed)
apiVersion: coordination.k8s.io/v1
kind: Lease
metadata:
  name: node-1
  namespace: kube-node-lease
spec:
  holderIdentity: node-1
  leaseDurationSeconds: 40
  renewTime: "2026-03-31T10:05:00Z"

# View node leases
kubectl get leases -n kube-node-lease

# Leader election lease (controllers)
kubectl get leases -n kube-system
# Shows: kube-scheduler, kube-controller-manager

You rarely create Leases manually. They’re used by the system and by custom controllers via the client-go/tools/leaderelection package.

Endpoints & Endpoint Slices

How Services Find Pods

  • Endpoints = list of pod IPs backing a Service
  • Auto-created when you create a Service with a selector
  • Updated when pods are added/removed or fail readiness probes
  • Endpoint Slices replaced Endpoints for scalability (K8s 1.21+)
Service: myapp-svc (10.100.0.50:80)
→
EndpointSlice: myapp-svc-abc
→
10.244.1.5:3000, 10.244.2.8:3000, 10.244.3.2:3000

Why Endpoint Slices?

  • Old Endpoints = one object with ALL IPs (huge for big Services)
  • Endpoint Slices split into chunks of 100 endpoints each
  • Less data transferred on each pod change
  • Better performance for kube-proxy and Ingress controllers
# View endpoints
kubectl get endpoints myapp-svc

# View endpoint slices
kubectl get endpointslices -l \
  kubernetes.io/service-name=myapp-svc

# Manual endpoint (no selector service)
apiVersion: v1
kind: Endpoints
metadata:
  name: external-db
subsets:
  - addresses:
      - ip: 10.0.1.50
    ports:
      - port: 5432

Ingress & Ingress Classes

HTTP/HTTPS Routing

Exposes routes from outside the cluster to Services. Like a managed reverse proxy.

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  tls:
    - hosts: [myapp.example.com]
      secretName: myapp-tls
  rules:
    - host: myapp.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: api-svc
                port: { number: 80 }
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend-svc
                port: { number: 80 }

Ingress Classes

  • Specify which Ingress Controller handles the Ingress
  • A cluster can have multiple controllers (nginx + ALB)
  • One class can be marked as default
apiVersion: networking.k8s.io/v1
kind: IngressClass
metadata:
  name: nginx
  annotations:
    ingressclass.kubernetes.io/is-default-class: "true"
spec:
  controller: k8s.io/ingress-nginx

Network Policies

Pod-Level Firewall

  • Control which pods can talk to which
  • By default, all pods can communicate with all other pods
  • NetworkPolicy restricts ingress and/or egress traffic
  • Requires a CNI that supports policies (Calico, Cilium, Weave)
  • AWS VPC CNI alone does NOT enforce — use Calico on EKS

Selection

  • podSelector — target pods by label
  • namespaceSelector — allow from specific namespaces
  • ipBlock — allow from IP CIDR ranges

NetworkPolicy YAML

# Only allow traffic from pods with app=frontend
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-allow-frontend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api                # apply to api pods
  policyTypes: [Ingress, Egress]
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend   # only frontend can reach api
        - namespaceSelector:
            matchLabels:
              env: production # only from production ns
      ports:
        - port: 3000
          protocol: TCP
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: postgres   # api can reach postgres
      ports:
        - port: 5432
    - to:                     # allow DNS
        - namespaceSelector: {}
      ports:
        - port: 53
          protocol: UDP

Calico — CNI & Network Security

Calico is a CNI plugin (Container Network Interface) that provides pod networking AND network policy enforcement. The most popular CNI for production K8s.

What Calico Does

  • Pod networking — assigns IPs, routes traffic between pods across nodes (BGP or VXLAN overlay)
  • NetworkPolicy enforcement — K8s NetworkPolicy only defines rules. Calico actually enforces them via iptables/eBPF
  • GlobalNetworkPolicy — Calico-specific: cluster-wide policies (K8s native only does per-namespace)
  • DNS-based policies — allow/deny by domain name, not just IP
  • Egress gateway — control outbound traffic from pods to external services

Why Not Just AWS VPC CNI?

  • VPC CNI assigns real VPC IPs to pods (good for performance)
  • But VPC CNI does NOT enforce NetworkPolicies
  • On EKS: use VPC CNI for networking + Calico for policy enforcement

Install on EKS

# EKS: VPC CNI handles networking,
# Calico handles policy enforcement only
kubectl apply -f https://raw.githubusercontent.com/\
  aws/amazon-vpc-cni-k8s/master/config/master/\
  calico-operator.yaml
kubectl apply -f https://raw.githubusercontent.com/\
  aws/amazon-vpc-cni-k8s/master/config/master/\
  calico-crs.yaml

# Bare-metal: Calico handles BOTH
# networking + policies
helm install calico projectcalico/tigera-operator \
  --namespace tigera-operator \
  --create-namespace

Calico GlobalNetworkPolicy

# Deny all egress except DNS cluster-wide
# (K8s NetworkPolicy can't do this)
apiVersion: projectcalico.org/v3
kind: GlobalNetworkPolicy
metadata:
  name: default-deny-egress
spec:
  selector: all()
  types: [Egress]
  egress:
    - action: Allow
      protocol: UDP
      destination:
        ports: [53]
    - action: Deny

Port Forwarding

Local Access to Cluster

  • Forward a local port to a pod or service inside the cluster
  • No Ingress or LoadBalancer needed
  • Great for debugging, development, and database access
  • Traffic goes through the API server (encrypted)
  • Only works while the kubectl process is running

Common Uses

  • Access a database that has no external endpoint
  • Reach a dashboard (Grafana, Prometheus, ArgoCD)
  • Debug a service without exposing it
  • Test an API before setting up Ingress

Commands

# Forward to a pod
kubectl port-forward pod/myapp-xyz \
  8080:3000
# localhost:8080 → pod:3000

# Forward to a service
kubectl port-forward svc/grafana \
  3000:80 -n monitoring
# localhost:3000 → grafana service

# Forward to a deployment
kubectl port-forward deploy/myapp \
  8080:3000

# Listen on all interfaces (not just localhost)
kubectl port-forward --address 0.0.0.0 \
  svc/myapp 8080:80

# Multiple ports at once
kubectl port-forward pod/myapp-xyz \
  8080:3000 9090:9090

# Background it
kubectl port-forward svc/postgres \
  5432:5432 -n databases &

Nautik has a Port Forwardings screen that lets you manage these with a GUI.

CSI — Container Storage Interface

CSI is a standard that lets K8s talk to any storage system via plugins. No need to modify K8s core.

Two Components

  • CSI Controller (Deployment) — creates/deletes/attaches volumes. Talks to storage API.
  • CSI Node Driver (DaemonSet) — mounts/formats volumes on each node. Runs privileged.

PVC Flow with CSI

# 1. You create a PVC
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: kafka-data
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: ebs-csi    # which driver
  resources:
    requests:
      storage: 100Gi

# What happens:
# 1. PVC created → K8s asks CSI Controller
#    "create a 100Gi volume"
# 2. Controller calls AWS API → EBS created
# 3. Pod scheduled to node-3 → "attach to node-3"
# 4. Controller calls AWS → EBS attaches to EC2
# 5. Node Driver formats (ext4), mounts
# 6. Pod starts, sees /data with 100Gi

Longhorn — Distributed Storage for K8s

Uses node local disks with automatic replication. No cloud vendor required.

The Problem

  • local-path: node dies → data is GONE
  • NFS server: single point of failure
  • Ceph: powerful but complex to operate
  • EBS: AWS only, costs money

Longhorn Solution

  • Replicates data across 2-3 nodes automatically
  • Node dies → pod moves, data follows (zero loss)
  • Built-in snapshots, S3 backup, encryption
  • Web UI for management
  • Free, open-source (CNCF)

Install & Use

# Install via Helm
helm install longhorn longhorn/longhorn \
  -n longhorn-system --create-namespace \
  --set defaultSettings.defaultReplicaCount=3

# StorageClass (auto-created, or custom)
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: longhorn-3x
provisioner: driver.longhorn.io
parameters:
  numberOfReplicas: "3"
  dataLocality: "best-effort"

# Use in PVC — identical to any storage
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: longhorn-3x
  resources:
    requests:
      storage: 50Gi

Postgres: local-path vs Longhorn

local-path:
  Node 1 dies → data on Node 1 disk
  → POD MOVES TO NODE 2
  → DATA IS GONE. DATABASE LOST.

Longhorn (3 replicas):
  Node 1 dies → data on Node 1, 2, AND 3
  → Pod moves to Node 2
  → Longhorn attaches replica from Node 2
  → ZERO DATA LOSS. ~30s downtime.
  → New replica auto-created to restore 3x

When to Use What

WorkloadUse Longhorn?
Postgres / MySQLYES — can't lose data
Kafka brokersYES — faster recovery
MongoDB / ElasticYES
Redis (cache)NO — ephemeral, needs sub-ms
Stateless appsNO — no PVC needed

Latency: ~0.5-1ms vs ~0.1ms local. Fine for Postgres/Kafka. Not for Redis.

RBAC & Security

Role-Based Access Control

  • Role — permissions within a namespace
  • ClusterRole — cluster-wide permissions
  • RoleBinding — assigns Role to user/group/SA
  • ServiceAccount — identity for pods (not humans)
kind: Role
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  namespace: staging
  name: pod-reader
rules:
  - apiGroups: [""]
    resources: ["pods", "pods/log"]
    verbs: ["get", "list", "watch"]
---
kind: RoleBinding
metadata:
  name: read-pods
  namespace: staging
subjects:
  - kind: ServiceAccount
    name: ci-deployer
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io

Security Context

spec:
  securityContext:         # pod level
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 1000
  containers:
    - name: app
      securityContext:     # container level
        allowPrivilegeEscalation: false
        readOnlyRootFilesystem: true
        capabilities:
          drop: [ALL]

Security Checklist

  • Run as non-root (runAsNonRoot: true)
  • Read-only root filesystem
  • Drop all capabilities
  • Use NetworkPolicies (deny by default)
  • Encrypt Secrets at rest in etcd
  • Use IRSA (IAM Roles for Service Accounts) on EKS
  • Scan images for CVEs (Trivy, Snyk)

Best Practices

1 Always set resource requests and limits on every container
2 Use readinessProbe and livenessProbe for self-healing
3 Never run as root — set runAsNonRoot: true
4 Use Namespaces + ResourceQuotas to isolate environments
5 Store manifests in Git — use GitOps (ArgoCD, Flux)
6 Pin image tags to specific versions — never use :latest
7 Set PodDisruptionBudgets for high-availability services
8 Use HPA for stateless, VPA for right-sizing stateful
9 Enable NetworkPolicies — deny by default
10 Use RBAC with least-privilege ServiceAccounts per workload
11 Encrypt Secrets at rest and use external secret managers
