Kubernetes

Container orchestration at scale

Why Kubernetes?

The Problem

  • You have 10s–1000s of containers to manage
  • Need to deploy, scale, and heal automatically
  • Docker runs containers — K8s orchestrates them
  • Handles networking, storage, secrets, config
  • Declarative: describe desired state, K8s makes it happen

What K8s Gives You

  • Self-healing — restarts crashed containers
  • Auto-scaling — HPA scales on CPU/memory
  • Rolling updates — zero-downtime deploys
  • Service discovery — built-in DNS
  • Secret management — secure config injection
  • Storage orchestration — attach volumes dynamically

Control Plane

  • API Server — front door for all operations
  • etcd — distributed key-value store for cluster state
  • Scheduler — assigns pods to nodes
  • Controller Manager — runs reconciliation loops

Worker Nodes

  • kubelet — agent that runs pods on each node
  • kube-proxy — handles networking and load balancing
  • Container Runtime — containerd or CRI-O
  • Nodes can be VMs, bare metal, or cloud instances

Pods

What is a Pod?

  • Smallest deployable unit in K8s
  • One or more containers sharing network and storage
  • Each pod gets its own IP address
  • Containers in a pod communicate via localhost
  • Pods are ephemeral — they can be killed and recreated
  • Never create pods directly — use Deployments

Pod YAML

apiVersion: v1
kind: Pod
metadata:
  name: myapp
  labels:
    app: myapp
spec:
  containers:
    - name: app
      image: myapp:1.0
      ports:
        - containerPort: 3000
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "256Mi"

Deployments

A Deployment manages a set of identical pods. It handles scaling, rolling updates, and rollbacks.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3                    # run 3 identical pods
  selector:
    matchLabels:
      app: myapp                 # manage pods with this label
  strategy:
    type: RollingUpdate          # zero-downtime deploys
    rollingUpdate:
      maxSurge: 1                # 1 extra pod during update
      maxUnavailable: 0          # never drop below desired count
  template:                      # pod template
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: app
          image: myapp:1.0
          ports:
            - containerPort: 3000
          readinessProbe:        # only route traffic when ready
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 5
          livenessProbe:         # restart if unhealthy
            httpGet:
              path: /health
              port: 3000
            periodSeconds: 10

Services & Networking

What is a Service?

  • Stable network endpoint for a set of pods
  • Pods come and go — Services provide a fixed IP and DNS name
  • Load balances traffic across matching pods
  • Selects pods by labels
TypeScope
ClusterIPInternal only (default)
NodePortExposes on each node’s IP
LoadBalancerCloud LB (AWS ALB/NLB)

Service YAML

apiVersion: v1
kind: Service
metadata:
  name: myapp-svc
spec:
  type: ClusterIP
  selector:
    app: myapp       # route to these pods
  ports:
    - port: 80       # service port
      targetPort: 3000  # container port

---
# Access from other pods:
# http://myapp-svc.default.svc.cluster.local
# or just: http://myapp-svc

Ingress

Ingress exposes HTTP/HTTPS routes from outside the cluster to Services. Like a reverse proxy managed by K8s.

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  tls:
    - hosts: [myapp.example.com]
      secretName: myapp-tls        # TLS certificate
  rules:
    - host: myapp.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: api-svc
                port: { number: 80 }
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend-svc
                port: { number: 80 }

Requires: An Ingress Controller (NGINX, Traefik, AWS ALB Controller) installed in the cluster.

ConfigMaps & Secrets

ConfigMap

Store non-sensitive configuration. Inject as env vars or mounted files.

apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  LOG_LEVEL: "info"
  API_URL: "https://api.example.com"
  config.json: |
    {"feature_flags": {"v2": true}}

---
# Use in pod:
envFrom:
  - configMapRef:
      name: app-config

Secret

Store sensitive data (passwords, tokens, keys). Base64-encoded, optionally encrypted at rest.

apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
data:
  username: cG9zdGdyZXM=    # base64
  password: czNjcjN0         # base64

---
# Use in pod:
env:
  - name: DB_USER
    valueFrom:
      secretKeyRef:
        name: db-credentials
        key: username

Storage: PV & PVC

Pods are ephemeral — data is lost when they restart. Use PersistentVolumes for durable storage.

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
spec:
  accessModes: [ReadWriteOnce]   # single node read/write
  storageClassName: gp3          # AWS EBS gp3
  resources:
    requests:
      storage: 20Gi

---
# Mount in pod spec:
volumes:
  - name: data
    persistentVolumeClaim:
      claimName: postgres-data
containers:
  - volumeMounts:
      - name: data
        mountPath: /var/lib/postgresql/data

Namespaces & RBAC

Namespaces

  • Virtual clusters within a physical cluster
  • Isolate teams, environments, or applications
  • Resource quotas per namespace
  • Default namespaces: default, kube-system, kube-public
kubectl create namespace staging
kubectl get pods -n staging
kubectl config set-context --current   --namespace=staging

RBAC

  • Role-Based Access Control
  • Role — permissions within a namespace
  • ClusterRole — cluster-wide permissions
  • RoleBinding — assigns role to user/group
kind: Role
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]

Scaling: HPA & VPA

Horizontal Pod Autoscaler

Adds or removes pod replicas based on metrics.

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70

Vertical Pod Autoscaler

Adjusts CPU/memory requests and limits based on actual usage.

  • Recommend — suggests values (safe)
  • Auto — applies changes (restarts pods)
  • Use VPA for right-sizing, HPA for scaling out
  • Don’t use both on the same metric

Scaling Strategy

HPA for stateless services (API, web)
VPA for stateful workloads (databases)
Cluster Autoscaler for adding/removing nodes

Helm: The Package Manager

Helm packages K8s manifests into reusable, versioned charts with templating and values.

# Install a chart from a repository
helm repo add bitnami https://charts.bitnami.com/bitnami
helm install my-postgres bitnami/postgresql   --set auth.postgresPassword=secret   --set primary.persistence.size=20Gi   --namespace databases --create-namespace

# Your own chart structure:
mychart/
  Chart.yaml          # name, version, description
  values.yaml         # default configuration
  templates/
    deployment.yaml   # {{ .Values.replicas }}
    service.yaml      # {{ .Values.service.port }}
    ingress.yaml

# Deploy with custom values
helm install myapp ./mychart -f production-values.yaml
helm upgrade myapp ./mychart -f production-values.yaml
helm rollback myapp 1         # rollback to revision 1
helm list                      # see installed releases

Managed control plane on AWS. Integrates with ALB, EBS, IAM, Fargate for serverless pods.

The most mature managed K8s. Autopilot mode fully manages nodes. Deep integration with GCP services.

Free control plane on Azure. Integrates with Azure AD, Container Registry, and Azure Monitor.

Local Development

minikube, kind, or k3s for running K8s locally. Docker Desktop also includes K8s.

Best Practices

1. Always set resource requests and limits on containers
2. Use readinessProbe and livenessProbe for self-healing
3. Never run as root — set securityContext.runAsNonRoot: true
4. Use Namespaces to separate environments (dev, staging, prod)
5. Store manifests in Git — use GitOps (ArgoCD, Flux) for deploys
6. Use Helm charts for reusable, parameterized deployments
7. Pin image tags to specific versions — never use :latest in prod
8. Set PodDisruptionBudgets for high-availability services
9. Use HPA for stateless services, VPA for right-sizing stateful workloads
10. Enable RBAC and use least-privilege ServiceAccounts per workload
