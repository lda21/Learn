What Is MLOps?

ML + DevOps

  • Applies DevOps principles to machine learning
  • Automates the full model lifecycle
  • Bridges data science and production engineering
  • Enables reproducible, auditable ML systems
  • Treats models as versioned artifacts, not notebooks

Why It Matters

  • 87% of ML models never reach production
  • Model decay — accuracy degrades over time
  • Data drift — input distributions shift
  • Compliance — audit trail for predictions
  • Velocity — retrain and deploy in hours, not months

Real-World: Trading Agent ML Pipeline

The AI Crypto Trading Agent runs three ML models in production:

XGBoost Signal Model

Tabular features (RSI, MACD, volume profile, order flow) predict buy/sell/hold signals. Gradient-boosted trees handle feature interactions naturally.

HMM Regime Classifier

Hidden Markov Model with 4 states: TRENDING, RANGING, VOLATILE, CRISIS. Adjusts position sizing and strategy selection per regime.

Sentiment NLP

Processes news headlines, social media, and on-chain metrics. Feeds sentiment scores as features into the signal model.

Claude Post-Mortem

Claude API analyzes every closed trade: entry timing, exit reasoning, regime context. Generates actionable insights for strategy refinement.

Walk-Forward Validation

Expanding Window

  • Train on all data up to time T
  • Test on period T to T+k
  • Slide window forward, retrain, repeat
  • Mimics real-world sequential deployment
  • Prevents look-ahead bias completely

Trading Agent Config

walk_forward:
  n_splits: 12
  train_period: "52w"    # expanding
  test_period: "4w"      # 1 month OOS
  gap: "1d"              # avoid leakage
  embargo: "24h"         # post-test buffer

thresholds:
  min_sharpe: 1.2
  max_drawdown: -0.15
  min_win_rate: 0.52
  min_profit_factor: 1.3

Key rule: Never use future data in training. The gap and embargo periods prevent subtle information leakage at split boundaries.

HMM for Regime Classification

4-State Model

  • TRENDING — Clear directional momentum, wide position sizing
  • RANGING — Mean-reversion signals, tight stops
  • VOLATILE — High uncertainty, reduced exposure
  • CRISIS — Extreme drawdown risk, hedge or exit
from hmmlearn import hmm

model = hmm.GaussianHMM(
    n_components=4,
    covariance_type="full",
    n_iter=200
)

How It Works

  • States — Latent regimes (not directly observed)
  • Transitions — Probability of switching between regimes
  • Emissions — Observable features per state (returns, volatility)

Shadow Mode Deployment

How Shadow Mode Works

  • New model runs alongside production model
  • Both receive identical live market data
  • Shadow predictions are logged but not acted on
  • Compare predictions over 1-2 weeks
  • Promote only if shadow outperforms

Implementation

class ShadowDeployment:
    def predict(self, features):
        # Production model makes the trade
        prod_signal = self.prod_model.predict(
            features
        )

        # Shadow model logs prediction only
        shadow_signal = self.shadow_model.predict(
            features
        )
        self.metrics.log_shadow(
            prod=prod_signal,
            shadow=shadow_signal,
            timestamp=now()
        )

        return prod_signal  # always use prod

Zero-risk validation: Shadow mode catches regressions before they cost real money.

A/B Testing & Canary Releases

Canary Release

Route 5% of trading capital to the new model. Monitor for 48 hours. Gradually increase to 25%, 50%, 100% if metrics hold.

A/B Split

Run model A on BTC/USDT and model B on ETH/USDT. Compare risk-adjusted returns over the same period. Statistical significance required.

Automatic Rollback

If drawdown exceeds -8% during canary, auto-revert to previous model version. Alert on-call engineer via PagerDuty.

Promotion Criteria

Sharpe ratio within 0.9x of shadow period. Win rate above 52%. Max drawdown under -15%. Latency under 50ms p99.

Feature Drift Detection

What Is Feature Drift?

  • Input feature distributions shift over time
  • Model was trained on different data patterns
  • Predictions become unreliable without retraining
  • Common in crypto: regime changes, new regulations, exchange API changes

Detection Methods

# Information Coefficient (IC) monitoring
# IC = rank correlation between predicted
# signal and actual forward returns

ic_rolling = compute_ic(
    predictions, actual_returns,
    window="7d"
)

# Alert if IC decays below threshold
if ic_rolling.mean() < 0.03:
    trigger_retrain("IC decay detected")

# KS-test for distribution shift
from scipy.stats import ks_2samp
stat, p = ks_2samp(
    train_feature_dist, live_feature_dist
)
if p < 0.01:
    alert("Feature drift: RSI distribution")

Model Serving Patterns

Batch Inference

  • Run predictions on a schedule (every 5 min)
  • Pre-compute signals for all symbols
  • Store in Redis for fast lookup
  • Good for: position rebalancing, portfolio scoring
  • Latency tolerance: seconds to minutes

Real-Time Inference

  • Model loaded in memory, predict on each tick
  • Feature store provides point-in-time features
  • Feast online store for low-latency reads

  • Latency target: < 50ms end-to-end

Monte Carlo Simulation

Drawdown Estimation

  • Simulate 10,000 trial return sequences
  • Sample from historical return distribution
  • Calculate max drawdown for each trial
  • Estimate probability of ruin at various confidence levels
  • Size positions to survive 99th percentile drawdown

Implementation

import numpy as np

def monte_carlo_drawdown(
    returns, n_trials=10_000, horizon=252
):
    max_drawdowns = []
    for _ in range(n_trials):
        # Resample returns with replacement
        sim = np.random.choice(
            returns, size=horizon, replace=True
        )
        equity = np.cumprod(1 + sim)
        peak = np.maximum.accumulate(equity)
        dd = (equity - peak) / peak
        max_drawdowns.append(dd.min())

    return {
        "median_dd": np.median(max_drawdowns),
        "p95_dd": np.percentile(max_drawdowns, 5),
        "p99_dd": np.percentile(max_drawdowns, 1),
    }
# Result: {median: -12%, p95: -28%, p99: -41%}

Best Practices

1. Version everything: data, features, models, configs, and code together
2. Use walk-forward validation, never random splits for time-series data
3. Deploy in shadow mode before routing real capital to a new model
4. Monitor feature drift continuously, not just model accuracy
5. Automate retraining with quality gates (Sharpe, drawdown, win rate)
6. Use a feature store for point-in-time correctness and feature reuse
7. Run Monte Carlo simulations to size positions for worst-case scenarios
8. Keep models simple: XGBoost on clean features beats deep learning on noise
9. Log every prediction with full context for post-mortem analysis
10. Use LLMs (Claude) for qualitative analysis humans cannot scale
