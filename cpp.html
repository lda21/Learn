C# vs C++ Mental Model

Familiar from C#

  • Classes, structs, interfaces (abstract classes)
  • Inheritance & polymorphism
  • Operator overloading
  • Namespaces
  • Exceptions (try/catch/throw)
  • Lambdas & closures
  • Generics → Templates (much more powerful)
  • async/await → coroutines (C++20)

New in C++

  • No garbage collector — you manage memory
  • No runtime — compiles to native machine code
  • Pointers & raw memory access
  • RAII (deterministic resource cleanup)
  • Move semantics & rvalue references
  • Header files & separate compilation
  • Undefined behavior (the compiler trusts you)
  • Multiple inheritance

Classes — Familiar But Different

C# — Single file

public class Player {
    public string Name { get; }
    private int _health;

    public Player(string name, int hp) {
        Name = name;
        _health = hp;
    }

    // No destructor needed - GC
    // handles cleanup
}

C++ — Header + Source

// player.h
class Player {
public:
    Player(std::string name, int hp);
    ~Player();  // Destructor: deterministic!

    std::string getName() const;
private:
    std::string name_;
    int health_;
};

// player.cpp
Player::Player(std::string name, int hp)
    : name_(std::move(name)), health_(hp) {}
Player::~Player() { /* cleanup here */ }

C++ destructors are deterministic — called exactly when the object goes out of scope. No finalizers, no GC delay.

The Big Difference: Value vs Reference

C# — Classes are reference types

Player a = new Player("Alice", 100);
Player b = a;  // b points to SAME object
b.TakeDamage(10);
// a.Health is also 90!

// Classes always live on the heap
// GC cleans them up eventually

C++ — Objects are values by default

Player a("Alice", 100);
Player b = a;  // COPIES the entire object!
b.takeDamage(10);
// a.health is still 100

// a lives on the stack
// Destroyed when scope ends
// No GC needed

This changes EVERYTHING. In C++, assignment copies by default. You must explicitly use pointers or references to share objects.

References & Pointers

// C# ref parameter (closest to C++ reference)
void Heal(ref int health, int amount) {
    health += amount;  // Modifies original
}

// C++ REFERENCE: int& (alias, cannot be null, cannot be reseated)
void heal(int& health, int amount) {
    health += amount;  // Modifies original
}

// C++ POINTER: int* (memory address, CAN be null, can be reseated)
void heal(int* health, int amount) {
    if (health != nullptr) {
        *health += amount;  // Dereference to access value
    }
}

// Usage
int hp = 50;
heal(hp, 20);      // Reference: hp is now 70
heal(&hp, 10);     // Pointer: pass address with &, hp is now 80

Rule of thumb: Use const T& for read-only params, T& for output params, T* only when null is a valid value.

RAII — The Most Important C++ Concept

C# — IDisposable + using

// Only works for IDisposable objects
using var file = File.OpenRead("data.txt");
// file.Dispose() called at scope end

// Without using, you must remember:
var conn = new SqlConnection(connStr);
try {
    conn.Open();
    // use connection
} finally {
    conn.Dispose(); // must not forget!
}

C++ — RAII for everything

// Constructor acquires, destructor releases
// Works for EVERY resource automatically
{
    std::ifstream file("data.txt");
    std::lock_guard lock(mutex);
    auto conn = db::connect(connStr);
    // use file, lock, connection...

}  // ALL destructors called automatically
   // file closed, mutex unlocked,
   // connection returned to pool
   // No try/finally needed!

RAII = Resource Acquisition Is Initialization. Every resource is tied to an object's lifetime. Scope-based, deterministic, exception-safe.

Smart Pointers

// NEVER use raw new/delete in modern C++!

// unique_ptr: sole ownership (like owning a resource outright)
auto player = std::make_unique<Player>("Alice", 100);
player->takeDamage(10);
// Automatically deleted when unique_ptr goes out of scope
// Cannot be copied, only moved

// shared_ptr: reference-counted (closest to C# reference behavior)
auto boss = std::make_shared<Enemy>("Dragon", 5000);
auto bossRef = boss;  // ref count = 2
// Deleted when last shared_ptr is destroyed

// weak_ptr: non-owning observer (breaks reference cycles)
std::weak_ptr<Enemy> observer = boss;
if (auto locked = observer.lock()) {
    // object still alive, use it
}

unique_ptr

Single owner. Zero overhead vs raw pointer. Use by default.

shared_ptr

Multiple owners. Ref-counted (small overhead). Use when ownership is truly shared.

Move Semantics

// Problem: copying large objects is expensive
std::vector<int> createData() {
    std::vector<int> result(1'000'000);  // 1 million ints
    // ... fill result ...
    return result;  // MOVE, not copy! Free!
}

// std::move transfers ownership instead of copying
std::string name = "Alice";
std::string other = std::move(name);
// other is "Alice", name is now in a valid-but-unspecified state

// Rvalue references (&&) enable move semantics
class Buffer {
public:
    Buffer(Buffer&& other) noexcept       // Move constructor
        : data_(other.data_), size_(other.size_) {
        other.data_ = nullptr;              // Steal the resource
        other.size_ = 0;
    }
private:
    char* data_;
    size_t size_;
};

Why C# doesn't need this: C# reference types are already just pointers under the hood. The GC handles the rest. C++ has no GC, so move semantics let you transfer ownership without copying.

Templates — Generics on Steroids

C# Generics (runtime)

// Type info preserved at runtime
T Max<T>(T a, T b)
    where T : IComparable<T> {
    return a.CompareTo(b) > 0 ? a : b;
}

// Constraints via interfaces
// Cannot use non-type parameters
// Cannot specialize

C++ Templates (compile-time)

// Code generated at compile time
template<typename T>
T max(T a, T b) {
    return (a > b) ? a : b;
}

// Non-type parameters (C# can't do this)
template<typename T, int Size>
class FixedArray {
    T data[Size];  // Size known at compile time!
};

// Concepts (C++20) = type constraints
template<std::integral T>
T gcd(T a, T b) { ... }

Templates generate separate code for each type at compile time. More powerful than C# generics, but can cause longer compile times and cryptic error messages.

Strings

C# — Immutable, GC-managed

string name = "Alice";
string greeting = $"Hello, {name}!";
// string interpolation built-in

ReadOnlySpan<char> slice
    = name.AsSpan(0, 3);
// "Ali" without allocating

// Strings are immutable
// StringBuilder for mutations

C++ — Mutable, value type

std::string name = "Alice";
// No string interpolation (pre-C++20)
// Use std::format (C++20):
auto greeting = std::format(
    "Hello, {}!", name);

// string_view = ReadOnlySpan<char>
std::string_view sv = name;
auto slice = sv.substr(0, 3);
// "Ali" without copying

// Strings ARE mutable
name[0] = 'E';  // "Elice"
// AVOID: char* (C-style, dangerous)

Use std::string for owned strings, std::string_view for non-owning references. Never use char* in modern C++.

Inheritance & Polymorphism

// C++ uses virtual functions like C# virtual/override
class Shape {
public:
    virtual double area() const = 0;   // = 0 makes it pure virtual (abstract)
    virtual ~Shape() = default;         // ALWAYS virtual destructor in base class!
};

class Circle : public Shape {
public:
    explicit Circle(double r) : radius_(r) {}
    double area() const override { return 3.14159 * radius_ * radius_; }
private:
    double radius_;
};

// Multiple inheritance: C# doesn't have this!
class Drawable {
public:
    virtual void draw() const = 0;
};

class VisualCircle : public Circle, public Drawable {
    void draw() const override { /* render circle */ }
};

// Polymorphism requires pointers or references
std::unique_ptr<Shape> shape = std::make_unique<Circle>(5.0);
double a = shape->area();  // Calls Circle::area() via vtable

Operator Overloading

class Vec2 {
public:
    double x, y;

    // Same concept as C#, but more operators available
    Vec2 operator+(const Vec2& rhs) const {
        return {x + rhs.x, y + rhs.y};
    }

    // Subscript operator (C# uses indexers)
    double& operator[](int i) {
        return i == 0 ? x : y;
    }

    // Spaceship operator (C++20) = auto-generates ==, !=, <, >, <=, >=
    auto operator<=>(const Vec2&) const = default;

    // Stream output (like C# ToString + Console.Write)
    friend std::ostream& operator<<(std::ostream& os, const Vec2& v) {
        return os << "(" << v.x << ", " << v.y << ")";
    }

    // Function call operator (makes object callable like a functor)
    double operator()() const { return std::sqrt(x*x + y*y); }
};

Vec2 a{1, 2}, b{3, 4};
Vec2 c = a + b;           // (4, 6)
std::cout << c;           // prints "(4, 6)"
double magnitude = c();   // 7.21...

Error Handling

C# Way — Exceptions everywhere

try {
    var data = File.ReadAllText(path);
    var parsed = JsonConvert.Deserialize(data);
} catch (FileNotFoundException ex) {
    Log.Error(ex.Message);
} catch (Exception ex) {
    Log.Error(ex);
    throw;
}

// Nullable reference types
string? name = GetName();
if (name is not null) { ... }

C++ Way — Multiple strategies

// Exceptions exist but many codebases
// AVOID them (Google, Unreal, embedded)
try {
    auto data = readFile(path);
} catch (const std::exception& e) {
    std::cerr << e.what();
}

// std::optional (like Nullable<T>)
std::optional<int> findUser(int id) {
    if (!exists(id)) return std::nullopt;
    return users[id];
}

// std::expected (C++23)
std::expected<Data, Error> parse(str) {
    if (bad) return std::unexpected(err);
    return data;
}

C++ philosophy: errors are expected paths, not exceptional ones. Many teams prefer std::optional, std::expected, or error codes over exceptions.

Const Correctness

// C# has readonly fields. C++ has const EVERYWHERE.

const int MAX_HP = 100;                  // Like C# const

void printName(const std::string& name);  // Promise: won't modify name

class Player {
public:
    std::string getName() const;          // Promise: won't modify object state
    int getHealth() const { return hp_; } // const member function

    void setHealth(int hp) { hp_ = hp; } // non-const: can modify
private:
    int hp_;
};

// Pointers + const: read right-to-left
const int* p1;       // Pointer to const int (can't change *p1)
int* const p2 = &x;  // Const pointer to int (can't change p2)
const int* const p3; // Const pointer to const int (can't change either)

// const_cast exists but using it is almost always a code smell

The rule: If something shouldn't change, make it const. Functions that don't modify state should be const. Parameters you only read should be const&. More strict than C# readonly.

Lambda Expressions

C# Lambdas

// Captures automatically (closure)
int threshold = 5;
var big = numbers
    .Where(x => x > threshold)
    .ToList();

// Delegates / Func<T>
Func<int, int> square = x => x * x;

// Capture is implicit — compiler
// decides value vs reference

C++ Lambdas

// EXPLICIT capture required
int threshold = 5;
auto big = vec
    | std::views::filter(
        [threshold](int x) {
            return x > threshold;
        });

// Capture modes:
// [=]    capture all by value (copy)
// [&]    capture all by reference
// [x]    capture x by value
// [&x]   capture x by reference
// [x, &y] mix both

// Generic lambda (C++14)
auto square = [](auto x) {
    return x * x;
};

C++ forces you to think about capture. Capturing by reference to a local variable that goes out of scope = dangling reference (undefined behavior).

Modern C++ (17/20/23)

// Structured bindings (C++17) — like C# tuple deconstruction
auto [name, age] = std::pair{"Alice", 30};
for (auto& [key, value] : myMap) { ... }

// if constexpr (C++17) — compile-time branching
template<typename T>
auto stringify(T val) {
    if constexpr (std::is_arithmetic_v<T>)
        return std::to_string(val);
    else
        return val;
}

// Concepts (C++20) — like C# generic constraints but more powerful
template<typename T>
concept Printable = requires(T t) { std::cout << t; };

template<Printable T>
void print(const T& val) { std::cout << val; }

// Ranges (C++20) — like LINQ
auto result = numbers | std::views::filter([](int n){ return n % 2 == 0; })
                      | std::views::transform([](int n){ return n * n; })
                      | std::views::take(5);

// Coroutines (C++20) — like C# async/await
task<std::string> fetchData(std::string url) {
    auto response = co_await http::get(url);
    co_return response.body();
}

Memory Layout

// Stack: fast, automatic, limited size (~1-8MB)
void foo() {
    int x = 42;              // 4 bytes on stack
    std::array<int, 100> a;  // 400 bytes on stack, contiguous
}  // Everything popped off stack instantly

// Heap: slower, manual, virtually unlimited
void bar() {
    auto p = std::make_unique<Player>("Bob", 100);  // Heap allocated
}  // unique_ptr calls delete automatically

// Why C++ devs obsess over layout:
struct Bad {          struct Good {
    char a;   // 1B       int b;    // 4B
    // 3B pad             char a;   // 1B
    int b;    // 4B       char c;   // 1B
    char c;   // 1B       // 2B pad
    // 3B pad
};  // 12 bytes!     };  // 8 bytes!

// Cache-friendly: vector<Player> (contiguous) beats list<Player*> (scattered)
// A cache miss costs ~200 CPU cycles. Layout matters for performance.

Common Pitfalls for C# Devs

1 Dangling reference — returning a reference to a local variable. Object is destroyed, reference points to garbage.
2 Object slicing — passing Derived by value to Base param. The derived part is chopped off. Use references or pointers.
3 Forgetting virtual destructor — deleting a derived object through a base pointer leaks the derived part. Always virtual ~Base() = default;.
4 Iterator invalidation — modifying a vector while iterating can invalidate iterators. Unlike C# which throws an exception, C++ is silently UB.
5 Undefined behavior — out-of-bounds access, null dereference, use-after-free. No exceptions thrown — anything can happen (including "working fine" until production).
6 Using raw new/delete — leads to leaks. Always use smart pointers and RAII containers.
7 Copying when you mean to reference — passing large objects by value in a loop. Use const&.
8 Forgetting const — C++ culture expects const-correctness. Mark everything that shouldn't mutate.

When to Use C++

Game Engines

Unreal Engine, Unity internals, custom engines. Deterministic performance, no GC pauses.

Systems Software

Browsers (Chrome, Firefox), databases (MySQL, MongoDB), operating systems, drivers.

High-Frequency Trading

Nanosecond latency. Zero allocations in hot paths. Cache-optimized data structures.

Embedded & Real-Time

IoT, automotive, medical devices. Predictable memory, no runtime overhead.

Stay in C# when: business apps, web APIs, cloud services, rapid prototyping, team productivity matters more than raw performance.

Interop: Use extern "C" for C-compatible functions, P/Invoke from C#, or C++/CLI for .NET integration.
