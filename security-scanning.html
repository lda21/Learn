Trivy — The Swiss Army Knife

Open-source by Aqua Security. Scans container images, filesystems, git repos, and IaC (Terraform, K8s, Helm, CloudFormation).

# Scan a container image for vulnerabilities
trivy image nginx:latest

# Scan only HIGH and CRITICAL
trivy image --severity HIGH,CRITICAL python:3.12

# Scan local filesystem (project dependencies)
trivy fs --severity HIGH,CRITICAL .

# Scan IaC files (Terraform, K8s manifests, Dockerfiles)
trivy config ./terraform/
trivy config ./k8s-manifests/

# Scan a remote git repository
trivy repo https://github.com/org/app

# Exit code 1 if vulnerabilities found (for CI/CD)
trivy image --exit-code 1 --severity CRITICAL myapp:latest

Severity levels: UNKNOWN LOW MEDIUM HIGH CRITICAL. The most versatile free tool available.

Trivy in CI/CD

GitHub Actions Workflow

# .github/workflows/trivy-scan.yml
name: Trivy Security Scan
on: [pull_request]

jobs:
  trivy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build Docker image
        run: docker build -t myapp:${{ github.sha }} .

      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'myapp:${{ github.sha }}'
          format: 'sarif'
          output: 'trivy-results.sarif'
          severity: 'HIGH,CRITICAL'
          exit-code: '1'

      - name: Upload to GitHub Security tab
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: 'trivy-results.sarif'

GitLab CI

# .gitlab-ci.yml
trivy-scan:
  image: aquasec/trivy:latest
  script:
    - trivy image --exit-code 1 --severity HIGH,CRITICAL $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA

Prowler — Cloud Security Posture

Open-source. Audits AWS, Azure, GCP for misconfigurations. 300+ checks across compliance frameworks.

# Scan AWS account (all checks)
prowler aws

# Scan against CIS Benchmarks 2.0
prowler aws --compliance cis_2.0_aws

# Scan specific services only
prowler aws --service s3 iam ec2

# Scan Azure
prowler azure

# Scan GCP
prowler gcp

# Output formats
prowler aws --output-formats csv json html

# Example findings:

# FAIL  IAM root account has no MFA configured

CIS Benchmarks PCI-DSS HIPAA SOC2 GDPR NIST 800-53

Schedule weekly runs and export dashboard reports for compliance audits.

Secrets Detection — Gitleaks & TruffleHog

Gitleaks (Regex-based, Fast)

# Scan current directory
gitleaks detect --source .

# Scan git history
gitleaks detect --source . --log-opts="--all"

# Pre-commit hook (prevent leaks!)
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.18.0
    hooks:
      - id: gitleaks

# Example output:
# Finding: AWS Access Key
# Secret:  AKIA...EXAMPLE
# File:    src/config.js
# Line:    42
# Commit:  a1b2c3d

TruffleHog (Entropy + Regex)

# Scan git repo (entire history)
trufflehog git file://. --only-verified

# Scan a GitHub org
trufflehog github --org=mycompany

# Scan filesystem
trufflehog filesystem --directory=./src

# What they catch:
# - AWS Access Keys (AKIA...)
# - GitHub tokens (ghp_...)
# - Database passwords
# - Private keys (RSA/EC)
# - Slack webhooks
# - Stripe API keys
# - SendGrid tokens

TruffleHog verifies secrets are active. Deeper but slower than Gitleaks.

SAST — Semgrep

Static Application Security Testing. Scans YOUR source code for vulnerabilities. Pattern-based, supports 30+ languages, custom rules.

# Run with auto-detected rules
semgrep scan --config auto

# Run OWASP Top 10 rules
semgrep scan --config p/owasp-top-ten

# Scan specific language
semgrep scan --config p/python .

# CI mode (machine-readable output)
semgrep ci --config auto

What Semgrep Catches

# SQL Injection (Python example)
# VULNERABLE:
cursor.execute("SELECT * FROM users WHERE id = " + user_id)
# Semgrep flags: "Detected string concatenation in SQL query"

# XSS (JavaScript example)
# VULNERABLE:
document.innerHTML = userInput;
# Semgrep flags: "User-controlled data in innerHTML"

# Hardcoded secrets
# VULNERABLE:
API_KEY = "sk-live-abc123def456"
# Semgrep flags: "Hardcoded secret detected"

# Insecure crypto
# VULNERABLE:
hashlib.md5(password.encode())
# Semgrep flags: "Use of weak hash function MD5"

Compare: CodeQL (deeper analysis, slower) vs Semgrep (pattern-based, faster, easier custom rules).

SAST — CodeQL (GitHub)

GitHub's built-in code analysis. Treats code as a database and runs semantic queries. Free for public repos, included in GitHub Advanced Security.

# .github/workflows/codeql.yml
name: CodeQL Analysis
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 6 * * 1'    # Weekly Monday 6AM

jobs:
  analyze:
    runs-on: ubuntu-latest
    permissions:
      security-events: write
    strategy:
      matrix:
        language: ['javascript', 'python']

    steps:
      - uses: actions/checkout@v4

      - name: Initialize CodeQL
        uses: github/codeql-action/init@v3
        with:
          languages: ${{ matrix.language }}
          queries: +security-extended

      - name: Autobuild
        uses: github/codeql-action/autobuild@v3

      - name: Perform CodeQL Analysis
        uses: github/codeql-action/analyze@v3

Catches: injection, auth bypass, insecure deserialization, path traversal, SSRF. Results appear in the GitHub Security tab.

DAST — OWASP ZAP

Dynamic Application Security Testing. Tests a RUNNING application by sending actual HTTP requests. Free, open-source.

# Quick scan with Docker
docker run --rm -t owasp/zap2docker-stable zap-baseline.py \
  -t http://host.docker.internal:3000

# Full active scan (more thorough, more intrusive)
docker run --rm -t owasp/zap2docker-stable zap-full-scan.py \
  -t http://host.docker.internal:3000

# API scan (OpenAPI/Swagger)
docker run --rm -t owasp/zap2docker-stable zap-api-scan.py \
  -t http://host.docker.internal:3000/openapi.json \
  -f openapi

# Generate HTML report
docker run --rm -v $(pwd):/zap/wrk owasp/zap2docker-stable \
  zap-baseline.py -t http://target:3000 \
  -r report.html

Passive Scan

Observes traffic without modifying requests. Safe for production. Finds: missing headers, cookie flags, info leaks.

Active Scan

Sends attack payloads (SQLi, XSS, etc). Never run on production. Finds: injection, CSRF, auth bypass.

Compare: Burp Suite (commercial, more powerful, manual testing) vs ZAP (free, automation-friendly).

Checkov — IaC Security

By Bridgecrew/Palo Alto. Scans Terraform, CloudFormation, K8s manifests, Helm charts, Dockerfiles. 1000+ built-in policies.

# Scan Terraform directory
checkov -d ./terraform/

# Scan specific file
checkov -f main.tf

# Scan Kubernetes manifests
checkov -d ./k8s/ --framework kubernetes

# Scan Dockerfile
checkov --dockerfile-path ./Dockerfile

# Output in JSON for CI
checkov -d ./terraform/ -o json

# Skip specific checks
checkov -d . --skip-check CKV_AWS_18,CKV_AWS_21

What Checkov Catches

# FAIL: S3 bucket without encryption
# CKV_AWS_19: "Ensure S3 bucket has server-side encryption"

# FAIL: RDS without encryption at rest
# CKV_AWS_16: "Ensure RDS instance has encryption enabled"

# FAIL: Security group allows 0.0.0.0/0 on port 22
# CKV_AWS_24: "Ensure no security group allows SSH from 0.0.0.0/0"

# FAIL: IAM policy with wildcard actions
# CKV_AWS_1: "Ensure IAM policy does not have * action"

Checkov has 1000+ policies vs Trivy config (fewer but growing). Use both for maximum coverage.

Falco — Runtime Security

Detects anomalous behavior in running containers. Watches syscalls via eBPF in real-time. Low overhead. Deployed as a DaemonSet on K8s.

# Install Falco on Kubernetes via Helm
helm repo add falcosecurity https://falcosecurity.github.io/charts
helm install falco falcosecurity/falco \
  --set driver.kind=ebpf \
  --set falcosidekick.enabled=true

# Example Falco rules:
- rule: Terminal shell in container
  desc: A shell was spawned in a container
  condition: >
    spawned_process and container and
    proc.name in (bash, sh, zsh)
  output: >
    Shell spawned in container
    (user=%user.name container=%container.name
     shell=%proc.name parent=%proc.pname)
  priority: WARNING

What Falco Detects

Shell Access

Interactive shell spawned inside a container (kubectl exec, docker exec)

Network Anomalies

Unexpected outbound connections, reverse shells, DNS exfiltration

File Tampering

Writes to /etc/shadow, /etc/passwd, sensitive config files

Privilege Escalation

setuid calls, namespace escapes, capability abuse

Sysdig — Commercial Falco on Steroids

Sysdig created Falco (donated to CNCF) and built a commercial platform on top:

  • Sysdig Secure — Falco runtime + image scanning + cloud posture + K8s security + compliance (SOC2, PCI, HIPAA) in one dashboard
  • Sysdig Capture — records every syscall like a flight recorder. After an incident, replay what files were read, what connections were made, what commands ran
  • Sysdig Monitor — Prometheus-compatible metrics and dashboards
  • Cost: ~$25-50/node/month. Worth it at 50+ nodes when managing 5 separate tools becomes a burden
  • When Sysdig over DIY: regulated industry, post-incident forensics, large fleet, security team wants one vendor

SBOM — Software Bill of Materials

A complete inventory of every component in your software. Required by US Executive Order 14028 for government software.

# Generate SBOM with Trivy (CycloneDX format)
trivy image --format cyclonedx --output sbom.json myapp:latest

# Generate SBOM with Syft (SPDX format)
syft myapp:latest -o spdx-json > sbom-spdx.json

# Generate SBOM from a directory
syft dir:./my-project -o cyclonedx-json > sbom.json

# Scan an existing SBOM for vulnerabilities
grype sbom:./sbom.json

# Combine: generate + scan in CI
syft myapp:latest -o cyclonedx-json | grype

Formats

  • SPDX — Linux Foundation standard, ISO/IEC 5962
  • CycloneDX — OWASP standard, richer vulnerability data

Why SBOMs Matter

  • Log4Shell — "Do we use Log4j?" answered instantly
  • SolarWinds — supply chain visibility
  • Customer/compliance transparency

Container Image Hardening

Before (150+ CVEs)

FROM node:20

WORKDIR /app
COPY . .
RUN npm install
EXPOSE 3000
CMD ["node", "server.js"]

# trivy image myapp:before
# Total: 158 vulnerabilities
#   CRITICAL: 12
#   HIGH:     34
#   MEDIUM:   67
#   LOW:      45

After (0 CVEs)

# Stage 1: Build
FROM node:20-slim AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .

# Stage 2: Runtime (distroless)
FROM gcr.io/distroless/nodejs20
WORKDIR /app
COPY --from=build /app .
USER nonroot:nonroot
EXPOSE 3000
CMD ["server.js"]

# trivy image myapp:after
# Total: 0 vulnerabilities
1 Use distroless or Alpine base images
2 Multi-stage builds — separate build from runtime
3 Run as non-root user (USER nonroot)
4 Use npm ci not npm install for reproducibility
5 Set read-only root filesystem in K8s
6 Pin image digests: node:20@sha256:abc...

Building a Security Pipeline

Complete GitHub Actions workflow combining multiple scanners with SARIF upload.

# .github/workflows/security-pipeline.yml
name: Security Pipeline
on: [pull_request]

jobs:
  secrets:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - uses: gitleaks/gitleaks-action@v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

  sast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: returntocorp/semgrep-action@v1
        with:
          config: >-
            p/owasp-top-ten
            p/security-audit

  iac:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Checkov IaC scan
        uses: bridgecrewio/checkov-action@master
        with:
          directory: ./terraform/
          soft_fail: false

  container:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t app:${{ github.sha }} .
      - name: Trivy scan
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: 'app:${{ github.sha }}'
          format: 'sarif'
          output: 'trivy.sarif'
          severity: 'HIGH,CRITICAL'
          exit-code: '1'
      - uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: 'trivy.sarif'

Quick Decision Guide

Start with Trivy + Gitleaks + Semgrep. These three free tools cover 80% of security scanning needs. Add tools as your team, compliance requirements, and attack surface grow.

Rule of thumb: One tool per category. Overlap is OK for defense in depth, but avoid tool fatigue.

🛡

Course Complete

  • No single tool covers everything — build a layered stack
  • Shift left: catch issues in IDE and pre-commit, not production
  • Start free: Trivy Gitleaks Semgrep
  • Automate scanning in CI/CD pipelines — never rely on manual checks
  • Manage vulnerabilities: triage, prioritize, fix with SLAs
  • Generate SBOMs for supply chain transparency
  • Add runtime protection (Falco) for defense in depth

Security is a journey, not a destination. Scan continuously.
