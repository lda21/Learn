What is MCP?

Model Context Protocol

  • An open standard for connecting AI models to external tools, data, and services
  • Created by Anthropic, now an open specification
  • Think of it as USB-C for AI — one protocol, many integrations
  • JSON-RPC 2.0 based — request/response + notifications
  • Replaces ad-hoc tool integrations with a universal standard

Why It Matters

  • Before MCP: Every AI app builds custom integrations (N×M problem)
  • After MCP: Build once, connect everywhere (N+M)
  • Server authors expose capabilities once
  • Client authors connect once
  • Adopted by Claude, Cursor, VS Code, Windsurf, Cline, and more

Key Roles

  • Host — application that needs AI capabilities (Claude Desktop, IDEs)
  • Client — maintains 1:1 connection with a server
  • Server — exposes tools, resources, and prompts via MCP

Protocol Basics

  • Built on JSON-RPC 2.0
  • Stateful connections with capability negotiation
  • Client and server declare capabilities during initialize
  • Supports bi-directional communication

MCP Primitives

🔧 Tools

Functions the model can invoke. Server exposes, model decides when to call. Requires user approval.

tools/list → tools/call

📁 Resources

Data the client can read. File contents, DB records, API responses. Application-controlled.

resources/list → resources/read

📝 Prompts

Reusable prompt templates with arguments. User-controlled — selected explicitly.

prompts/list → prompts/get

🔔 Notifications

One-way messages for events. No response expected. Resource changes, progress updates, log messages.

notifications/*

Tools — Model-Controlled Actions

How Tools Work

  • Server declares available tools with JSON Schema inputs
  • Model decides when to call a tool based on context
  • Client sends tools/call with arguments
  • Server executes and returns results
  • Can return text, images, or embedded resources

Example: GitHub Tool

// Server declares tool
{
  "name": "create_issue",
  "description": "Create a GitHub issue",
  "inputSchema": {
    "type": "object",
    "properties": {
      "repo": {"type": "string"},
      "title": {"type": "string"},
      "body": {"type": "string"}
    },
    "required": ["repo", "title"]
  }
}

// Client calls tool
{"method": "tools/call",
 "params": {
   "name": "create_issue",
   "arguments": {
     "repo": "org/repo",
     "title": "Bug in auth"
   }
 }
}

Resources & Prompts

Resources

  • Identified by URIs (e.g., file:///path, postgres://db/table)
  • Can be text or binary (base64-encoded)
  • Static resources — listed via resources/list
  • Dynamic resources — discovered via resources/templates with URI templates
  • Servers can notify clients when resources change
  • Subscriptions — clients subscribe to specific resource updates

Prompts

  • Pre-defined message templates with typed arguments
  • Can produce multi-turn conversations (user + assistant messages)
  • Can embed resources in the generated messages
  • User-controlled — explicitly selected, not auto-triggered
// Prompt with arguments
{
  "name": "explain_code",
  "arguments": [
    {"name": "language", "required": true},
    {"name": "code", "required": true}
  ]
}

stdio Transport

  • Simplest — no network, no auth needed
  • Server lifecycle managed by client
  • stderr used for logging (not protocol)
  • Used by Claude Desktop, Claude Code, Cursor

Streamable HTTP Transport

  • Single endpoint accepts POST requests
  • Server may respond with JSON or SSE stream
  • Optional Mcp-Session-Id header for state
  • Supports resumability via Last-Event-ID
  • Requires authentication for remote access

Server Capabilities

  • tools — supports tools/list and tools/call
  • resources — supports resource reading and subscriptions
  • prompts — supports prompt templates
  • logging — can emit log messages
  • completions — supports argument auto-complete

Client Capabilities

  • roots — can provide filesystem roots
  • sampling — supports LLM sampling requests
  • elicitation — can prompt user for input

Sampling & Elicitation

Sampling (Server → LLM)

  • Server can request the AI model to generate text
  • Enables agentic behaviors — server-driven reasoning
  • Client maintains human-in-the-loop control
  • Server sends sampling/createMessage
  • Client routes to LLM, may modify or reject
  • Supports maxTokens, temperature, model preferences

Elicitation (Server → User)

  • Server can ask the user for input directly
  • Structured forms with JSON Schema validation
  • Client renders appropriate UI (text, confirm, select)
  • Server sends elicitation/create
  • Results: accepted, declined, or cancelled
// Server asks user to confirm
{"method": "elicitation/create",
 "params": {
   "message": "Deploy to production?",
   "requestedSchema": {
     "type": "object",
     "properties": {
       "confirm": {
         "type": "boolean"
       }
     }
   }
 }
}

Authentication for MCP

Why Auth Matters

  • stdio transport — no auth needed (local process)
  • HTTP transport — auth is critical for remote servers
  • MCP spec mandates OAuth 2.1 as the standard authorization framework
  • Servers can also accept API keys, tokens, or other schemes
  • Auth is negotiated via HTTP headers, not the JSON-RPC layer

Auth Methods Landscape

  • OAuth 2.1 — MCP’s official standard (recommended)
  • OAuth 2.0 — widely deployed, predecessor to 2.1
  • OAuth 1.0a — legacy, signature-based
  • API Keys — simple but limited
  • mTLS — certificate-based mutual auth
  • SAML 2.0 — enterprise SSO (XML-based)
  • Basic / Digest — HTTP-native, rarely used alone
  • OIDC — identity layer on top of OAuth 2.0

OAuth 1.0a — The Original

How It Works

  • Released 2007, defined in RFC 5849
  • Signature-based — every request signed with HMAC-SHA1
  • No bearer tokens — client must prove possession of secret
  • Three-legged flow: request token → authorize → access token
  • Tokens never expire by default
Authorization: OAuth
  oauth_consumer_key="app123",
  oauth_token="user456",
  oauth_signature_method="HMAC-SHA1",
  oauth_signature="kYjzVBB8Y...",
  oauth_timestamp="1318622958",
  oauth_nonce="kllo9940pd9333jh"

Pros & Cons

  • Secure without HTTPS — signatures protect integrity
  • No token leakage risk — signatures are request-specific
  • Complex implementation — signature base string is error-prone
  • Not scalable — no scopes, no token refresh
  • Deprecated — replaced by OAuth 2.0

Still used by: Twitter/X API v1.1, some legacy enterprise systems

OAuth 2.0 — The Industry Standard

Core Concepts

  • Released 2012, defined in RFC 6749
  • Bearer tokens — whoever has the token can use it
  • Scopes — limit what a token can access
  • Refresh tokens — get new access tokens without re-auth
  • Requires HTTPS for security (no signatures)
  • Multiple grant types for different use cases

Grant Types

  • Authorization Code — web apps, most secure flow
  • Client Credentials — machine-to-machine, no user
  • Implicit — SPAs (now deprecated)
  • Resource Owner Password — direct login (now deprecated)
  • Device Code — TVs, CLIs, IoT (RFC 8628)
  • Refresh Token — renew access silently

Step by Step

  • 1. Client redirects user to auth server with client_id, redirect_uri, scope
  • 2. User authenticates and grants consent
  • 3. Auth server redirects back with authorization code
  • 4. Client exchanges code for access token (+ refresh token)
  • 5. Client uses access token in Authorization: Bearer header

Security Considerations

  • PKCE required for public clients (RFC 7636)
  • State parameter prevents CSRF attacks
  • Short-lived access tokens (minutes to hours)
  • Refresh tokens stored securely server-side
  • redirect_uri must be exact match

OAuth 2.1 — MCP’s Standard

What Changed from 2.0

  • PKCE required for ALL clients (not just public)
  • Implicit grant removed — was vulnerable to token leakage
  • Password grant removed — anti-pattern for modern apps
  • Redirect URI exact match — no more wildcard matching
  • Refresh token rotation — one-time use, detect replay
  • Consolidates best practices from OAuth 2.0 Security BCP

PKCE (Proof Key for Code Exchange)

// 1. Client generates random verifier
code_verifier = random(43-128 chars)

// 2. Client creates challenge
code_challenge = BASE64URL(
  SHA256(code_verifier)
)

// 3. Send challenge in /authorize
GET /authorize?
  code_challenge=E9Melhoa2O...
  &code_challenge_method=S256

// 4. Send verifier in /token
POST /token
  code_verifier=dBjftJeZ4CVP...

Prevents authorization code interception attacks

Discovery & Registration

  • Client fetches /.well-known/oauth-authorization-server from MCP server
  • Gets authorization, token, and registration endpoints
  • Uses Dynamic Client Registration (RFC 7591) — no pre-registration needed
  • Falls back to /.well-known/openid-configuration for OIDC servers

Token Usage

  • Access token sent via Authorization: Bearer header
  • Server validates token on every HTTP request
  • On 401, client must re-authorize
  • Third-party auth servers supported — MCP server may delegate to Auth0, Okta, Azure AD
  • Resource metadata at /.well-known/oauth-protected-resource (RFC 9728)

Other Authentication Methods

API Keys

Simple string token in header (X-API-Key) or query param. No expiry, no scopes, no refresh. Best for internal/dev use.

mTLS (Mutual TLS)

Both client and server present X.509 certificates. Strongest transport-level auth. Used in banking, healthcare, zero-trust networks.

SAML 2.0

XML-based SSO protocol. IdP sends signed assertions to SP. Dominant in enterprise (Okta, Azure AD, PingFederate). Heavy, complex.

Basic / Digest Auth

HTTP-native. Basic sends base64 credentials (requires HTTPS). Digest uses challenge-response with nonce. Legacy — rarely used alone.

OpenID Connect (OIDC)

Identity on Top of OAuth

  • OAuth 2.0 = authorization (what can you do?)
  • OIDC = authentication (who are you?)
  • Adds an ID Token (JWT) to the OAuth flow
  • Standardizes /userinfo endpoint for profile data
  • Discovery via /.well-known/openid-configuration
  • Used by Google, Microsoft, Apple, Okta, Auth0, Keycloak

Building an MCP Server

TypeScript (Official SDK)

import { McpServer } from
  "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from
  "@modelcontextprotocol/sdk/server/stdio.js";

const server = new McpServer({
  name: "my-server",
  version: "1.0.0"
});

server.tool("greet",
  { name: { type: "string" } },
  async ({ name }) => ({
    content: [{
      type: "text",
      text: "Hello " + name
    }]
  })
);

const transport = new StdioServerTransport();
await server.connect(transport);

Python (Official SDK)

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("my-server")

@mcp.tool()
def greet(name: str) -> str:
    """Greet a user by name."""
    return f"Hello {name}"

@mcp.resource("config://app")
def get_config() -> str:
    """Return app configuration."""
    return json.dumps(config)

@mcp.prompt()
def review_code(code: str) -> str:
    """Create a code review prompt."""
    return f"Review this code:\n{code}"

# Run with stdio transport
mcp.run()
