Why Risk Management?

Capital Preservation

  • A 50% loss requires a 100% gain to recover
  • Surviving drawdowns is more important than maximizing returns
  • Markets reward those who stay in the game longest
  • One catastrophic loss can wipe out years of profits

Systematic Discipline

  • Removes emotion — rules execute regardless of fear or greed
  • Consistent execution — same rules every trade, every time
  • Auditable — every decision has a traceable reason
  • Scalable — works for $10K or $10M portfolios

Rule #1: Never lose money. Rule #2: Never forget Rule #1.

Position Sizing

Percent Risk Per Trade

Risk a fixed percentage of equity on each trade. Standard: 1.5% max per trade.

Position Size = (Equity x Risk%) / Stop Distance

ATR-Based Sizing

Use Average True Range to adapt size to volatility. High ATR = smaller position, low ATR = larger.

Size = Risk$ / (ATR x Multiplier)

Kelly Criterion

Mathematically optimal bet size based on win rate and payoff ratio.

f* = (bp - q) / b

where b=odds, p=win%, q=loss%

Fractional Kelly

Use 25-50% of full Kelly to reduce variance. Full Kelly is too aggressive for real trading.

Practical Size = 0.25 x Kelly

Drawdown Management

24-Hour Rolling Drawdown

  • Track peak equity over rolling 24h window
  • Threshold: 3% max drawdown
  • Action: HALT — stop opening new positions
  • Existing positions allowed to close normally
  • Auto-resume when drawdown recovers below 2%

Total Drawdown from Peak

  • Track all-time high-water mark
  • Threshold: 10% max drawdown
  • Action: SHUTDOWN — flatten everything
  • Requires manual restart and review
  • Signals potential strategy breakdown
# Drawdown calculation
peak_24h = max(equity_snapshots[-24h:])
current_dd = (peak_24h - current_equity) / peak_24h

if current_dd >= 0.03:
    risk_gate.halt("24h drawdown breached: {:.2%}".format(current_dd))

hwm = max(all_time_equity_peaks)
total_dd = (hwm - current_equity) / hwm

if total_dd >= 0.10:
    risk_gate.shutdown("Total drawdown breached: {:.2%}".format(total_dd))

Concentration & Correlation Limits

Token Concentration

  • No single token > 10% of portfolio
  • Prevents blowup from one asset collapse
  • Checked against notional value, not count
  • Applies to both long and short exposure
  • Rebalance alerts at 8% threshold

BTC Correlation Check

  • New positions must have correlation ≤ 0.7 with BTC
  • Crypto is highly correlated in crashes
  • Rolling 30-day Pearson correlation
  • Prevents portfolio becoming a leveraged BTC bet
  • Recalculated every 15 minutes
# Correlation check before new position
btc_returns = get_returns("BTC", window=30*24*60)
token_returns = get_returns(token, window=30*24*60)
corr = pearson_correlation(btc_returns, token_returns)

if corr > 0.7:
    return RiskReject(
        reason="BTC_CORRELATION_TOO_HIGH",
        detail=f"{token} correlation with BTC: {corr:.2f} > 0.70"
    )

Leverage Controls

Gross Leverage Max 2x

Sum of absolute position values divided by equity. |longs| + |shorts| ≤ 2x equity

Margin Requirements

Maintain 50% free margin minimum. Never let margin usage exceed exchange maintenance threshold.

Liquidation Prevention

Pre-calculate liquidation price for every position. Auto-reduce if price within 5% of liquidation.

Net Exposure Limit

Net long/short exposure capped at 1.5x. Prevents directional over-commitment.

# Leverage check
gross_exposure = sum(abs(pos.notional) for pos in positions)
equity = account.total_equity
gross_leverage = gross_exposure / equity

if gross_leverage + new_order.notional / equity > 2.0:
    return RiskReject(
        reason="LEVERAGE_EXCEEDED",
        detail=f"Gross leverage would be {gross_leverage:.2f}x > 2.0x"

Funding Rate Risk

What Are Funding Rates?

  • Perpetual futures have no expiry date
  • Funding rate keeps perp price close to spot
  • Paid every 8 hours (or 1h on some exchanges)
  • Positive rate = longs pay shorts
  • Negative rate = shorts pay longs
  • At 0.1%/8h, that is ~36.5%/year drag

Monitoring Thresholds

  • > 0.01%/8h — Normal, no action
  • > 0.05%/8h — Log and monitor closely
  • > 0.1%/8h — Alert: reduce position size
  • > 0.5%/8h — Auto-close: cost is too high
  • Track cumulative funding cost per position
  • Include funding in P&L calculations
# Funding rate check
funding_rate = exchange.get_funding_rate(token)

if abs(funding_rate) > 0.005:    # 0.5% per 8h
    kill_switch.close_position(token, reason="EXTREME_FUNDING")
elif abs(funding_rate) > 0.001:  # 0.1% per 8h
    alerting.send("HIGH_FUNDING", token=token, rate=funding_rate)
    risk_gate.reduce_max_size(token, multiplier=0.5)

The Kill Switch

Emergency halt: flatten all positions in < 30 seconds

Manual Triggers

  • Dashboard panic button
  • CLI command: risk kill --confirm
  • Slack bot command
  • Phone call escalation to on-call

Automatic Triggers

  • Total drawdown > 10% from peak
  • Data feed stale > 5 minutes
  • Exchange API errors > 50% of requests
  • Reconciliation divergence > 2%
  • System health check failure
# Kill switch execution flow
async def kill_switch_activate(reason: str):

Reconciliation

Internal Model vs Exchange

  • Trading system tracks positions internally
  • Exchange is the source of truth
  • Drift can occur from partial fills, network issues, manual trades
  • Unreconciled positions = hidden risk

5-Minute Check Cycle

  • Every 5 minutes: fetch exchange positions
  • Compare against internal position model
  • Flag divergence > 0.1% for review
  • Auto-correct if divergence is small (< 0.5%)
  • Kill switch if divergence > 2%
# Reconciliation loop (every 5 minutes)
async def reconcile():
    exchange_positions = await exchange.get_all_positions()
    internal_positions = position_store.get_all()

    for token in all_tokens:
        exchange_qty = exchange_positions.get(token, 0)
        internal_qty = internal_positions.get(token, 0)
        divergence = abs(exchange_qty - internal_qty) / max(abs(internal_qty), 1)

        if divergence > 0.02:
            await kill_switch_activate(f"RECON_DIVERGENCE: {token} {divergence:.2%}")
        elif divergence > 0.001:
            position_store.correct(token, exchange_qty)
            alert.warn(f"Recon corrected {token}: {internal_qty} -> {exchange_qty}")

Loss Streak & Regime Adaptation

Loss Streak Tracking

After 3 consecutive losses, reduce position size to 0.5x normal. Reset counter on first win.

Weekend Sizing

Reduced liquidity on weekends. Apply 0.75x size multiplier Friday 20:00 UTC through Monday 04:00 UTC.

High Volatility Regime

When ATR exceeds 2x its 30-day average, restrict new positions to 0.5x max size.

Multiplier Stacking

Multipliers compound: weekend (0.75) + loss streak (0.5) = 0.375x normal size.

# Size multiplier calculation
def get_size_multiplier(token: str) -> float:
    multiplier = 1.0

    if consecutive_losses >= 3:
        multiplier *= 0.5   # Loss streak reduction

    if is_weekend():
        multiplier *= 0.75  # Weekend reduction

    if get_atr(token) > 2 * get_atr_avg(token, days=30):
        multiplier *= 0.5   # High volatility reduction

    return multiplier

Monte Carlo & Stress Testing

Monte Carlo Simulation

  • Simulate 10,000 trade sequences
  • Shuffle actual trade results randomly
  • Build distribution of outcomes
  • Find worst-case drawdown at 95th percentile
  • Validate that risk limits survive tail scenarios

Stress Scenarios

  • Flash crash: -20% in 5 minutes
  • Exchange halt: no fills for 30 minutes
  • Correlation spike: all alts move with BTC
  • Liquidity dry-up: spreads widen 10x
  • API outage: cannot cancel orders
# Monte Carlo drawdown estimation
import numpy as np

def monte_carlo_max_drawdown(trade_returns, n_simulations=10000):
    results = []
    for _ in range(n_simulations):
        shuffled = np.random.permutation(trade_returns)
        equity_curve = np.cumprod(1 + shuffled)
        running_max = np.maximum.accumulate(equity_curve)
        drawdowns = (running_max - equity_curve) / running_max
        results.append(drawdowns.max())

    p95 = np.percentile(results, 95)
    p99 = np.percentile(results, 99)
    return {"p95_max_dd": p95, "p99_max_dd": p99, "worst": max(results)}

Risk Parameter Hot-Reload

Why Hot-Reload?

  • Markets change faster than deploy cycles
  • Tighten limits during high volatility without restart
  • Loosen limits after a calm period
  • Zero downtime = zero missed risk events
  • Audit trail of every parameter change

Implementation

  • Store parameters in Redis/etcd/ConfigMap
  • Risk gate watches for changes (pub/sub)
  • New values take effect on next check
  • Validation layer prevents invalid configs
  • Rollback to previous config on failure
# risk_config.yaml (hot-reloadable)
risk_limits:

  funding_alert_rate: 0.001     # 0.1% per 8h
  funding_close_rate: 0.005     # 0.5% per 8h

Best Practices

1. Risk gate is deterministic — no ML, no discretion, 100% rules
2. Never risk more than 1.5% equity on a single trade
3. Monitor drawdowns at two levels: 24h rolling (3%) and total (10%)
4. Reconcile positions with exchange every 5 minutes
5. Build a kill switch that flattens everything in under 30 seconds
6. Treat stale data as a risk event — progressive response from reject to flatten
7. Track funding rates and auto-close on extreme costs (>0.5%/8h)
8. Run Monte Carlo simulations to validate parameter choices
9. Make risk parameters hot-reloadable with full audit trail
10. Dashboard everything — if you cannot see it, you cannot manage it
