Why PostgreSQL?

Features

  • Full ACID compliance
  • Advanced SQL: CTEs, window functions, lateral joins
  • JSONB for document-style queries
  • Full-text search built in
  • Extensions: PostGIS, pgvector, TimescaleDB

When to Use

  • Web applications (Django, Rails, Next.js)
  • Complex queries and reporting
  • OLTP workloads at scale
  • GIS / geospatial data (PostGIS)
  • AI/ML vector search (pgvector)

Tables & Data Types

CREATE TABLE users (
  id          SERIAL PRIMARY KEY,       -- auto-increment integer
  uuid        UUID DEFAULT gen_random_uuid(),
  name        VARCHAR(100) NOT NULL,
  email       TEXT UNIQUE NOT NULL,
  age         INTEGER CHECK (age >= 0),
  role        VARCHAR(20) DEFAULT 'user',
  metadata    JSONB DEFAULT '{}',
  tags        TEXT[] DEFAULT '{}',       -- array type
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  updated_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Common types: INTEGER, BIGINT, NUMERIC(10,2), TEXT, VARCHAR(n),
-- BOOLEAN, DATE, TIMESTAMP, TIMESTAMPTZ, UUID, JSONB, ARRAY,
-- INET, CIDR, BYTEA, INTERVAL, SERIAL, BIGSERIAL

CRUD Operations

-- INSERT
INSERT INTO users (name, email, role)
VALUES ('Alice', 'alice@example.com', 'admin')
RETURNING id, uuid, created_at;

-- SELECT
SELECT name, email, created_at
FROM users
WHERE role = 'admin' AND age > 25
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;

-- UPDATE
UPDATE users
SET role = 'admin', updated_at = NOW()
WHERE email = 'alice@example.com'
RETURNING *;

-- DELETE
DELETE FROM users WHERE id = 42 RETURNING *;

-- UPSERT (insert or update on conflict)
INSERT INTO users (email, name) VALUES ('alice@ex.com', 'Alice')
ON CONFLICT (email) DO UPDATE SET name = EXCLUDED.name, updated_at = NOW();

Joins & Relationships

-- Foreign key
CREATE TABLE posts (
  id       SERIAL PRIMARY KEY,
  user_id  INTEGER REFERENCES users(id) ON DELETE CASCADE,
  title    TEXT NOT NULL,
  body     TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- INNER JOIN (matching rows only)
SELECT u.name, p.title
FROM users u
JOIN posts p ON p.user_id = u.id;

-- LEFT JOIN (all users, even without posts)
SELECT u.name, COUNT(p.id) AS post_count
FROM users u
LEFT JOIN posts p ON p.user_id = u.id
GROUP BY u.id;

-- Many-to-many
CREATE TABLE post_tags (
  post_id INTEGER REFERENCES posts(id),
  tag_id  INTEGER REFERENCES tags(id),
  PRIMARY KEY (post_id, tag_id)
);

Indexes

-- B-tree (default, most common)
CREATE INDEX idx_users_email ON users(email);

-- Unique index
CREATE UNIQUE INDEX idx_users_uuid ON users(uuid);

-- Composite index
CREATE INDEX idx_posts_user_created ON posts(user_id, created_at DESC);

-- Partial index (only index active users)
CREATE INDEX idx_active_users ON users(email) WHERE role = 'admin';

-- GIN index for JSONB
CREATE INDEX idx_users_metadata ON users USING gin(metadata);

-- GIN index for arrays
CREATE INDEX idx_users_tags ON users USING gin(tags);

-- GIN for full-text search
CREATE INDEX idx_posts_search ON posts USING gin(to_tsvector('english', title || ' ' || body));

EXPLAIN ANALYZE your queries to verify indexes are being used. Don't over-index — each index slows writes.

Advanced SQL

-- CTE (Common Table Expression)
WITH active_users AS (
  SELECT * FROM users WHERE last_login > NOW() - INTERVAL '30 days'
)
SELECT role, COUNT(*) FROM active_users GROUP BY role;

-- Window functions
SELECT name, role, salary,
  RANK() OVER (PARTITION BY role ORDER BY salary DESC) AS rank,
  AVG(salary) OVER (PARTITION BY role) AS avg_salary
FROM employees;

-- Lateral join (correlated subquery as a join)
SELECT u.name, latest.title
FROM users u
CROSS JOIN LATERAL (
  SELECT title FROM posts WHERE user_id = u.id ORDER BY created_at DESC LIMIT 3
) latest;

-- GROUPING SETS
SELECT department, role, COUNT(*)
FROM employees
GROUP BY GROUPING SETS ((department), (role), (department, role), ());

JSONB

-- Store JSON data
UPDATE users SET metadata = '{"theme": "dark", "lang": "en", "prefs": {"notify": true}}';

-- Query JSONB
SELECT * FROM users WHERE metadata->>'theme' = 'dark';
SELECT * FROM users WHERE metadata @> '{"prefs": {"notify": true}}';

-- Extract values
SELECT metadata->'prefs'->>'notify' AS notify FROM users;

-- Update nested field
UPDATE users SET metadata = jsonb_set(metadata, '{prefs,notify}', 'false');

-- Array operations in JSONB
SELECT * FROM users WHERE metadata->'tags' ? 'premium';

-- Index for fast JSONB queries
CREATE INDEX idx_meta ON users USING gin(metadata);

When to use JSONB: Flexible schemas, user preferences, API responses, logs. Not for highly relational data.

Transactions & Concurrency

Transactions

BEGIN;
  UPDATE accounts SET balance = balance - 100
    WHERE id = 1;
  UPDATE accounts SET balance = balance + 100
    WHERE id = 2;
COMMIT;
-- Or ROLLBACK on error

-- Savepoints
BEGIN;
  INSERT INTO orders ...;
  SAVEPOINT sp1;
  INSERT INTO payments ...;  -- fails
  ROLLBACK TO sp1;
  INSERT INTO payments ...;  -- retry
COMMIT;

Isolation Levels

  • Read Committed — default. Sees committed data only.
  • Repeatable Read — snapshot at start. No phantom reads.
  • Serializable — strictest. As if transactions ran sequentially.
SET TRANSACTION ISOLATION LEVEL
  SERIALIZABLE;

Performance & EXPLAIN

-- Analyze query plan
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'alice@example.com';

-- Output:
-- Index Scan using idx_users_email on users (cost=0.29..8.30 rows=1)
--   Index Cond: (email = 'alice@example.com')
-- Planning Time: 0.1 ms
-- Execution Time: 0.05 ms

Seq Scan

Full table scan. Fine for small tables. Bad for large ones — add an index.

Index Scan

Uses an index to find rows. Fast. What you want for WHERE clauses.

VACUUM

Reclaims dead tuple space. Autovacuum runs automatically. Essential for MVCC health.

Connection Pooling

Use PgBouncer or Supavisor. PostgreSQL forks a process per connection — limit to 100-200.

Extensions

pgvector

Vector similarity search for AI/ML embeddings. Nearest neighbor search with HNSW or IVFFlat indexes.

PostGIS

Geospatial data types, functions, and indexes. Distance queries, geometry, geography, routing.

TimescaleDB

Time-series extension. Automatic partitioning by time, compression, continuous aggregates.

pg_cron

Run scheduled SQL jobs inside PostgreSQL. Cron syntax: VACUUM, aggregation, cleanup.

Marten

.NET document DB & event store built on PostgreSQL. JSONB-backed, ACID-compliant event sourcing and projections.

CREATE EXTENSION pgvector; — Extensions are one of PostgreSQL s greatest strengths.

TimescaleDB & Marten

TimescaleDB — Time-Series

  • PostgreSQL extension — standard SQL
  • Hypertables — auto-partition by time
  • Compression — 90%+ storage reduction
  • Continuous aggregates — materialized rollups
  • Use cases: IoT, metrics, logs, financial data
SELECT time_bucket('1 hour', ts) AS hour,
       avg(cpu), max(memory)
FROM metrics
WHERE ts > now() - interval '24 hours'
GROUP BY hour ORDER BY hour;

Marten — Event Store (.NET)

  • Document DB + Event Store on PostgreSQL
  • Event sourcing — append-only event streams
  • Projections — build read models from events
  • JSONB storage — ACID with NoSQL flexibility
  • No separate event store needed (Postgres is enough)
// Append events to a stream
session.Events.Append(orderId,
  new OrderPlaced(items),
  new PaymentReceived(amount));
await session.SaveChangesAsync();

// Build projection from events
var order = await session.Events
  .AggregateStreamAsync<Order>(orderId);

Backup & Replication

Backup

# Logical backup (SQL dump)
pg_dump -Fc mydb > backup.dump
pg_restore -d mydb backup.dump

# Physical backup (point-in-time)
pg_basebackup -D /backup -Fp -Xs -P

# Continuous archiving (WAL)
archive_mode = on
archive_command = 'cp %p /wal/%f'

Replication

  • Streaming replication — async/sync replicas
  • Logical replication — selective table replication
  • Read replicas — scale reads across replicas
  • Managed: AWS RDS, Supabase, Neon, Render

Best Practices

1. Always use TIMESTAMPTZ, never TIMESTAMP
2. Use UUID primary keys for distributed systems
3. EXPLAIN ANALYZE before optimizing
4. Use connection pooling (PgBouncer)
5. Index foreign keys and WHERE clause columns
6. Use JSONB for flexible data, not for everything
7. Monitor: pg_stat_statements, auto_explain
8. Automate backups and test restores
9. Use migrations (Prisma, Flyway, Alembic)
10. Keep transactions short to avoid lock contention
