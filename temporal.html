What is Temporal?

  • An open-source durable execution platform for running reliable distributed applications
  • Founded by the creators of Uber Cadence (2020), backed by Sequoia
  • Your code runs as if failures don't exist — automatic retries, state recovery
  • SDKs for Go, Java, TypeScript, Python, .NET
  • Used by Netflix, Stripe, Snap, Datadog and thousands more

Key Features

Durable Execution

Code state survives process crashes, server outages, and deployments

Automatic Retries

Configurable retry policies with exponential backoff for transient failures

Visibility

Full observability into every workflow execution, its state, and history

Scalability

Handles millions of concurrent workflows across distributed workers

Core Concept: Workflows

  • A Workflow is a durable function that orchestrates your business logic
  • Must be deterministic — same input always produces same sequence of commands
  • Can run for seconds to years — Temporal persists state automatically
  • Survives crashes: Temporal replays the workflow from event history on recovery
// TypeScript Workflow Example
export async function orderWorkflow(orderId: string): Promise<string> {
  await validateOrder(orderId);       // Activity
  await chargePayment(orderId);       // Activity
  await shipOrder(orderId);           // Activity
  return `Order ${orderId} completed`;
}

Core Concept: Activities

  • An Activity is a function that performs a single, well-defined action
  • Can be non-deterministic — API calls, DB writes, file I/O, etc.
  • Automatically retried on failure according to retry policies
  • Support heartbeats for long-running activities
// TypeScript Activity Example
export async function chargePayment(orderId: string): Promise<void> {
  const order = await db.getOrder(orderId);
  await paymentGateway.charge(order.amount, order.paymentMethod);
  await db.updateOrderStatus(orderId, 'paid');
}

Workers

  • Workers are processes you run that execute workflow and activity code
  • Poll the Temporal Server for tasks on specific task queues
  • Scale horizontally — add more workers for higher throughput
  • Stateless — all state is managed by the Temporal Server
// TypeScript Worker Setup
import { Worker } from '@temporalio/worker';
import * as activities from './activities';

const worker = await Worker.create({
  workflowsPath: require.resolve('./workflows'),
  activities,
  taskQueue: 'order-processing',
});

await worker.run();

Signals & Queries

Signals

Send data into a running workflow asynchronously. The workflow can react and change behavior.

// Define signal
const approvalSignal = defineSignal<
  [boolean]
>('approval');

// Handle in workflow
setHandler(approvalSignal,
  (approved) => {
    isApproved = approved;
  }
);

Queries

Read workflow state without affecting execution. Synchronous request/response.

// Define query
const statusQuery = defineQuery<
  string
>('status');

// Handle in workflow
setHandler(statusQuery,
  () => currentStatus
);

// Client queries
const status = await handle
  .query(statusQuery);

Retry Policies

Configure how activities are retried on failure:

const { chargePayment } = proxyActivities<typeof activities>({
  startToCloseTimeout: '30s',
  retry: {
    initialInterval: '1s',          // First retry after 1s
    backoffCoefficient: 2,           // Double the interval each retry
    maximumInterval: '60s',          // Cap at 60s between retries
    maximumAttempts: 5,              // Give up after 5 attempts
    nonRetryableErrorTypes: [        // Don't retry these
      'InvalidCardError',
      'InsufficientFundsError',
    ],
  },
});
  • Default policy retries indefinitely with exponential backoff
  • Use nonRetryableErrorTypes for business logic failures

Timers & Durable Sleep

Workflows can sleep for days, weeks, or months — durably:

export async function subscriptionWorkflow(userId: string) {
  while (true) {
    await chargeMonthlySub(userId);    // Activity

    // Sleep for 30 days - survives server restarts!
    await sleep('30 days');

    if (await isCancelled()) break;
  }
}
  • sleep() is persisted — not an in-memory timer
  • Workflow can sleep for years without consuming resources
  • Use condition() to wait for a signal with a timeout

Task Queues

  • Task Queues route tasks from the server to the right workers
  • Multiple workers can listen on the same queue for load balancing
  • Use different queues for different workloads or priorities
  • Enable task routing — send specific activities to specific workers

Error Handling

export async function orderWorkflow(orderId: string) {
  try {
    await chargePayment(orderId);
    await shipOrder(orderId);
  } catch (err) {
    // Compensate on failure (Saga pattern)
    await refundPayment(orderId);
    await cancelShipment(orderId);
    throw ApplicationFailure.nonRetryable(
      `Order ${orderId} failed: ${err.message}`
    );
  }
}
  • ApplicationFailure — use .retryable() or .nonRetryable()
  • Saga pattern — compensating actions to undo partial work
  • Cancellation — workflows can handle cancel requests gracefully

Starting & Interacting with Workflows

import { Client } from '@temporalio/client';

const client = new Client();

// Start a workflow
const handle = await client.workflow.start(orderWorkflow, {
  taskQueue: 'order-processing',
  workflowId: `order-${orderId}`,
  args: [orderId],
});

// Wait for result
const result = await handle.result();

// Send a signal
await handle.signal(approvalSignal, true);

// Query state
const status = await handle.query(statusQuery);

// Cancel
await handle.cancel();

Best Practices

1 Keep workflows deterministic — no random, time, or I/O
2 Put all side effects in activities, never in workflows
3 Use unique workflow IDs for idempotency (e.g., order ID)
4 Set appropriate timeouts on all activities
5 Use versioning (patched()) for workflow code changes
6 Keep event history small — use Continue-As-New for long loops
7 Use child workflows for complex sub-processes
8 Implement the Saga pattern for distributed transactions
