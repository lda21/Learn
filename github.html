Pull Requests (PRs)

  • Create a PR from a feature branch to propose changes
  • Review — teammates comment, approve, or request changes
  • CI checks run automatically via GitHub Actions
  • Merge strategies: merge commit, squash, or rebase
# PR workflow
git checkout -b feature/add-auth
# ... make changes ...
git add .
git commit -m "Add JWT authentication"
git push -u origin feature/add-auth

# Then open PR on GitHub or via CLI:
gh pr create --title "Add JWT auth"   --body "Implements token-based authentication"

# Review and merge
gh pr review 42 --approve
gh pr merge 42 --squash

Branch Protection Rules

  • Require PR reviews — no direct pushes to main
  • Require status checks — CI must pass before merge
  • Require signed commits — verify author identity
  • Restrict push access — only certain teams can merge
  • Require linear history — enforce squash or rebase merges

GitHub Actions: Basics

# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Run tests
        run: npm test

      - name: Run linter
        run: npm run lint

GitHub Actions: Advanced

# Deploy to production on tag push
name: Deploy

on:
  push:
    tags: ['v*']

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production      # Requires approval
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}

      - name: Deploy to ECS
        run: |
          aws ecs update-service             --cluster prod             --service api             --force-new-deployment

Issues & Projects

  • Issues — track bugs, features, tasks with labels, assignees, milestones
  • Projects (v2) — Kanban boards with custom fields, views, and automations
  • Labels — categorize: bug feature docs good first issue
  • Milestones — group issues into releases or sprints
  • Templates — standardize bug reports and feature requests

Code Review Best Practices

1 Keep PRs small — under 400 lines when possible
2 Write a clear PR description explaining the why
3 Use draft PRs for work in progress
4 Review within 24 hours to keep flow
5 Use CODEOWNERS for automatic reviewer assignment
6 Be constructive — suggest solutions, not just problems
7 Require CI passing before merge
8 Use squash merge for clean main branch history

GitHub Packages & Releases

  • GitHub Packages — host npm, Docker, Maven, NuGet packages alongside your code
  • Container Registry — ghcr.io/user/image:tag
  • Releases — versioned downloads with auto-generated changelogs
# Publish Docker image to GitHub Container Registry

# Create a release with auto-generated notes

Forking & Open Source

  • Fork — create your own copy of someone else's repo
  • Make changes in your fork, then open a PR to upstream
  • Stars — bookmark repos you like
  • Watch — get notifications for repo activity
# Fork workflow for open-source contributions
gh repo fork owner/repo --clone
cd repo
git checkout -b fix/typo-readme
# ... make changes ...
git push origin fix/typo-readme
gh pr create --repo owner/repo   --title "Fix typo in README"

# Keep fork in sync
git remote add upstream https://github.com/owner/repo.git
git fetch upstream
git merge upstream/main

GitHub Copilot

  • Copilot Chat — ask questions, explain code, fix bugs
