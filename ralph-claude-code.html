What Is Ralph?

Geoffrey Huntley coined the "Ralph" technique: run an AI code assistant in a loop until the project is done. Instead of one-shot prompts, you let the AI iterate continuously.

Ralph for Claude Code implements this with production-grade safety:

  • Exit Detection — dual-condition gate prevents premature termination
  • Rate Limiting — hourly API call caps and token budgets
  • Circuit Breakers — halts when no progress is being made
  • Session Management — preserves Claude context across iterations
  • Monitoring — live tmux dashboard for real-time visibility

Source: github.com/frankbria/ralph-claude-code

Why Autonomous Loops?

Manual AI Coding

  • One prompt, one response
  • Human must review and re-prompt
  • Context lost between sessions
  • Limited to what fits in one turn
  • Human bottleneck on iteration speed
  • Good for small, isolated tasks

Ralph Workflow

  • Continuous loop until done
  • Each iteration picks up where the last left off
  • Session continuity preserves context
  • Builds entire features autonomously
  • Runs overnight, unattended
  • Circuit breakers prevent waste

Result: Build features, fix bugs, and implement specs while you sleep.

How Ralph Runs: Process Architecture

Ralph is a bash script that orchestrates the Claude Code CLI in a while loop. No daemon, no server — just processes and files.

Terminal (your shell)
├── ralph_loop.sh              ← main bash script (parent process)
│   ├── claude -p "..."          ← child process, spawned each iteration
│   │                               blocks until Claude finishes
│   ├── response_analyzer.sh    ← parses Claude's stdout
│   ├── circuit_breaker.sh      ← reads/writes state file
│   └── (loops back to claude --continue)
│
└── ralph_monitor.sh           ← SEPARATE process (tmux pane)
    └── reads log files + state files on disk

Why tmux?

Ralph needs two panes: one running the loop, one showing the live dashboard. tmux provides split-terminal without a GUI. The monitor reads shared log files every few seconds.

Why jq?

Parses the RALPH_STATUS JSON block from Claude's output. Also reads/writes .ralph/state.json (circuit breaker state, session IDs, API counts). Falls back to grep/sed if output isn't valid JSON.

CLI Communication Flow

Each iteration follows this sequence — Ralph captures Claude's stdout and decides what to do next:

# Iteration 1 — new session
claude -p "$(cat .ralph/PROMPT.md)" --allowedTools "Bash,Read,Write,Edit"

# Iterations 2+ — continue same session (context preserved)
claude --continue -p "Continue working on fix_plan.md. Report RALPH_STATUS."
StepWhoWhat Happens
1RalphBuilds prompt string from PROMPT.md + fix_plan.md
2RalphSpawns claude --continue -p "$prompt" as child process
3ClaudeRuns 1-10 min: reads files, writes code, runs builds, appends RALPH_STATUS
4RalphClaude exits → stdout captured in bash variable
5Ralphresponse_analyzer.sh extracts STATUS, EXIT_SIGNAL, tasks, files (jq then grep fallback)
6Ralphcircuit_breaker.sh updates state — progress or no progress?
7RalphRate limiter checks API call count for this hour
8RalphDecision: loop again, wait (rate limit), or exit

The --continue flag is the key — it resumes the same Claude session so context (completed tasks, file changes) carries over between iterations.

RALPH_STATUS Protocol

Claude communicates progress back to Ralph via a structured status block appended to each response:

---RALPH_STATUS---
STATUS: IN_PROGRESS | COMPLETE | BLOCKED
TASKS_COMPLETED_THIS_LOOP: 3
FILES_MODIFIED: 5
EXIT_SIGNAL: true | false
---END_RALPH_STATUS---
  • STATUS — current state: working, done, or stuck
  • TASKS_COMPLETED_THIS_LOOP — how many checklist items finished this iteration
  • FILES_MODIFIED — number of files changed (used by circuit breaker)
  • EXIT_SIGNAL — explicit flag: should the loop stop?

response_analyzer.sh parses this block from Claude's output using pattern matching with fallback heuristics.

Exit Detection: Dual-Condition Gate

The most critical safety mechanism. Ralph exits ONLY when both conditions are met:

Condition 1: Heuristic

completion_indicators >= 2 — pattern-matches phrases like "all tasks complete", "nothing left to do", "fix_plan fully checked".

Condition 2: Explicit

EXIT_SIGNAL: true — Claude must explicitly set this flag in the RALPH_STATUS block.

Why Both?

  • Prevents premature exit when docs merely mention "done" or "complete"
  • Conflict resolution: STATUS: COMPLETE + EXIT_SIGNAL: false = keep going (more phases remain)
  • Only stops when Claude is genuinely finished with all work

Circuit Breaker Pattern

CLOSED
Normal Operation
→
HALF_OPEN
Monitoring
→
OPEN
Halted

Prevents infinite loops when Claude is stuck or making no progress.

Opens When

  • 3 loops with no file changes
  • 5 loops with the same errors repeating
  • Question loops (Claude asking for input) don't count as no-progress

Recovery

  • Cooldown timer (default 30 minutes)
  • Transitions to HALF_OPEN after cooldown
  • Auto-recovers to CLOSED when progress resumes
  • Manual reset available via CLI

Rate Limiting & Token Budgets

Hourly API Limits

Default: 100 calls/hour (configurable). Automatic reset on the hour. Prevents runaway costs.

Token Budget

Optional: track input_tokens + output_tokens per hour. Set TOKEN_BUDGET_PER_HOUR in .ralphrc.

5-Hour API Limit

Three-layer detection for Claude's rate limits. Auto-wait in unattended mode. Resumes when limit resets.

Cost Control

Start conservative (50/hour), increase as you gain confidence. Monitor via ralph --monitor.

Session Management

Ralph preserves Claude's conversation context across loop iterations, so Claude remembers what it was doing.

Session Continuity

Same session ID across iterations. Claude retains memory of completed tasks, file changes, and current goals.

Session Expiry

Default: 24 hours. After expiry, a new session starts fresh with the PROMPT.md context.

Reset Triggers

Circuit breaker opens, Ctrl+C interrupt, task completion, or expiration timer.

Transition Log

Last 50 session transitions logged. Useful for debugging why a session ended or restarted.

File Protection

Multi-layer strategy to prevent Claude from modifying Ralph's own configuration files:

Layer 1: ALLOWED_TOOLS

Restrictive tool permissions. No git clean, git rm, or other destructive operations on .ralph/ files.

Layer 2: PROMPT.md Warning

Explicit instruction: "DO NOT MODIFY files in .ralph/ directory". Claude respects this directive.

Layer 3: Integrity Check

Pre-loop validation confirms all required files exist and haven't been corrupted before each iteration.

Recovery

ralph-enable --force restores missing or corrupted files to their original state.

Error Handling & Recovery

Three-Layer Detection

1. Timeout guard (per-iteration limit). 2. JSON parsing of RALPH_STATUS. 3. Filtered text fallback for malformed output.

Productive vs Idle Timeouts

Checks git diff for actual file changes. Long-running but productive iterations are not killed prematurely.

Permission Denial

Detects when Claude is blocked by tool permissions. Guides user to update ALLOWED_TOOLS in .ralphrc.

Graceful Shutdown

Ctrl+C triggers SIGINT handler: saves session state, logs final status, and exits cleanly. No data loss.

# Error handling flow
1. Timeout guard (TIMEOUT_SECONDS exceeded?)
   └─ Check git diff: files changed? → productive timeout, extend
   └─ No changes? → idle timeout, count toward circuit breaker
2. Parse RALPH_STATUS JSON block
   └─ Parse failure? → fallback to text pattern matching
3. Evaluate circuit breaker state
   └─ OPEN? → cooldown and retry later

Developer Workflow: Exact Steps

You have a task (assignment, PRD, feature spec). Here's exactly what you do, command by command:

# ── STEP 1: Create your project folder ──────────────────────
mkdir battleship && cd battleship
git init

# ── STEP 2: Run the Ralph wizard ────────────────────────────
ralph-enable
#   → Detects: empty project (or scans for tech stack)
#   → Asks: where are your tasks? (manual / GitHub Issues / PRD file)
#   → Asks: tool permissions, rate limits, timeouts
#   → Creates: .ralph/ folder + .ralphrc config file
#   You now have this structure:
#     .ralphrc              ← config (edit rate limits, tools)
#     .ralph/
#       PROMPT.md           ← EMPTY — you must fill this in
#       fix_plan.md         ← EMPTY — you must fill this in
#       AGENT.md            ← pre-filled with detected build commands
#       specs/              ← empty folder for detailed specs
#       logs/               ← empty folder for execution logs

# ── STEP 3: Write your instructions (THIS IS YOUR JOB) ─────
#   Open .ralph/PROMPT.md in your editor and describe:
#   - What to build (the goal)
#   - Architecture decisions (tech stack, ports, structure)
#   - Constraints and rules
#   - "DO NOT MODIFY files in .ralph/ directory"

# ── STEP 4: Break work into tasks ──────────────────────────
#   Open .ralph/fix_plan.md and create a checklist:
#   - [ ] Task 1 (do this first — no dependencies)
#   - [ ] Task 2 (depends on task 1)
#   - [ ] ...
#   Order matters! Dependencies first. Claude works top to bottom.

# ── STEP 5: Add detailed specs (optional but recommended) ──
#   Create files in .ralph/specs/ for complex features:
#     .ralph/specs/api-contract.md
#     .ralph/specs/game-rules.md

# ── STEP 6: Review .ralphrc and AGENT.md ───────────────────
#   Edit .ralphrc: adjust MAX_LOOPS, RATE_LIMIT_PER_HOUR
#   Edit AGENT.md: add your build/test/lint commands

# ── STEP 7: Start Ralph ────────────────────────────────────
ralph                # basic — runs in foreground
ralph --monitor      # better — opens tmux with dashboard pane

# ── STEP 8: Go do something else ───────────────────────────
#   Ralph loops until fix_plan.md is fully checked off.
#   Come back, review git log, check the result.

Your Job vs Ralph's Job

You Write (Once)

  • PROMPT.md — the "what" and "why". Think of it as a brief to a contractor: what to build, what tech, what constraints.
  • fix_plan.md — ordered checklist. Each line = one unit of work Claude can complete in 1-3 iterations.
  • AGENT.md — how to verify. The build, test, and lint commands Claude runs to check itself.
  • specs/ — detailed requirements for anything non-obvious (API contracts, data models, business rules).
  • .ralphrc — safety knobs. Rate limits, max loops, timeouts.

Ralph Does (Automatically)

  • Reads your PROMPT.md and fix_plan.md
  • Picks the next unchecked - [ ] task
  • Claude writes code, creates files, runs commands
  • Claude runs your AGENT.md verify commands
  • Claude checks off - [x] the completed task
  • Reports RALPH_STATUS (progress, files changed)
  • Ralph checks circuit breaker and rate limits
  • Loops back — picks the next unchecked task
  • Exits when all tasks are [x] done

Key insight: The quality of your PROMPT.md and fix_plan.md directly determines the quality of Ralph's output. Garbage in, garbage out.

E2E Example: Battleship Game

Let's build a full-stack Battleship game from a home assignment spec using Ralph — showing every step from task to running app.

Backend

ASP.NET Core Web API — game state, ship placement, turn logic, hit/miss/sunk detection via REST endpoints.

MCP Server

C# MCP tool server over Streamable HTTP — exposes game actions as tools so an AI agent can play.

Frontend

React + TypeScript — two grids (your board + attack view), ship placement, fire shots. Player 1 controls only.

Infrastructure

Docker Compose — all services containerized. Single docker compose up --build brings everything up.

6×6 board, 3 ships (sizes 2, 3, 4), two players, first to sink all wins.

Step 1: Write PROMPT.md

The PROMPT.md tells Claude what to build and what rules to follow every iteration:

# Battleship Full-Stack Application

## Goal
Build a two-player Battleship game as a full-stack web app.

## Architecture
- **API**: ASP.NET Core Web API (C#) on port 5000
- **MCP**: C# MCP tool server (Streamable HTTP) on port 5001
- **UI**: React + TypeScript (Vite) on port 3000
- **Infra**: Docker Compose at project root

## Rules
- 6x6 board, 3 ships per player (sizes: 2, 3, 4)
- Players place ships, then alternate turns firing shots
- A player must NEVER see opponent ship positions
- Only expose hits/misses on the attack grid
- First to sink all 3 opponent ships wins

## Constraints
- No database needed (in-memory state is fine)
- No authentication required
- No tests required
- Player 2 can be idle (no UI controls for P2)

## DO NOT MODIFY files in .ralph/ directory

Best Practices

1. Start with ralph-enable wizard — don't manually create .ralph/
2. Write a clear PROMPT.md with specific, measurable goals
3. Break work into small tasks in fix_plan.md (one per line)
4. Set conservative rate limits initially (50/hour)
5. Use --monitor to watch progress in real-time
6. Review git diff after each loop for quality assurance
7. Set ALLOWED_TOOLS to minimum needed permissions
8. Use unattended mode for overnight runs
9. Keep specs/ detailed for complex features
10. Run ralph-enable --force if files get corrupted
