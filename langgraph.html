What Is LangGraph?

Overview

  • Framework by LangChain for building stateful, multi-step agent applications
  • Models agent workflows as directed graphs
  • Nodes = functions, Edges = transitions
  • Built-in state management, persistence, and streaming
  • First-class support for cycles (loops, retries, human-in-the-loop)

Key Differentiator

  • Unlike linear chains, graphs can branch, loop, and converge
  • State is explicit and typed — not hidden in prompts
  • Controllable: you decide routing logic, not the LLM
  • Production-ready: checkpointing, error recovery, observability
  • Python and JavaScript/TypeScript SDKs

Core Concepts

StateGraph

The main class. Define a graph with typed state, add nodes and edges, compile it into a runnable.

StateGraph(AgentState)

Nodes

Functions that receive state and return state updates. Each node performs one step of your workflow.

graph.add_node("research", research_fn)

Edges

Connections between nodes. Normal edges always transition; conditional edges choose the next node dynamically.

graph.add_edge("research", "analyze")

Conditional Routing

A function inspects state and returns the name of the next node. This is how supervisors route to specialists.

graph.add_conditional_edges(...)

Building Your First Graph

from langgraph.graph import StateGraph, START, END
from typing import TypedDict, Annotated
from operator import add

class AgentState(TypedDict):
    messages: Annotated[list, add]   # reducer appends new messages

def call_model(state: AgentState) -> dict:
    response = llm.invoke(state["messages"])
    return {"messages": [response]}

def call_tools(state: AgentState) -> dict:
    last_msg = state["messages"][-1]
    results = execute_tools(last_msg.tool_calls)
    return {"messages": results}

def should_continue(state: AgentState) -> str:
    if state["messages"][-1].tool_calls:
        return "tools"
    return END

# Build the graph
graph = StateGraph(AgentState)
graph.add_node("agent", call_model)
graph.add_node("tools", call_tools)
graph.add_edge(START, "agent")
graph.add_conditional_edges("agent", should_continue, ["tools", END])
graph.add_edge("tools", "agent")  # loop back after tool execution

app = graph.compile()
result = app.invoke({"messages": [("user", "What is AAPL price?")]})

State Management

TypedDict State

class TradingAgentState(TypedDict):
    messages: Annotated[list, add]
    current_agent: str
    research_output: str
    sentiment_score: float
    positions: list[dict]
    requires_approval: bool

State is a typed dictionary. Every node reads from and writes to this shared state.

Reducers

# Append reducer (most common)
messages: Annotated[list, add]
# Each node returns {"messages": [new_msg]}
# Reducer appends, never overwrites

# Overwrite (default for non-annotated)
current_agent: str
# Returns {"current_agent": "research"}
# Overwrites previous value

# Custom reducer
def merge_dicts(a, b):
    return {**a, **b}
metadata: Annotated[dict, merge_dicts]

Conditional Edges & Routing

def supervisor_router(state: TradingAgentState) -> str:
    """Supervisor decides which specialist agent to call next."""
    last_message = state["messages"][-1].content.lower()

    if "research" in last_message or "briefing" in last_message:
        return "research_agent"
    elif "sentiment" in last_message:
        return "sentiment_agent"
    elif "post-mortem" in last_message or "review" in last_message:
        return "postmortem_agent"
    else:
        return "research_agent"  # default

graph.add_node("supervisor", supervisor_node)
graph.add_node("research_agent", research_node)
graph.add_node("sentiment_agent", sentiment_node)
graph.add_node("postmortem_agent", postmortem_node)

graph.add_conditional_edges(
    "supervisor",
    supervisor_router,
    ["research_agent", "sentiment_agent", "postmortem_agent"]
)

The supervisor inspects state and routes to the correct specialist — the LLM does not decide the control flow.

Tool Integration

from langchain_core.tools import tool
from langchain_anthropic import ChatAnthropic

@tool
def get_positions() -> list[dict]:
    """Get current open trading positions from the MCP server."""
    return mcp_client.call("trading/positions")

@tool
def get_pnl(timeframe: str = "24h") -> dict:
    """Get profit and loss for the given timeframe."""
    return mcp_client.call("trading/pnl", {"timeframe": timeframe})

@tool
def get_market_data(symbol: str) -> dict:
    """Fetch market data from Alpha Vantage."""
    return mcp_client.call("alphavantage/quote", {"symbol": symbol})

# Bind tools to the LLM
llm = ChatAnthropic(model="claude-sonnet-4-20250514")
llm_with_tools = llm.bind_tools([get_positions, get_pnl, get_market_data])

# In a graph node, the LLM decides which tools to call
def research_node(state: AgentState) -> dict:
    response = llm_with_tools.invoke(state["messages"])
    return {"messages": [response]}

MCP: Model Context Protocol

What Is MCP?

  • Open protocol by Anthropic for connecting LLMs to external data & tools
  • Standardized interface — like USB-C for AI
  • Server exposes tools, resources, and prompts
  • Client (LLM app) discovers and calls them
  • Transport: stdio, SSE, or HTTP streaming

MCP in the Trading Agent

  • Internal MCP Server exposes trading data to LLM agents
  • Tools: positions, pnl, features
  • External MCP: Alpha Vantage for market data
  • External MCP: TradingView for charts and indicators
  • LLM agents call MCP tools via standard protocol

Human-in-the-Loop

Interrupt & Approve

# Compile with interrupt points
app = graph.compile(
    checkpointer=checkpointer,
    interrupt_before=["execute_trade"]
)

# Run until interrupt
result = app.invoke(
    {"messages": [("user", "Analyze BTC")]},
    config={"configurable": {"thread_id": "1"}}
)
# Graph pauses before execute_trade

# Human reviews, then resumes
app.invoke(None, config={"configurable":
    {"thread_id": "1"}})

Why It Matters

  • Safety gate before any consequential action
  • In the trading agent: LLM output is advisory — human always approves
  • interrupt_before pauses the graph at a named node
  • interrupt_after pauses after a node completes
  • State is persisted via checkpointer — resume anytime
  • Critical for financial, medical, and high-stakes applications

Streaming & Real-Time Output

# Stream events as the graph executes
async for event in app.astream_events(
    {"messages": [("user", "Daily market briefing")]},
    config={"configurable": {"thread_id": "1"}},
    version="v2"
):
    kind = event["event"]

    if kind == "on_chat_model_stream":
        # Token-by-token streaming from the LLM
        token = event["data"]["chunk"].content
        print(token, end="", flush=True)

    elif kind == "on_tool_start":
        # A tool is being called
        print(f"\n[Calling tool: {event['name']}]")

    elif kind == "on_chain_end":
        # A node finished execution
        print(f"\n[Node complete: {event['name']}]")

Three streaming modes: stream() for node outputs, astream_events() for fine-grained events, astream_log() for JSON patches.

Multi-Agent Patterns

Supervisor

One coordinator routes tasks to specialist agents. Used in the trading agent: supervisor dispatches to research, sentiment, post-mortem.

Hierarchical

Supervisors managing supervisors. Top-level routes to team leads, who route to specialists. For complex orgs.

Swarm

Agents hand off to each other based on context. No central coordinator. Each agent decides who goes next.

Debate / Critique

Multiple agents review the same input. One proposes, another critiques. Converge on higher-quality output.

Hallucination Control

The Problem

  • LLMs can confidently state false information
  • In trading: a hallucinated price or position is dangerous
  • LLMs may invent trades, misreport PnL, or fabricate market data
  • Advisory output must be verified before any human acts on it

Verification Strategy

  • Ground truth comparison: validate LLM output against the data store
  • Every claimed position checked against DB
  • Every PnL figure cross-referenced with actual records
  • Market data verified against MCP tool responses
  • Flag discrepancies before presenting to user
def verify_output(state: AgentState) -> dict:
    """Fact-check LLM claims against ground truth data."""
    llm_output = state["messages"][-1].content
    actual_positions = mcp_client.call("trading/positions")
    actual_pnl = mcp_client.call("trading/pnl")

    discrepancies = compare_claims(llm_output, actual_positions, actual_pnl)
    if discrepancies:
        return {"messages": [f"WARNING: {len(discrepancies)} claims unverified"],
                "verified": False}
    return {"verified": True}

LLM Safety in Production

Advisory Only

LLM outputs are never in the execution hot path. The trading engine makes all decisions. AI provides research and analysis for human review.

Rate Limiting

Cap API calls per minute/hour. Prevent runaway agents from burning budget. Use exponential backoff on 429 errors.

Token Budgets

Set max_tokens per request and per session. Track cumulative usage. Alert when approaching budget limits.

Cost Tracking

Log every API call with model, tokens, and cost. Dashboard for daily spend. Per-agent cost attribution.

Best Practices

1. Keep nodes small and focused — one responsibility per node
2. Use TypedDict state with explicit reducers, never untyped dicts
3. Route with code, not LLM — supervisor logic should be deterministic where possible
4. Always add a verification node after LLM output to catch hallucinations
5. Use checkpointers in production for crash recovery and audit trails
6. Set token budgets and rate limits per agent to control costs
7. Keep LLM outputs advisory — never let AI directly execute critical actions
8. Use MCP for clean tool interfaces — decouple data access from agent logic
9. Stream responses for better UX — users should see progress, not wait
10. Route models by task: Opus for deep analysis, Sonnet for fast routing
