Why Graph Databases?

The Problem with SQL Joins

  • Highly connected data = many JOINs
  • JOINs get exponentially slower with depth
  • "Friends of friends of friends" = 3 JOINs
  • Graph queries stay fast at any depth
  • Relationships are first-class citizens

Use Cases

  • Social Networks — friends, followers, feeds
  • Recommendations — "people also bought"
  • Fraud Detection — suspicious patterns
  • Knowledge Graphs — connected knowledge
  • Network/IT — dependencies, impact analysis
  • Identity & Access — permission graphs

Cypher: The Query Language

// CREATE nodes
CREATE (alice:Person {name: 'Alice', age: 30})
CREATE (bob:Person {name: 'Bob', age: 28})
CREATE (acme:Company {name: 'Acme Inc', industry: 'Tech'})

// CREATE relationships
CREATE (alice)-[:KNOWS {since: 2020}]->(bob)
CREATE (alice)-[:WORKS_AT {role: 'Engineer'}]->(acme)
CREATE (bob)-[:WORKS_AT {role: 'Designer'}]->(acme)

// Pattern: (node)-[relationship]->(node)
// () = node,  [] = relationship,  -> = direction

Cypher uses ASCII art patterns to match graph structures. Think of it as "drawing" your query.

Cypher Queries: MATCH

// Find a person
MATCH (p:Person {name: 'Alice'})
RETURN p

// Find Alice's friends
MATCH (alice:Person {name: 'Alice'})-[:KNOWS]->(friend)
RETURN friend.name, friend.age

// Friends of friends (2 hops)
MATCH (alice:Person {name: 'Alice'})-[:KNOWS*2]->(fof)
WHERE fof <> alice
RETURN DISTINCT fof.name

// Variable-length paths (1 to 5 hops)
MATCH path = (a:Person)-[:KNOWS*1..5]->(b:Person)
WHERE a.name = 'Alice' AND b.name = 'Dave'
RETURN path, length(path)

// Shortest path
MATCH path = shortestPath(
  (a:Person {name: 'Alice'})-[:KNOWS*]->(b:Person {name: 'Dave'})
)
RETURN path

Cypher: Filtering & Aggregation

// WHERE clause
MATCH (p:Person)-[:WORKS_AT]->(c:Company)
WHERE p.age > 25 AND c.industry = 'Tech'
RETURN p.name, c.name

// Aggregation
MATCH (c:Company)<-[:WORKS_AT]-(employee)
RETURN c.name, count(employee) AS headcount, avg(employee.age) AS avgAge
ORDER BY headcount DESC

// OPTIONAL MATCH (left outer join equivalent)
MATCH (p:Person)
OPTIONAL MATCH (p)-[:REVIEWED]->(product)
RETURN p.name, count(product) AS reviews

// WITH for pipeline (like subquery)
MATCH (p:Person)-[:KNOWS]->(friend)
WITH p, count(friend) AS friendCount
WHERE friendCount > 5
RETURN p.name, friendCount
ORDER BY friendCount DESC

Cypher: UPDATE & DELETE

Update

// Set properties
MATCH (p:Person {name: 'Alice'})
SET p.age = 31, p.city = 'Portland'

// Add a label
MATCH (p:Person {name: 'Alice'})
SET p:Employee

// Merge (create if not exists)
MERGE (p:Person {email: 'bob@ex.com'})
ON CREATE SET p.name = 'Bob',
              p.createdAt = datetime()
ON MATCH SET p.lastSeen = datetime()

Delete

// Delete a node (must remove
// relationships first)
MATCH (p:Person {name: 'Charlie'})
DETACH DELETE p
// DETACH removes all relationships

// Delete a relationship
MATCH (a)-[r:KNOWS]->(b)
WHERE a.name = 'Alice'
  AND b.name = 'Bob'
DELETE r

// Delete all (careful!)
MATCH (n)
DETACH DELETE n

Neo4j Architecture

Storage Engine

  • Native graph storage — not mapped onto tables
  • Index-free adjacency: each node directly references its neighbors
  • Traversals are O(1) per hop, regardless of total graph size
  • Separate stores for nodes, relationships, properties

Deployment

  • Neo4j Community — free, single instance
  • Neo4j Enterprise — clustering, causal consistency
  • Neo4j Aura — fully managed cloud (AWS, GCP)
  • APOC library — 450+ procedures & functions
  • GDS library — graph algorithms (PageRank, community detection)

Graph Algorithms

PageRank

Measures node importance by counting and weighting incoming connections. Used by Google for web ranking.

CALL gds.pageRank.stream('myGraph')
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).name, score
ORDER BY score DESC LIMIT 10

Community Detection

Find clusters of densely connected nodes. Louvain algorithm identifies natural groups.

CALL gds.louvain.stream('myGraph')
YIELD nodeId, communityId
RETURN communityId, count(*) AS size
ORDER BY size DESC

Shortest Path

Find the shortest route between two nodes. Dijkstra for weighted, BFS for unweighted.

Node Similarity

Find similar nodes based on shared neighbors. Jaccard or Cosine similarity. Powers recommendations.

Knowledge Graphs & AI

Knowledge Graphs

  • Structured representation of real-world entities
  • Google Knowledge Graph, Wikidata, DBpedia
  • Entity → Relationship → Entity triples
  • Ontologies define schemas (RDF/OWL)
  • Power search engines, chatbots, recommendations

Graphs + LLMs (RAG)

  • GraphRAG — retrieval augmented generation using graph context
  • Store embeddings on nodes for vector search
  • LLM generates Cypher from natural language
  • Combine vector similarity + graph traversal
  • Better context than flat document chunks

Graphs give LLMs structured context — relationships, hierarchies, and provenance that flat text lacks.

Gremlin Query Language

Apache TinkerPop's graph traversal language. Used by Neptune, Cosmos DB, JanusGraph.

// Add vertices and edges
g.addV('Person').property('name', 'Alice').property('age', 30)
g.addV('Person').property('name', 'Bob')
g.V().has('name', 'Alice').addE('KNOWS').to(g.V().has('name', 'Bob'))

// Traversal: find Alice's friends
g.V().has('name', 'Alice').out('KNOWS').values('name')

// Friends of friends
g.V().has('name', 'Alice').out('KNOWS').out('KNOWS')
  .dedup().values('name')

// Filter and aggregate
g.V().hasLabel('Person')
  .has('age', gt(25))
  .groupCount().by('city')

// Path queries
g.V().has('name', 'Alice')
  .repeat(out('KNOWS')).until(has('name', 'Dave'))
  .path().by('name')

Modeling Patterns

Intermediate Nodes

Convert complex relationships into nodes. A :Purchase node between :Person and :Product captures date, quantity, price.

Temporal Graphs

Add validFrom / validTo properties on relationships to model how connections change over time.

Hyperedges

When a relationship connects 3+ nodes, model it as an intermediate node connected to all participants.

Tree Structures

Org charts, file systems, categories. Use [:CHILD_OF] or [:REPORTS_TO] relationships. Traverse with * depth.

When NOT to Use a Graph DB

Use a Graph DB When

  • Relationships are as important as the data
  • You need deep traversals (3+ hops)
  • Schema is highly connected and evolving
  • Pattern matching and pathfinding are key
  • Real-time recommendations or fraud detection

Use Something Else When

  • Data is tabular with simple relationships → SQL
  • High-volume time series or logs → TimescaleDB
  • Simple key-value lookups → Redis
  • Document storage with flexible schema → MongoDB
  • Full-text search → Elasticsearch

Best Practices

1. Model as a whiteboard — nodes and arrows should map directly to your domain
2. Use specific relationship types: :PURCHASED not :RELATED_TO
3. Store properties on relationships, not just nodes
4. Create indexes on properties you filter by frequently
5. Limit variable-length paths with max depth: *1..5 not *
6. Use MERGE instead of CREATE to avoid duplicate nodes
7. Profile queries with EXPLAIN and PROFILE in Neo4j
8. Batch large imports with UNWIND or neo4j-admin import
9. Use the GDS library for graph algorithms, not custom Cypher
10. Start with Neo4j Aura free tier for learning and prototyping
