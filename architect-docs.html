Software Requirements (SRS)

Functional Requirements

  • FR-001: Users can register with email + password
  • FR-002: System sends verification email within 30s
  • FR-003: Users can search orders by date range
  • FR-004: Admin can export user data as CSV

Each FR maps to acceptance criteria and test cases.

Non-Functional Requirements

  • NFR-001: API response < 200ms (p95)
  • NFR-002: 99.9% uptime (8.7h downtime/year)
  • NFR-003: Support 10K concurrent users
  • NFR-004: Data encrypted at rest (AES-256)
  • NFR-005: GDPR compliant — right to deletion

NFRs drive architecture decisions.

Architecture Decision Records (ADR)

# ADR-003: Use PostgreSQL for User Data

## Status: Accepted (2024-03-15)

## Context
We need a database for user profiles, authentication, and order history.
Data is relational with complex queries. Team has SQL expertise.

## Decision
Use PostgreSQL 16 on AWS RDS with read replicas.

## Alternatives Considered
- MongoDB: Flexible schema but our data is relational. Joins are needed.
- DynamoDB: Good scale but complex access patterns don't fit key-value.

## Consequences
+ Strong ACID guarantees for financial data
+ Rich ecosystem (pgvector, PostGIS, TimescaleDB)
+ Team expertise reduces onboarding time
- Vertical scaling has limits (mitigated by read replicas)
- Schema migrations require careful planning

ADRs are immutable history — supersede, never edit.

Failure Handling & Resilience

Resilience Patterns

  • Circuit Breaker — stop calling failing services
  • Retry + Backoff — exponential with jitter
  • Bulkhead — isolate failures to one component
  • Timeout — deadline on every external call
  • Graceful Degradation — serve stale data, feature flags
  • Chaos Engineering — test failures proactively

Circuit Breaker States

✅ Closed
Normal — requests pass
↓
❌ Open
Failing — reject all
↓
⏳ Half-Open
Test — allow one request

Libraries: Polly (.NET), Resilience4j (Java), Hystrix

Consistency Models

CAP Theorem

Pick 2 of 3 during network partition:

  • CP — Consistency + Partition tolerance (PostgreSQL, MongoDB)
  • AP — Availability + Partition tolerance (Cassandra, DynamoDB)
PatternConsistency
ACID transactionsStrong
2PC (two-phase commit)Strong (slow)
Saga patternEventual
Event sourcingEventual
CQRSEventual (reads)

Decision Guide

  • 💰 Financial data?
    → Strong consistency (ACID, 2PC)
  • 📦 Order workflow?
    → Saga pattern (compensating txns)
  • 📊 Analytics / feeds?
    → Eventual consistency is fine
  • 📝 Audit trail?
    → Event sourcing (append-only)
  • 🔍 Heavy reads, few writes?
    → CQRS (separate read models)

Scaling Strategy

CDN — cache static content at edge
↓
Load Balancer — distribute traffic
↓
App Tier (N instances) — horizontal scale
↓

↓
DB: Primary + Read Replicas — scale reads

Vertical Scaling

Bigger machine. Simple but has limits. Good for databases initially.

Horizontal Scaling

More machines. Requires stateless apps. Use for app tier + workers.

API & Contract Documentation

What to Document

  • OpenAPI / Swagger — REST endpoints, schemas, auth
  • Proto files — gRPC service contracts
  • AsyncAPI — event-driven message schemas
  • GraphQL SDL — queries, mutations, subscriptions

Contracts are the source of truth between teams. Generate docs from code, not the other way around.

Contract-First Workflow

Define contract (proto / OpenAPI)
→
Generate client & server stubs
→
Implement & test against contract
→
Publish versioned docs (CI/CD)

Breaking changes require a major version bump and migration guide.

Best Practices

1 Document decisions not just outcomes — write ADRs
2 Diagrams at multiple zoom levels — C4 model
3 Security is architecture — not an afterthought
4 Define SLOs before building — they drive design
5 Plan for failure from day one — circuit breakers, retries
6 Keep docs close to code — docs-as-code in repo
7 Review architecture quarterly — fitness functions
8 Version your architecture docs with the codebase
