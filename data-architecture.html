What Is Data Architecture?

Definition

  • The design of data systems, storage, flow, and governance
  • How data is collected, transformed, stored, and consumed
  • Rules for data quality, lineage, and access control
  • Alignment between data infrastructure and business needs

Core Pillars

  • Storage — where data lives (OLTP, OLAP, object stores)
  • Flow — how data moves (batch, streaming, CDC)
  • Governance — who owns it, schema contracts, quality
  • Serving — how consumers access data (APIs, caches, views)

TimescaleDB Fundamentals

What Is It?

  • PostgreSQL extension for time-series data
  • Full SQL support — no new query language
  • Automatic partitioning via hypertables
  • 10-100x faster inserts vs vanilla PostgreSQL
  • Transparent — looks like a normal table

Hypertables & Chunks

-- Create a regular table
CREATE TABLE market_ticks (
  time        TIMESTAMPTZ NOT NULL,
  instrument  TEXT        NOT NULL,
  price       DOUBLE PRECISION,
  volume      DOUBLE PRECISION
);

-- Convert to hypertable (auto-partitions by time)
SELECT create_hypertable(
  'market_ticks', 'time',
  chunk_time_interval => INTERVAL '1 day'
);

Each chunk is an independent PostgreSQL table. Queries only scan relevant chunks via constraint exclusion.

Continuous Aggregates

Automatically roll up raw ticks into candles. TimescaleDB materializes results and refreshes incrementally — no cron jobs needed.

-- 5-minute OHLCV candles from raw ticks
CREATE MATERIALIZED VIEW candles_5m
WITH (timescaledb.continuous) AS
SELECT
  time_bucket('5 minutes', time) AS bucket,
  instrument,
  first(price, time)  AS open,
  max(price)           AS high,
  min(price)           AS low,
  last(price, time)    AS close,
  sum(volume)          AS volume
FROM market_ticks
GROUP BY bucket, instrument;

-- Auto-refresh: materialize new data every 5 minutes
SELECT add_continuous_aggregate_policy('candles_5m',
  start_offset  => INTERVAL '1 hour',
  end_offset    => INTERVAL '5 minutes',
  schedule_interval => INTERVAL '5 minutes'
);

Cascade: 1min ticks → 5m candles → 1h candles → 1d candles. Each layer is a continuous aggregate on the layer below.

Compression & Retention

Compression

-- Enable compression on hypertable
ALTER TABLE market_ticks
  SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'instrument',
    timescaledb.compress_orderby = 'time DESC'
  );

-- Auto-compress chunks older than 7 days
SELECT add_compression_policy(
  'market_ticks',
  compress_after => INTERVAL '7 days'
);

Typical compression: 90-95% size reduction. Compressed chunks are still queryable.

Retention

-- Drop raw ticks older than 2 years
SELECT add_retention_policy(
  'market_ticks',
  drop_after => INTERVAL '2 years'
);

-- Keep aggregates longer (10 years)
SELECT add_retention_policy(
  'candles_5m',
  drop_after => INTERVAL '10 years'
);

Lifecycle: Hot (uncompressed, <7d) → Warm (compressed, 7d-2yr) → Cold (S3 archive)

Kafka Streams for Feature Pipelines

Stateful stream processing to compute 500+ indicators per instrument in real time.

// Kafka Streams topology for trading features
StreamsBuilder builder = new StreamsBuilder();

KStream<String, MarketTick> ticks = builder.stream("market-ticks");

// Windowed aggregation: 5-minute VWAP
KTable<Windowed<String>, VwapAccumulator> vwap = ticks
    .groupByKey()
    .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5)))
    .aggregate(
        VwapAccumulator::new,
        (key, tick, acc) -> acc.add(tick.price(), tick.volume()),
        Materialized.with(Serdes.String(), vwapSerde)
    );

// Emit computed features to output topic
vwap.toStream()
    .map((windowed, acc) -> KeyValue.pair(
        windowed.key(),
        new Feature("vwap_5m", acc.vwap(), windowed.window().endTime())
    ))
    .to("computed-features");

Key design: Each instrument is a Kafka partition key, ensuring all ticks for one instrument go to the same Streams task — enabling stateful windowed aggregations without shuffles.

Schema Design with Protobuf

Canonical schemas define the contract between producers and consumers. Protobuf ensures backward compatibility and efficient serialization.

syntax = "proto3";
package trading.v1;

message MarketTick {
  string instrument    = 1;   // e.g. "BTC-USDT-PERP"
  int64  timestamp_ms  = 2;   // unix millis
  double bid_price     = 3;
  double ask_price     = 4;
  double bid_size      = 5;
  double ask_size      = 6;
  string exchange      = 7;
}

message FundingRate {
  string instrument    = 1;
  int64  timestamp_ms  = 2;
  double rate          = 3;   // e.g. 0.0001 = 0.01%
  int64  next_funding  = 4;   // next settlement time
}

message OrderEvent {
  string order_id      = 1;
  string instrument    = 2;
  enum Side { BUY = 0; SELL = 1; }
  Side   side          = 3;
  double price         = 4;
  double quantity      = 5;
  enum Status { NEW = 0; FILLED = 1; CANCELLED = 2; REJECTED = 3; }
  Status status        = 6;
  int64  timestamp_ms  = 7;
}

Rules: Never reuse field numbers. Add new fields with new numbers. Use Schema Registry for version management.

Data Quality & Validation

Crossed Quotes

Bid price > ask price. Indicates bad data from exchange or feed handler. Action: discard tick, alert on 3+ consecutive.

if tick.bid > tick.ask: reject()

Stale Quotes

No update for >30 seconds on an active instrument. May indicate disconnection. Action: mark stale, reduce position sizing.

if now() - last_tick > 30s: stale()

Flash Moves (>5σ)

Price moves more than 5 standard deviations from rolling mean. Likely a flash crash or fat-finger trade. Action: filter from features, flag for review.

if abs(z_score) > 5: quarantine()

Schema Validation

Every message validated against Protobuf schema at ingestion. Malformed messages routed to dead-letter topic for investigation.

kafka: dead-letter-topic

Batch vs Streaming Architecture

Lambda Architecture

  • Separate batch layer (complete, slow) and speed layer (fast, approximate)
  • Serving layer merges both views
  • Dual codepaths = dual maintenance burden
  • Good when batch accuracy is required alongside real-time

Kappa Architecture

  • Everything is a stream — single processing path
  • Reprocess by replaying Kafka from offset 0
  • Simpler: one codebase for batch and real-time
  • Trading agent uses Kappa — Kafka Streams handles both live and backfill
CriteriaLambdaKappa
ComplexityHigh (two systems)Low (one system)
ReprocessingBatch jobReplay stream
LatencyMinutes (batch) + seconds (speed)Seconds (stream only)
Best forML training + live scoringEvent-driven microservices

Change Data Capture (CDC)

Debezium + Outbox

  • Debezium captures row-level changes from PostgreSQL WAL
  • Publishes INSERT/UPDATE/DELETE events to Kafka topics
  • Outbox pattern: write events to an outbox table in the same transaction as your business data
  • Debezium reads the outbox table and publishes to Kafka
  • Guarantees atomicity: business state + event in one commit

Trading Agent Use Case

-- Outbox table in risk DB
CREATE TABLE outbox_events (
  id          UUID PRIMARY KEY,
  aggregate   TEXT NOT NULL,
  event_type  TEXT NOT NULL,
  payload     JSONB NOT NULL,
  created_at  TIMESTAMPTZ DEFAULT now()
);

-- In same transaction:
BEGIN;
  UPDATE positions
    SET size = size - 0.5
    WHERE instrument = 'BTC-USDT';
  INSERT INTO outbox_events VALUES (
    gen_random_uuid(),
    'position', 'REDUCED',
    '{"instrument":"BTC-USDT","delta":-0.5}'
  );
COMMIT;
-- Debezium picks up the outbox row

Consistency Models

Eventual Consistency

  • Acceptable for market data — ticks arrive out of order across exchanges
  • Feature vectors are "best effort latest" in Redis
  • Kafka guarantees ordering within a partition, not across
  • Continuous aggregates converge over the refresh window

Strong Consistency

  • Required for risk state — position sizes, balances, margin
  • PostgreSQL transactions with serializable isolation
  • Outbox pattern ensures event + state are atomic
  • Order fills must be idempotent (dedup by order_id)
Data TypeConsistencyRationale
Market ticksEventualApproximation OK; stale data detected by quality checks
Computed featuresEventualKafka Streams provides at-least-once; dedup downstream
Position stateStrongFinancial correctness — must never double-count fills
Risk limitsStrongSafety critical — incorrect limits can cause liquidation

Data Freshness Budgets

Every component has a maximum acceptable data age. Exceeding the budget triggers defensive actions.

ComponentMax AgeStaleness Action
Redis feature vectors120 secondsReduce position sizing by 50%
Market tick stream30 secondsMark instrument stale, halt new orders
Funding rate5 minutesUse last known rate, log warning
Risk state (positions)1 secondReject new trades until synced
Continuous aggregates10 minutesAlert on-call; backfill on recovery

Key principle: stale data is worse than no data. When in doubt, reduce exposure and alert.

Backup & Disaster Recovery

TimescaleDB / RDS

Multi-AZ deployment with automated failover. Daily snapshots with 35-day retention. Point-in-time recovery via WAL archiving.

RPO: ~5 min | RTO: ~5 min

Kafka

Replication factor 3 across availability zones. MirrorMaker 2 for cross-region replication. 7-day retention enables replay-based recovery.

RPO: 0 (sync repl) | RTO: ~2 min

Redis

Ephemeral by design — rebuilt from Kafka on restart. ElastiCache Multi-AZ for availability. No backup needed if reconstructible.

RPO: N/A | RTO: ~60s (warm-up)

S3 Cold Archive

Versioning enabled. Cross-region replication for critical audit data. Lifecycle rules: Standard → IA (30d) → Glacier (90d).

11 nines durability

Best Practices

1. Define canonical schemas (Protobuf/Avro) before writing any pipeline code
2. Partition Kafka by entity key, TimescaleDB by time — match access patterns
3. Use continuous aggregates instead of scheduled batch rollup jobs
4. Validate data quality at ingestion — never trust upstream sources
5. Set explicit freshness budgets and automate staleness responses
6. Use the outbox pattern for atomic state + event publishing
7. Compress aggressively — TimescaleDB achieves 90-95% reduction
8. Track data lineage from raw source to model input
9. Choose consistency model per data type: eventual for market data, strong for risk
10. Design for replay: Kappa architecture lets you reprocess by resetting Kafka offsets
