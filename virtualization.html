Virtualization & MicroVMs

From bare metal to microVMs — KVM, hypervisors, containers, and Firecracker

What is Virtualization?

The Concept

  • Creating virtual versions of physical hardware resources
  • Multiple isolated environments on a single machine
  • Each guest thinks it has dedicated hardware
  • Foundation of cloud computing — AWS, GCP, Azure all built on it
  • Enables efficient resource utilization, isolation, and portability

Why It Matters

  • Isolation — workloads can’t interfere with each other
  • Density — run hundreds of VMs on one server
  • Portability — move workloads between machines
  • Security — hardware-enforced boundaries
  • Snapshotting — save and restore entire machine states

Type 1 vs Type 2 Hypervisors

Type 1 — Bare Metal

  • Runs directly on hardware, no host OS
  • The hypervisor is the operating system
  • Better performance — no host OS overhead
  • Used in production data centers and cloud providers

Examples: VMware ESXi, Microsoft Hyper-V, Xen, KVM*

Type 2 — Hosted

  • Runs on top of a host OS as a regular application
  • Host OS manages hardware, hypervisor creates VMs above it
  • Easier setup but more overhead
  • Common for development and testing

Examples: VirtualBox, VMware Workstation, Parallels, QEMU (without KVM)

*KVM is unique — it turns the Linux kernel itself into a Type 1 hypervisor via a kernel module

KVM — Kernel-based Virtual Machine

Architecture

  • Kernel module (kvm.ko) that turns Linux into a hypervisor
  • Uses Intel VT-x or AMD-V hardware extensions
  • Each VM is a regular Linux process managed by the scheduler
  • Memory managed via EPT/NPT (Extended/Nested Page Tables)
  • In mainline Linux kernel since 2.6.20 (2007)

How It Works

# Check hardware support
grep -E "vmx|svm" /proc/cpuinfo

# Load KVM module
modprobe kvm_intel  # or kvm_amd

# KVM exposes /dev/kvm
ls -la /dev/kvm

# Each VM = a process with
# file descriptors for:
#   - VM control (ioctls)
#   - vCPUs
#   - Memory regions

VM Exits

  • Guest executes a privileged instruction (I/O, MSR access)
  • CPU traps into KVM kernel module
  • KVM handles the operation or delegates to userspace VMM
  • Fewer exits = better performance

Key Components

  • VMCS/VMCB — per-vCPU control structure storing guest state
  • EPT/NPT — two-level page tables for memory translation
  • IOMMU (VT-d) — direct device assignment to guests
  • virtio — para-virtualized device framework

VMM: The Userspace Component

What is a VMM?

  • Virtual Machine Monitor — the userspace process that manages a VM
  • KVM provides the hardware virtualization (kernel-side)
  • VMM provides the device model (userspace-side)
  • Emulates hardware devices the guest OS expects to see
  • Handles I/O, storage, networking for the guest

QEMU — The Swiss Army Knife

Overview

  • Full system emulator and VMM
  • Can emulate entire CPUs in software (x86, ARM, RISC-V, etc.)
  • With KVM: near native performance
  • Without KVM: full emulation (slow but cross-architecture)
  • Millions of lines of C code

Device Model

  • Emulates hundreds of devices: PCI, USB, GPU, sound, NVMe, etc.
  • Live migration — move running VMs between hosts
  • Snapshots — save/restore full VM state
  • BIOS/UEFI boot support
  • Powers libvirt, OpenStack, Proxmox
# Launch a VM with QEMU + KVM
qemu-system-x86_64 \
  -enable-kvm -m 4G -smp 4 \
  -drive file=disk.qcow2 \
  -net nic -net user

Firecracker — The MicroVM Pioneer

What Is It?

  • Open-source VMM built by AWS, written in Rust
  • Creates microVMs — lightweight VMs with minimal device model
  • Boots in <125ms with <5MB memory overhead
  • Uses KVM for hardware virtualization
  • Released in 2018, Apache 2.0 license

Design Philosophy

  • Minimal attack surface — ~50K lines of Rust (vs millions in QEMU)
  • No PCI bus — uses virtio over MMIO instead
  • No USB, GPU, sound — only what workloads need
  • No BIOS/UEFI — direct kernel boot only
  • No live migration — ephemeral by design
  • REST API over Unix socket for configuration

Containers vs Virtual Machines

Containers (Docker, OCI)

  • Share the host kernel — no separate OS
  • Isolation via namespaces (PID, network, mount) and cgroups (resource limits)
  • Start in milliseconds, minimal overhead
  • Portable via container images (OCI format)
  • Weaker isolation — kernel exploit = host compromise

Virtual Machines (KVM)

  • Own kernel — full OS per VM
  • Isolation via hardware virtualization (VT-x/AMD-V)
  • Boot in seconds, GBs of overhead
  • Portable via disk images (qcow2, raw)
  • Strong isolation — hardware-enforced boundaries

MicroVMs bridge the gap: container-like speed with VM-like isolation

Namespaces & Cgroups

Linux Namespaces

  • PID — isolated process ID trees
  • NET — separate network stacks, interfaces, IP addresses
  • MNT — independent filesystem mount points
  • UTS — separate hostname and domain
  • IPC — isolated inter-process communication
  • USER — separate user/group ID mappings
  • CGROUP — isolated cgroup hierarchies

Control Groups (cgroups)

  • CPU — limit CPU time and cores
  • Memory — set memory limits and OOM behavior
  • I/O — throttle disk read/write bandwidth
  • PIDs — limit number of processes
  • cgroups v2 unifies the hierarchy
# View cgroup limits
cat /sys/fs/cgroup/memory.max
cat /sys/fs/cgroup/cpu.max

virtio — Para-virtualized I/O

The Problem

  • Emulating real hardware (e.g., Intel e1000 NIC) is slow
  • Every I/O operation causes VM exits — traps into the hypervisor
  • Full emulation wastes CPU cycles on translating device protocols

The Solution: virtio

  • Guest knows it’s virtualized — uses optimized drivers
  • Shared memory virt-queues for efficient data transfer
  • Standardized by OASIS (virtio 1.0, 1.1, 1.2)

virtio Device Types

  • virtio-net — network interface
  • virtio-blk — block storage
  • virtio-scsi — SCSI storage
  • virtio-fs — shared filesystem (FUSE-based)
  • virtio-gpu — graphics adapter
  • virtio-vsock — host-guest socket communication

Transport Modes

  • PCI — used by QEMU (full PCI bus emulation)
  • MMIO — used by Firecracker (simpler, no PCI)
  • CCW — used on s390x (IBM mainframes)

Other MicroVM Projects

Kata Containers

  • Runs OCI containers inside lightweight VMs
  • Compatible with Docker and Kubernetes (CRI-O, containerd)
  • Supports multiple VMMs: QEMU, Firecracker, Cloud Hypervisor
  • CNCF project — bridges container UX with VM security
  • kata-runtime replaces runc as the OCI runtime

Cloud Hypervisor

  • Written in Rust, built on top of rust-vmm crates
  • Richer feature set than Firecracker: PCI, VFIO, hotplug
  • Supports UEFI boot and Windows guests
  • Backed by Intel, Microsoft, ARM, and others
  • Good middle ground between QEMU and Firecracker

rust-vmm — Building Blocks

What is rust-vmm?

  • A set of shared Rust crates for building VMMs
  • Used by Firecracker, Cloud Hypervisor, and CrosVM
  • Avoids duplicating VMM components across projects
  • Backed by AWS, Intel, Google, Red Hat, ARM

Key Crates

  • kvm-ioctls — safe Rust wrappers for KVM API
  • vm-memory — guest memory management
  • vm-virtio — virtio device framework
  • vm-superio — legacy serial/keyboard
  • linux-loader — kernel/initrd loading
  • event-manager — epoll-based event loop
  • seccompiler — seccomp BPF filter builder

Choosing the Right Approach

1 Containers — trusted workloads, maximum density, fast CI/CD, microservices
2 Full VMs (QEMU) — legacy OS, Windows guests, live migration, GPU passthrough
3 MicroVMs (Firecracker) — multi-tenant serverless, ephemeral sandboxes, untrusted code
4 Kata Containers — Kubernetes with VM isolation, compliance requirements
5 Cloud Hypervisor — cloud-native VMs with PCI/VFIO, richer than Firecracker
6 gVisor — user-space kernel for containers, intercepting syscalls without full VM
7 WASM (Wazero, Wasmtime) — language-level sandboxing, portable bytecode, edge computing
8 Confidential VMs (SEV, TDX) — encrypted memory, attestation, zero-trust cloud
