Why Redis?

Overview

  • In-memory data structure store
  • Sub-millisecond latency
  • Rich data types beyond key-value
  • Optional persistence (RDB + AOF)
  • Single-threaded event loop (no locks)

Use Cases

  • Caching — database query results, API responses
  • Sessions — user session storage
  • Rate Limiting — API throttling
  • Leaderboards — sorted sets
  • Queues — task queues, pub/sub
  • Real-time — chat, notifications

Data Types

# Strings (most basic)
SET user:1:name "Alice"
GET user:1:name            # "Alice"
SET counter 0
INCR counter               # 1
INCRBY counter 10          # 11
SET session:abc "data" EX 3600  # expires in 1 hour

# Hash (object / map)
HSET user:1 name "Alice" email "alice@ex.com" age 30
HGET user:1 name           # "Alice"
HGETALL user:1             # {name: "Alice", email: ..., age: 30}
HINCRBY user:1 age 1       # 31

# List (ordered, duplicates allowed)
LPUSH queue:tasks "task1" "task2"
RPOP queue:tasks           # "task1" (FIFO queue)
LRANGE queue:tasks 0 -1    # all items

# Set (unique, unordered)
SADD tags:post:1 "redis" "database" "cache"
SMEMBERS tags:post:1       # {"redis", "database", "cache"}
SINTER tags:post:1 tags:post:2  # intersection

Sorted Sets & Streams

Sorted Sets

# Leaderboard
ZADD leaderboard 100 "alice"
ZADD leaderboard 85 "bob"
ZADD leaderboard 92 "charlie"

# Top 3 (highest first)
ZREVRANGE leaderboard 0 2 WITHSCORES
# charlie:92, alice:100... wait
ZREVRANGEBYSCORE leaderboard +inf -inf LIMIT 0 3

# Rank
ZREVRANK leaderboard "alice"  # 0 (top)

# Increment score
ZINCRBY leaderboard 15 "bob"  # 100

Streams

# Append to stream
XADD events * user "alice" action "login"
XADD events * user "bob" action "purchase"

# Read latest
XRANGE events - + COUNT 10

# Consumer groups (like Kafka)
XGROUP CREATE events grp1 0
XREADGROUP GROUP grp1 consumer1 COUNT 5 BLOCK 2000 STREAMS events >

# Acknowledge processing
XACK events grp1 1234567890-0

Caching Patterns

Cache-Aside

App checks Redis first. Cache miss? Query DB, store in Redis. Most common pattern.

Write-Through

Write to Redis and DB simultaneously. Ensures cache is always fresh. Higher write latency.

Write-Behind

Write to Redis first, async flush to DB. Fastest writes. Risk of data loss on crash.

TTL Strategy

Set expiry on all cached data. Short TTL (5min) for volatile data. Long TTL (24h) for stable data.

# Cache-aside in pseudocode
result = redis.get("user:42")
if not result:
    result = db.query("SELECT * FROM users WHERE id = 42")
    redis.setex("user:42", 3600, serialize(result))  # cache 1 hour
return result

Pub/Sub & Rate Limiting

Pub/Sub

# Publisher
PUBLISH chat:room1 "Hello everyone!"

# Subscriber (blocking)
SUBSCRIBE chat:room1

# Pattern subscribe
PSUBSCRIBE chat:*

Fire-and-forget messaging. No persistence. Use Streams for durability.

# Allow 100 requests per minute per user

key = f"rate:{user_id}:{minute}"
current = redis.incr(key)
if current == 1:
    redis.expire(key, 60)
if current > 100:

Persistence

RDB Snapshots

  • Point-in-time snapshots to disk
  • Compact binary format
  • Great for backups
  • May lose data between snapshots
  • save 60 1000 — snapshot if 1000 keys changed in 60s

AOF (Append Only File)

  • Logs every write operation
  • More durable (fsync every second or always)
  • Larger files, slower recovery
  • appendonly yes
  • appendfsync everysec (recommended)

Production: Enable both RDB + AOF. RDB for fast restarts, AOF for minimal data loss.

Redis Cluster & Sentinel

Sentinel (HA)

  • Monitors master + replicas
  • Automatic failover if master dies
  • 3+ Sentinel processes for quorum
  • Clients connect via Sentinel for discovery
  • Good for single-shard high availability

Cluster (Sharding)

  • Data split across 16384 hash slots
  • Multiple masters, each owns a slot range
  • Each master has 1+ replicas
  • Automatic resharding and failover
  • Horizontal scaling for large datasets

Client Libraries

Node.js (ioredis)

import Redis from "ioredis";
const redis = new Redis("redis://localhost:6379");

await redis.set("key", "value", "EX", 3600);
const val = await redis.get("key");

// Pipeline (batch commands)
const pipe = redis.pipeline();
pipe.set("a", 1);
pipe.set("b", 2);
pipe.get("a");
const results = await pipe.exec();

Python (redis-py)

import redis

r = redis.Redis(host="localhost", port=6379, decode_responses=True)

r.set("key", "value", ex=3600)
val = r.get("key")

# Pipeline
with r.pipeline() as pipe:
    pipe.set("a", 1)
    pipe.set("b", 2)
    pipe.get("a")
    results = pipe.execute()

Best Practices

1. Use descriptive key names: entity:id:field
2. Always set TTL on cache keys to prevent memory bloat
3. Use pipelines for bulk operations (10-100x faster)
4. Avoid large keys (keep values under 1MB)
5. Use SCAN instead of KEYS in production
6. Monitor memory with INFO memory and maxmemory-policy
7. Use Sentinel or Cluster for production HA
8. Enable both RDB + AOF persistence
9. Use Lua scripts for atomic multi-step operations
10. Consider managed: AWS ElastiCache, Upstash, Redis Cloud
