What is Kafka?

Overview

  • Distributed event streaming platform
  • Publish-subscribe messaging at massive scale
  • Durable, fault-tolerant, high-throughput
  • Millions of messages per second
  • Created at LinkedIn, now Apache project

Use Cases

  • Event Streaming — real-time data pipelines
  • Log Aggregation — centralize app logs
  • Microservices — async communication
  • CDC — Change Data Capture from databases
  • Analytics — real-time dashboards

Broker

A Kafka server that stores data and serves clients. A cluster has 3+ brokers for fault tolerance.

Topic

A named category/feed of messages. Like a database table. Producers write to topics, consumers read from them.

Partition

A topic is split into partitions for parallelism. Each partition is an ordered, immutable log of records.

Offset

A unique sequential ID for each record within a partition. Consumers track their position by offset.

Topics & Partitions

# Create a topic with 6 partitions and replication factor 3
kafka-topics.sh --create \
  --bootstrap-server localhost:9092 \
  --topic orders \
  --partitions 6 \
  --replication-factor 3

# List topics
kafka-topics.sh --list --bootstrap-server localhost:9092

# Describe topic
kafka-topics.sh --describe --topic orders --bootstrap-server localhost:9092

# Topic: orders  Partitions: 6  Replication: 3
# Partition 0:  Leader: 1  Replicas: 1,2,3  ISR: 1,2,3
# Partition 1:  Leader: 2  Replicas: 2,3,1  ISR: 2,3,1

Key rule: Messages with the same key always go to the same partition (ordering guarantee per key).

Replication: Each partition has 1 leader + N-1 followers. ISR = in-sync replicas.

Producers

// Java producer example
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("acks", "all");  // strongest durability guarantee

Producer<String, String> producer = new KafkaProducer<>(props);

// Send message (async)
producer.send(new ProducerRecord<>("orders", orderId, orderJson),
  (metadata, exception) -> {
    if (exception != null) log.error("Send failed", exception);
    else log.info("Sent to partition {} offset {}", metadata.partition(), metadata.offset());
  });

// Key determines partition: hash(key) % numPartitions
// null key = round-robin across partitions

acks=0

Fire and forget. Fastest, may lose data.

acks=all

Wait for all ISR replicas. Slowest, no data loss.

Consumers & Consumer Groups

Consumer Basics

  • Subscribe to one or more topics
  • Poll for new records in a loop
  • Track position via offsets
  • Can seek to any offset (replay)
  • At-least-once delivery by default

Consumer Groups

  • Group of consumers sharing the work
  • Each partition assigned to ONE consumer in group
  • More consumers than partitions = idle consumers
  • Consumer dies? Partitions rebalance automatically
  • Different groups read independently

Scaling rule: Max parallelism = number of partitions. 6 partitions = max 6 consumers in a group.

Kafka Streams

A Java library for real-time stream processing on top of Kafka. No separate cluster needed.

StreamsBuilder builder = new StreamsBuilder();

// Read from topic, process, write to topic
KStream<String, String> orders = builder.stream("raw-orders");

orders
  .filter((key, value) -> value.contains("premium"))
  .mapValues(value -> enrichOrder(value))
  .to("processed-orders");

// Aggregation: count orders per customer
KTable<String, Long> orderCounts = orders
  .groupByKey()
  .count(Materialized.as("order-counts"));

// Windowed aggregation: orders per minute
orders
  .groupByKey()
  .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(1)))
  .count();

Schema Registry

Why Schemas?

  • Producers and consumers need to agree on data format
  • Schema Registry stores Avro, Protobuf, or JSON schemas
  • Compatibility checks prevent breaking changes
  • Schema evolution: add fields safely
  • Each schema gets a version and ID

Compatibility Modes

  • BACKWARD — new schema reads old data (default)
  • FORWARD — old schema reads new data
  • FULL — both directions compatible
  • NONE — no checks (dangerous)

Monitoring & Operations

Key Metrics

Consumer lag, under-replicated partitions, request latency, bytes in/out, active controller count.

Tools

Kafka UI, Confluent Control Center, Burrow (consumer lag), JMX metrics exported to Prometheus + Grafana.

Common Issues

Consumer lag growing, ISR shrinking, uneven partition distribution, rebalance storms.

Managed Options

Confluent Cloud, AWS MSK, Redpanda (Kafka-compatible, no JVM), Aiven, Upstash.

Delivery Semantics

Guarantees

  • At-most-once — may lose messages. Commit before processing.
  • At-least-once — may get duplicates. Process then commit (default).
  • Exactly-once — idempotent producer + transactions. enable.idempotence=true.

Exactly-Once Setup

// Producer config
enable.idempotence=true
transactional.id=my-tx-id
acks=all

// Transactional send
producer.initTransactions();
producer.beginTransaction();
producer.send(record);
producer.commitTransaction();

Best Practices

1. Use descriptive topic names: domain.entity.event
2. Choose partition count carefully (hard to reduce later)
3. Always use message keys for ordering guarantees
4. Set replication factor to 3, min.insync.replicas to 2
5. Use Schema Registry for all topics
6. Monitor consumer lag closely
7. Use Avro or Protobuf over JSON for efficiency
8. Enable compression (lz4 or zstd)
9. Design for idempotent consumers
10. Use KRaft mode (no ZooKeeper) for new clusters
