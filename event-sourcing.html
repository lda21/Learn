What Is Event Sourcing?

Traditional: Store Current State

  • Database holds the latest snapshot of each entity
  • UPDATE overwrites previous values
  • History is lost unless you build audit logging separately
  • Like a bank showing only your current balance

Event Sourcing: Store Every Change

  • Database holds an ordered sequence of events
  • Current state is derived by replaying events
  • Nothing is ever deleted or overwritten
  • Like a bank showing every transaction that led to your balance

Why Event Sourcing?

Complete Audit Trail

Every state change is recorded as an immutable event. Perfect for regulated industries like finance and trading.

Temporal Queries

Reconstruct state at any point in time. "What was the portfolio value at 2:37 PM on March 15th?"

Debugging & Replay

Reproduce any bug by replaying the exact sequence of events that caused it. No more "works on my machine."

Event-Driven Integration

Events are a natural integration point. Publish to Kafka, trigger workflows, build multiple read models from the same stream.

Trading systems are a perfect fit: every order, fill, and position change must be traceable.

Event Store Fundamentals

  • Append-only — events are never updated or deleted, only appended
  • Immutable — once written, an event is permanent. Corrections are new events (e.g., TradeCorrected)
  • Stream per aggregate — each entity (trade intent, position) has its own event stream
  • Global ordering — a sequence number or timestamp guarantees total order within a stream
  • Optimistic concurrency — expected version checked on write to prevent conflicts

Marten: Event Sourcing on PostgreSQL

Why Marten?

  • Event sourcing + document DB on PostgreSQL
  • No separate event store infrastructure needed
  • Uses PostgreSQL JSONB for event payloads
  • Built-in projections, snapshots, and multi-tenancy
  • Integrates natively with Wolverine for CQRS

Setup

// Program.cs
builder.Services.AddMarten(opts =>
{
    opts.Connection(connectionString);

    // Register event types
    opts.Events.AddEventType<IntentCreated>();
    opts.Events.AddEventType<RiskApproved>();
    opts.Events.AddEventType<Filled>();
    opts.Events.AddEventType<PositionOpened>();

    // Use string stream identity
    opts.Events.StreamIdentity =
        StreamIdentity.AsString;
});

Defining Events & Aggregates

// Domain Events (immutable records)
public record IntentCreated(
    string Symbol, decimal Quantity, string Signal, DateTimeOffset Timestamp);

public record RiskApproved(
    decimal MaxLoss, double RiskScore, DateTimeOffset Timestamp);

public record Filled(
    decimal Price, decimal Fee, string ExchangeOrderId, DateTimeOffset Timestamp);

public record Closed(string Reason, decimal Pnl, DateTimeOffset Timestamp);

// Aggregate - rebuilt from events via Apply methods
public class TradeIntent
{
    public string Id { get; set; }
    public string Symbol { get; private set; }
    public string Status { get; private set; } = "Pending";
    public decimal? FilledPrice { get; private set; }

    public void Apply(IntentCreated e)  { Symbol = e.Symbol; Status = "Created"; }
    public void Apply(RiskApproved e)   { Status = "Approved"; }
    public void Apply(Filled e)         { Status = "Filled"; FilledPrice = e.Price; }
    public void Apply(Closed e)         { Status = "Closed"; }
}

Projections

Transform events into optimized read models for dashboards and queries

Inline Projection

// Updated within same transaction
public class TradeSummaryProjection
    : SingleStreamProjection<TradeSummary>
{
    public void Apply(IntentCreated e,
        TradeSummary view)
    {
        view.Symbol = e.Symbol;
        view.Quantity = e.Quantity;
        view.Status = "Created";
    }

    public void Apply(Filled e,
        TradeSummary view)
    {
        view.FilledPrice = e.Price;
        view.Status = "Filled";
    }
}

Async Projection

// Processed in background daemon
// Eventual consistency, higher throughput
opts.Projections.Add<TradeSummaryProjection>(
    ProjectionLifecycle.Async);

// Analytics dashboard read model
public class DailyPnlProjection
    : MultiStreamProjection<DailyPnl, string>
{
    public DailyPnlProjection()
    {
        Identity<Closed>(e =>
            e.Timestamp.ToString("yyyy-MM-dd"));
    }

    public void Apply(Closed e,
        DailyPnl view)
    {
        view.TotalPnl += e.Pnl;
        view.TradeCount++;
    }
}

  • Commands — validate business rules, emit events, modify state
  • Queries — read from pre-built projections, no business logic
  • Write model and read model can use different schemas and databases
  • Natural pairing with event sourcing: events feed projections that serve queries

Why CQRS?

Optimize Independently

Write model is normalized for consistency. Read model is denormalized for fast queries. Each is tuned for its workload.

Scale Independently

Scale read replicas without affecting write throughput. Most systems are 90% reads — scale what matters.

Multiple Read Models

One event stream feeds many projections: analytics dashboard, search index, notification service, compliance report.

Simpler Code

Command handlers focus on business rules. Query handlers focus on data retrieval. No mixed concerns.

Trading Agent example: Commands validate risk & submit orders. Queries serve the analytics dashboard with pre-aggregated PnL data.

CQRS with Wolverine

Wolverine: message bus + command handler framework integrated with Marten

// Command
public record CreateTradeIntent(string Symbol, decimal Quantity, string Signal);

// Command Handler - Wolverine discovers by convention
public static class CreateTradeIntentHandler
{
    public static async Task<TradeIntentCreated> Handle(
        CreateTradeIntent command,
        IDocumentSession session)
    {
        var intentId = CombGuidIdGeneration.NewGuid();
        var @event = new IntentCreated(
            command.Symbol, command.Quantity, command.Signal, DateTimeOffset.UtcNow);

        // Append event to stream
        session.Events.StartStream<TradeIntent>(intentId, @event);
        await session.SaveChangesAsync();

        return new TradeIntentCreated(intentId);
    }
}

// API endpoint dispatches command via Wolverine
app.MapPost("/api/intents", (CreateTradeIntent cmd, IMessageBus bus)
    => bus.InvokeAsync<TradeIntentCreated>(cmd));

The Outbox Pattern

Preventing dual-write problems when publishing events to Kafka

The Dual-Write Problem

  • Save event to PostgreSQL
  • Publish to Kafka
  • What if step 2 fails? Data inconsistency
  • What if step 1 succeeds but app crashes before step 2?
  • You cannot have an atomic transaction across two systems

Outbox Solution

  • Save event + outbox message in same PostgreSQL transaction
  • Background process reads outbox and publishes to Kafka
  • Marten + Wolverine handle this automatically
  • Guaranteed at-least-once delivery
  • Consumers must be idempotent
// Marten + Wolverine outbox: zero configuration needed
builder.Services.AddMarten(opts => { /* ... */ })
    .IntegrateWithWolverine();  // Enables transactional outbox automatically

// Events saved to PostgreSQL + outbox in same transaction
// Wolverine daemon polls outbox and publishes to Kafka/RabbitMQ

Decision Replay

Reproduce any historical trade decision with the exact original context

  • Why? — debug why the AI agent opened a losing position at 3:42 AM
  • How? — replay the event stream up to that point to reconstruct exact-as-of state
  • Includes: market data snapshot, portfolio state, risk parameters, model signals
  • Deterministic: same events in, same state out, every time
// Replay aggregate state to a specific point in time
var intentAtDecision = await session.Events
    .AggregateStreamAsync<TradeIntent>(
        streamId,
        timestamp: new DateTimeOffset(2026, 3, 15, 3, 42, 0, TimeSpan.Zero));

// Fetch all events for forensic analysis
var events = await session.Events
    .FetchStreamAsync(streamId);

foreach (var e in events)
    Console.WriteLine($"[{e.Sequence}] {e.Timestamp} - {e.EventType.Name}");

Regulatory value: demonstrate to auditors exactly why each trade decision was made.

Snapshots & Performance

The Problem

  • Long-lived aggregates accumulate thousands of events
  • Replaying 10,000 events to load current state is slow
  • A position stream with frequent price modifications can grow quickly

Snapshots

  • Periodically save a snapshot of the aggregate state
  • Load snapshot + replay only events since snapshot
  • Marten handles this automatically with configuration
  • Snapshots are an optimization, not a source of truth
// Configure Marten to snapshot every 100 events
opts.Projections.Snapshot<TradeIntent>(SnapshotLifecycle.Inline, snapshotConfig =>
{
    snapshotConfig.CreateEvery = 100;  // Snapshot every 100 events
});

// Marten automatically:
// 1. Checks for latest snapshot
// 2. Replays only events AFTER snapshot
// 3. Returns fully hydrated aggregate
var intent = await session.Events.AggregateStreamAsync<TradeIntent>(streamId);

Schema Evolution

Events are forever — but your schema will change

  • Events are immutable — you cannot change events already stored
  • Upcasting — transform old event shapes into new ones on read
  • Versioning — IntentCreated_V1, IntentCreated_V2 with explicit version numbers
  • Backward compatibility — new fields should have defaults; never remove fields consumers depend on
// V1: Original event
public record IntentCreated_V1(string Symbol, decimal Quantity, string Signal);

// V2: Added confidence score
public record IntentCreated(
    string Symbol, decimal Quantity, string Signal, double Confidence);

// Upcast V1 -> V2 in Marten
opts.Events.Upcast<IntentCreated_V1, IntentCreated>(old =>
    new IntentCreated(old.Symbol, old.Quantity, old.Signal, Confidence: 0.5));

// Old events stored as V1 are transparently upcasted to V2 on read
// No data migration needed!

Event Sourcing + Kafka

Publishing domain events to Kafka topics for cross-service integration

// Wolverine Kafka transport configuration
builder.Services.AddWolverineWithKafka(opts =>
{
    opts.UseKafka("localhost:9092")
        .ConfigureConsumers(consumer =>
        {
            consumer.AutoOffsetReset = AutoOffsetReset.Earliest;
        });

    // Publish specific events to Kafka topics
    opts.PublishMessage<IntentCreated>()
        .ToKafkaTopic("trades.intent.created");

    opts.PublishMessage<Filled>()
        .ToKafkaTopic("trades.filled");

    opts.PublishMessage<PositionOpened>()
        .ToKafkaTopic("positions.opened");

    // Subscribe to external events
    opts.ListenToKafkaTopic("market.signals")
        .ProcessInline();
});

Key pattern: Marten is the event store (source of truth). Kafka is the event bus (distribution). The outbox guarantees delivery between them.

Best Practices

1. Make events immutable records — never modify stored events
2. Name events in past tense: IntentCreated, not CreateIntent
3. Keep events small — store facts, not derived data
4. Use the outbox pattern to prevent dual-write inconsistencies
5. Design consumers to be idempotent (at-least-once delivery)
6. Use snapshots for aggregates with more than 100-200 events
7. Version events from day one — schema evolution is inevitable
8. Separate read models per use case — don't share projections
9. Use async projections for analytics, inline for transactional reads
10. Test with event replay: assert state after applying a sequence of events
