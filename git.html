What is Git?

  • A distributed version control system created by Linus Torvalds (2005)
  • Tracks every change to every file with a complete history
  • Distributed — every clone is a full copy of the repository
  • The foundation of modern software development workflows
  • Used by virtually every developer and company worldwide

Essential Commands

# Setup
git init                        # Initialize new repo
git clone url                   # Clone existing repo

# Daily workflow
git status                      # See what changed
git add file.txt                # Stage specific file
git add .                       # Stage all changes
git commit -m "Fix login bug"   # Commit with message
git push                        # Push to remote
git pull                        # Fetch + merge from remote

# View history
git log --oneline               # Compact log
git log --graph --all           # Visual branch graph
git diff                        # Unstaged changes
git diff --staged               # Staged changes
git show abc123                 # Show a specific commit

Branching

# Create and switch to new branch

# or (newer syntax)

# List branches

# Switch branches
git checkout main
git switch main

# Delete branch

Merging

# Merge feature branch into main
git checkout main
git merge feature/auth

# Three merge strategies:

# 1. Fast-forward (linear history)
git merge --ff-only feature/auth

# 2. Merge commit (preserves branch history)
git merge --no-ff feature/auth

# 3. Squash merge (combine all commits into one)
git merge --squash feature/auth
git commit -m "Add authentication feature"
  • Fast-forward — clean linear history, no merge commit
  • Merge commit — preserves full branch context
  • Squash — clean history, one commit per feature

Rebasing

# Rebase feature branch onto latest main
git checkout feature/auth
git rebase main

# Interactive rebase: edit, squash, reorder commits
git rebase -i HEAD~3

# In the editor:
pick abc123 Add login form
squash def456 Fix typo in login
pick ghi789 Add logout button

# Rebase vs Merge:
# Rebase = linear history (rewrites commits)
# Merge  = preserves branch topology
  • Golden rule: never rebase commits that others have pulled
  • Use rebase for local cleanup before pushing
  • Use merge for integrating shared branches

Resolving Merge Conflicts

# When a conflict occurs:
<<<<<<< HEAD
const port = 3000;
=======
const port = 8080;
>>>>>>> feature/new-port

# Resolution steps:
# 1. Edit the file - keep what you want
const port = 8080;

# 2. Stage the resolved file
git add server.js

# 3. Continue the merge or rebase
git merge --continue
# or
git rebase --continue

# Abort if needed
git merge --abort
git rebase --abort

Stashing

# Save work-in-progress without committing
git stash                       # Stash changes
git stash push -m "WIP: auth"   # Stash with message

# View stashes
git stash list
# stash@{0}: On feature: WIP: auth
# stash@{1}: On main: quick experiment

# Restore stashed changes
git stash pop                   # Apply and remove from stash
git stash apply                 # Apply but keep in stash
git stash apply stash@{1}       # Apply specific stash

# Other operations
git stash drop stash@{0}        # Delete a stash
git stash clear                 # Delete all stashes
git stash show -p               # Show stash diff

Undoing Changes

# Unstage a file (keep changes in working dir)
git restore --staged file.txt
git reset HEAD file.txt         # Older syntax

# Discard changes in working directory
git restore file.txt
git checkout -- file.txt        # Older syntax

# Amend last commit (message or content)
git commit --amend -m "Better message"

# Revert a commit (creates new undo commit)
git revert abc123

# Reset to previous commit (DESTRUCTIVE)
git reset --soft HEAD~1     # Undo commit, keep staged
git reset --mixed HEAD~1    # Undo commit + staging
git reset --hard HEAD~1     # Undo everything (danger!)

# Recover from mistakes
git reflog                  # History of HEAD movements
git reset --hard HEAD@{2}   # Go back to reflog entry

Remote Operations

# Manage remotes
git remote -v                       # List remotes
git remote add origin url           # Add remote
git remote add upstream url         # Add upstream (forks)
git remote remove origin            # Remove remote

# Fetch and pull
git fetch origin                    # Download changes (no merge)
git pull origin main                # Fetch + merge
git pull --rebase origin main       # Fetch + rebase

# Push
git push origin main                # Push to remote
git push -u origin feature/auth     # Push + set upstream
git push --force-with-lease         # Safe force push

# Track remote branches
git checkout -b local origin/remote # Track remote branch
git branch -vv                      # Show tracking info

Git Log & History

# Useful log formats
git log --oneline --graph --all     # Visual graph
git log --since="2 weeks ago"       # Time-based
git log --author="John"             # By author
git log -- path/to/file             # File history
git log -p                          # Show diffs
git log -S "functionName"           # Search for string changes

# Blame - who changed each line
git blame file.txt
git blame -L 10,20 file.txt        # Lines 10-20 only

# Find when a bug was introduced
git bisect start
git bisect bad                      # Current commit is bad
git bisect good v1.0                # This tag was good
# Git binary searches through commits
git bisect reset                    # Done

Tags & Releases

# Lightweight tag
git tag v1.0.0

# Annotated tag (recommended)
git tag -a v1.0.0 -m "Release 1.0.0"

# Tag a specific commit
git tag -a v0.9.0 abc123 -m "Beta release"

# List and show tags
git tag                     # List all tags
git show v1.0.0             # Show tag details

# Push tags
git push origin v1.0.0      # Push specific tag
git push origin --tags       # Push all tags

# Delete tag
git tag -d v1.0.0                   # Local
git push origin --delete v1.0.0     # Remote

# Semantic versioning: MAJOR.MINOR.PATCH
# v1.0.0 -> v1.0.1 (patch) -> v1.1.0 (minor) -> v2.0.0 (major)

Git Flow — Full SDLC

The most structured branching model — maps directly to the software development lifecycle with versioned releases and hotfix support.

Long-Lived Branches

  • main — production code, every commit is a release (tagged v1.0, v2.0)
  • develop — integration branch for the next release, always ahead of main

Short-Lived Branches

  • feature/* — branch from develop, merge back via PR
  • release/* — branch from develop when ready, merge to main + develop
  • hotfix/* — branch from main, merge to main + develop

Release & Hotfix Lifecycle

Planned Release (v2.0)

  • 1. Features developed on feature/* branches
  • 2. PRs merged into develop
  • 3. Create release/2.0 from develop
  • 4. QA testing & bug fixes on release branch
  • 5. Merge to main → tag v2.0
  • 6. Merge back to develop
  • 7. Deploy v2.0 to production

Hotfix (v2.0.1)

  • 1. Critical bug found in production
  • 2. Create hotfix/2.0.1 from main
  • 3. Fix the bug, test locally
  • 4. Merge to main → tag v2.0.1
  • 5. Merge to develop (don't lose the fix)
  • 6. Deploy immediately to production

Semantic Versioning (SemVer)

MAJOR.MINOR.PATCH — e.g., v2.1.3

  • MAJOR (2.x.x) — breaking changes, new architecture
  • MINOR (x.1.x) — new features, backward-compatible
  • PATCH (x.x.3) — bug fixes, hotfixes

Best Practices

1 Write clear commit messages — explain why, not just what
2 Commit often, push regularly — small atomic commits
3 Never commit secrets — use .gitignore and .env
4 Use branches for every feature or fix
5 Pull before push to avoid conflicts
6 Use --force-with-lease instead of --force
7 Learn interactive rebase to clean up before merging
8 Use tags for releases and milestones

Commit Message Convention

# Conventional Commits format:
# type(scope): description

fix(api): handle null response from payment gateway
docs(readme): update installation instructions
refactor(db): extract query builder into separate module

feat(api)!: change response format for /users endpoint

Key Takeaways

  • Git is the universal version control system
  • Master the flow: add commit push pull
  • Branches are cheap — use them for everything
  • Know merge vs rebase and when to use each
  • Reflog is your safety net — almost nothing is truly lost
  • Write good commit messages — your future self will thank you
