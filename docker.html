What is Docker?

Containers vs VMs

  • VM — full OS, hypervisor, GBs, minutes to boot
  • Container — shared kernel, MBs, starts in seconds
  • Containers isolate processes, not entire OS
  • Uses Linux namespaces + cgroups

Images & Containers

Images

  • Read-only template with app + dependencies
  • Built from layers (each instruction = layer)
  • Stored in registries (Docker Hub, ECR)
  • Tagged: nginx:1.25-alpine
  • Layers are cached and shared

Containers

  • Running instance of an image
  • Thin writable layer on top
  • Isolated process with own filesystem
  • Ephemeral by default — data lost on removal
  • Lifecycle: create → start → stop → remove

Dockerfile Basics

# Start from a base image
FROM node:20-alpine

# Set working directory
WORKDIR /app

# Copy dependency files first (layer caching!)
COPY package*.json ./
RUN npm ci --production

# Copy application code
COPY . .

# Expose port (documentation only)
EXPOSE 3000

# Health check
HEALTHCHECK CMD curl -f http://localhost:3000/health || exit 1

# Default command
CMD ["node", "server.js"]
FROM Base image to build on

COPY Copy files into image
CMD Default run command (overridable)
ENTRYPOINT Fixed run command

Volumes & Storage

Volume Types

  • Named Volume — managed by Docker, best for persistence
  • Bind Mount — maps host path, good for dev
  • tmpfs — in-memory, ephemeral

When to Use

  • Databases: named volumes
  • Source code (dev): bind mounts
  • Secrets/temp: tmpfs
  • Never store data in container layer
# Named volume
docker volume create pgdata
docker run -d -v pgdata:/var/lib/postgresql/data postgres

# Bind mount (development)
docker run -d -v $(pwd)/src:/app/src -p 3000:3000 myapp:dev

# tmpfs (in-memory)
docker run -d --tmpfs /tmp:size=100m myapp

# List and inspect volumes
docker volume ls
docker volume inspect pgdata

Docker Compose

What is Compose?

  • Define multi-container apps in a single YAML file
  • One command to start everything: docker compose up
  • Manages services, networks, and volumes together
  • depends_on controls startup order
  • healthcheck ensures services are ready before dependents start
  • Environment variables for configuration
  • Named volumes persist data across restarts
  • Custom networks isolate service communication

docker-compose.yml

services:
  api:
    build: ./api           # build from Dockerfile
    ports: ["3000:3000"]   # host:container
    environment:
      - DATABASE_URL=postgres://db:5432/app
    depends_on:
      db:
        condition: service_healthy

  db:
    image: postgres:16-alpine
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      POSTGRES_DB: myapp
      POSTGRES_PASSWORD: secret
    healthcheck:
      test: ["CMD", "pg_isready"]
      interval: 5s

volumes:
  pgdata:        # named volume for persistence

Compose Commands & Features

Commands

docker compose up -d      # start all
docker compose down        # stop & remove
docker compose logs -f api # follow logs
docker compose ps          # list services
docker compose exec api sh # shell into service
docker compose build       # rebuild images
docker compose pull        # pull latest images

Advanced Features

  • profiles — optional services (debug, test)
  • healthcheck — readiness probes
  • depends_on + condition — startup order
  • env_file — load from .env file
  • deploy.replicas — scale services
  • extends — compose inheritance
# Scale a service
docker compose up -d --scale worker=3

# Override for development
docker compose -f docker-compose.yml -f docker-compose.dev.yml up

Multi-stage Builds

Use multiple FROM statements to create minimal production images.

# Stage 1: Build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: Production (only copies artifacts)
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
EXPOSE 3000
CMD ["node", "dist/server.js"]

Before Multi-stage

~800 MB

Build tools, source code, dev dependencies all included

After Multi-stage

~120 MB

Only production artifacts and runtime dependencies

Container Security

Run as Non-Root

Never run containers as root in production.

RUN addgroup -S app && adduser -S app -G app
USER app

Read-Only Filesystem

Prevent runtime file modifications.

docker run --read-only --tmpfs /tmp myapp

Image Scanning

Scan for CVEs before deploying.

docker scout cves myapp:latest

Tools: Trivy, Snyk, Scout, Grype

Secrets Management

Never put secrets in images or ENV.

  • Use Docker secrets or mounted files
  • Use Vault or AWS Secrets Manager

Resource Limits

# Memory limit (hard + soft)
docker run --memory=512m --memory-reservation=256m myapp

# CPU limit
docker run --cpus=1.5 myapp         # 1.5 CPU cores
docker run --cpu-shares=512 myapp   # relative weight

# Combined
docker run -d \
  --memory=1g \
  --cpus=2 \
  --pids-limit=100 \
  --name api myapp

What Happens at Limit?

  • Memory — OOM killer terminates container
  • CPU — throttled, not killed
  • PIDs — fork bomb protection

DNS Resolution

  • Containers on same network resolve by name
  • Docker embedded DNS: 127.0.0.11
  • Service discovery without external tools

Network Inspection

# List networks
docker network ls

# Inspect network
docker network inspect bridge

# Connect running container
docker network connect mynet api

# Disconnect
docker network disconnect mynet api

Tip: Always create custom bridge networks. The default bridge doesn't support DNS resolution between containers.

Docker in CI/CD

# .github/workflows/docker.yml
name: Build and Push
on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: docker/setup-buildx-action@v3

      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

Layer caching in CI dramatically speeds up builds — use cache-from / cache-to.

Best Practices

1. Use specific image tags, never latest in prod
2. Use multi-stage builds for smaller images
3. Run as non-root user
4. Use .dockerignore to exclude unnecessary files
5. Order Dockerfile for optimal layer caching
6. Use Alpine or distroless base images
7. Scan images for vulnerabilities
8. One process per container
9. Use health checks everywhere
10. Set resource limits in production
