Why This Role Exists

Everything else in the factory is scaffolding. The agents are where quality actually happens.

Excellent Agents → Ships

  • Prompt catches spec ambiguity before it becomes a bug
  • Red-team agent finds the escape before production does
  • Evidence bundle forces the handoff to be reviewable
  • Calibration closes the loop on every new failure class

Mediocre Agents → Doesn't

  • Orchestrator retries the same broken reasoning
  • Escapes slip through silently; no feedback loop
  • Reviewers become the bottleneck the factory was meant to remove
  • Every sprint re-learns yesterday's failures

Your leverage point: the five system prompts, their tools, and the loop that hardens them.

The Five Agents

📋 Product

Turns vague requests into testable specs. Owns acceptance criteria, edge cases, and the "done" definition.

🏗 Architect

Decomposes the spec into an implementation plan. Chooses patterns, interfaces, data contracts.

🧑‍💻 Developer

Writes and edits code against the plan. Runs tests locally, produces the diff and the evidence bundle.

🚨 Red-Team Gemini

Cross-model adversarial review. Hunts for spec gaps, security holes, and plausible-looking wrong code.

⚙ DevOps

Ships the diff. Owns CI, deploy, feature flags, rollback. The last line before production and the first line of telemetry back.

Deterministic Gates vs Probabilistic Agents

Two problem classes. Confusing them is the #1 way factories fail.

Deterministic Gate

  • Same input → same output, every time
  • Type checker, unit test, JSON schema, lint rule
  • Verdict is binary: pass / fail
  • Cheap to run, impossible to bypass accidentally
  • Where you encode non-negotiables

Probabilistic Agent

  • Same input → distribution of outputs
  • LLM call with prompt + tools
  • Verdict is a judgment, with confidence
  • Expensive, eventually wrong on some %
  • Where you encode judgment

Rule: if a check can be deterministic, it must be. Agents are for what gates can't express.

Claude Code's 27-event hook taxonomy (PreToolUse, PostToolUse, PreCompact, UserPromptSubmit, PermissionRequest, SubagentStop…) is the enforcement surface — an agent can't ignore a hook the way it can ignore a prompt.

Product Agent Deep Dive

Job

  • Turn a request into a testable spec
  • Enumerate acceptance criteria & edge cases
  • Surface ambiguity explicitly (don't resolve it)
  • Define "done" as a set of verifiable assertions

Evidence output

{
  "spec_version": "2.1",
  "title": "...",
  "acceptance": [
    {"id":"A1","given":"...","when":"...","then":"..."}
  ],
  "edge_cases": [...],
  "ambiguities": [
    {"id":"Q1","question":"...","blocks":["A3"]}
  ],

Failure mode it prevents: "works on my machine" — a spec the Developer and Red-Team can't both evaluate against.

Architect Agent Deep Dive

Job

  • Consume Product evidence, emit a plan
  • Choose interfaces, data contracts, seams
  • Flag risks & reversibility of each choice
  • Never write code

Signature moves

  • Tool config denies Edit / Write
  • Skill architect-docs loaded for ADR patterns
  • Outputs a strict JSON plan; schema gated in CI
  • Halts on unresolved ambiguity — upstream fix, not guess

Monday's work: revise the Architect prompt after last week's interface mismatches.

Developer Agent Deep Dive

Job

  • Execute the plan — the only agent with write tools
  • Run tests locally before handoff
  • Emit a diff + test results + coverage delta
  • Halt & escalate if plan is unbuildable

Evidence output

{
  "plan_id": "arch-2026-04-20-17",
  "diff": "...",
  "tests_added": 4,
  "tests_passing": 412,
  "coverage_delta": "+0.8%",
  "unresolved": [],
  "plan_deviations": [
    {"why": "interface X not feasible",
     "action": "escalated"}
  ]
}

Temptation to resist: letting the Developer also plan. Blurs responsibilities, wrecks eval signal.

Red-Team Agent (Gemini)

Why a different model

  • Claude wrote the code; Claude has correlated blind spots
  • Gemini's training distribution differs — catches failures Claude rationalizes
  • Model diversity is a cheap orthogonal signal
  • Adversarial stance baked into the prompt, not an afterthought

What it hunts

  • Spec gaps: "what about input X?"
  • Plausible-but-wrong code (looks fine, breaks subtly)
  • Security: injection, auth bypass, data leaks
  • Regression risk against the evidence bundle

Calibration tip: every finding it misses becomes a red-team eval case next sprint.

DevOps Agent Deep Dive

Job

  • Merge, deploy, monitor, roll back
  • Owns CI pipelines & feature-flag state
  • Closes the telemetry loop back to calibration
  • The last deterministic gate before production

Guardrails

  • Cannot deploy without green evidence bundle
  • Rollback is the default on anomaly, not an exception
  • Emits post-deploy observability snapshot as evidence
  • Feeds production escapes into the calibration queue

DevOps is where probabilistic work meets deterministic infrastructure. Keep the seam sharp.

What makes a bundle work

  • Schema-versioned — plan.v3.json, review.v2.json
  • Self-contained — next agent needs no hidden context
  • Machine-checkable — deterministic gates before the next agent runs
  • Auditable — every decision traces back to an input field

The community default is Markdown (spec-kit, OpenSpec, BMAD). Cheaper to author, but unvalidated. JSON schemas trade authoring cost for gate enforcement.

Prompts as Code

Treat them like code

  • In git. Reviewed via PR. Never edited in a playground
  • Semver: architect.v3.2.md
  • Changes require an eval run; CI blocks regressions
  • Diffs are readable; avoid monolithic rewrites

Red flag if you see

  • "Let me just tweak the prompt in prod"
  • Prompt lives in a Notion doc
  • No one knows what version is deployed
  • Changes shipped with zero eval evidence

Believing prompts don't need version control — immediate disqualifier for this role.

A/B Testing Prompt Changes

The question

Is this prompt change actually better, or did I overfit to the last failure?

  • Run old and new on identical eval set
  • Score with the same rubric, same model
  • Paired comparison: per-case win / loss / tie
  • Look at variance, not just mean

Signals to trust

  • Fewer regressions on the gold set
  • Lower variance across N reruns
  • Same or better on held-out cases
  • Cheaper at equal quality (tokens, latency)

"It felt better" is not a signal.

Eval Harnesses

Anatomy of a good harness

  • Gold set — N cases with expected outputs or rubric
  • Runner — deterministic; same seed, same tools, same env
  • Scorer — mix of exact match, schema check, LLM-judge for soft criteria
  • Report — per-case verdict, per-dimension scores, deltas vs baseline
  • Storage — every run archived; reproducible six months later
✓Every failure ticket becomes an eval case
✓Gold answers reviewed by a human, versioned
✓LLM-judge scorers are themselves eval'd
✓Cost & latency reported alongside quality

Regression Benchmarks in CI

Prompt PR workflow

  1. Engineer edits architect.v3.2.md
  2. CI runs full eval on gold set + recent escapes
  3. Report posted to PR: per-case diff vs baseline
  4. Reviewer inspects regressions before approving
  5. Merge blocks if any gold case regresses

What a good report looks like

architect.v3.1 -> v3.2
  gold set (240 cases):   +3 wins, 0 regressions
  recent escapes (18):    +5 fixed, 0 regressions
  latency p50:            -4% (faster)
  tokens/case p50:        +8%
  LLM-judge rubric:       +0.12 / 5.0
VERDICT: merge-eligible

Eval Harness — Code

// eval/run.ts  (Claude Agent SDK)
import Anthropic from "@anthropic-ai/sdk";
import { loadGoldSet, scoreCase, archive } from "./lib";

const client = new Anthropic();
const prompt = await readPrompt("architect.v3.2.md");
const cases  = await loadGoldSet("architect");

const results = await Promise.all(cases.map(async (c) => {
  const out = await client.messages.create({
    model: "claude-opus-4-7",
    system: prompt,
    messages: [{ role: "user", content: c.input }],
    max_tokens: 4096,
  });
  return { id: c.id, output: out, score: scoreCase(c, out) };
}));

await archive({ prompt_version: "v3.2", results });
const delta = diffAgainst("architect.v3.1", results);
if (delta.regressions.length) process.exit(1);   // CI gate

TS for the harness, Python for the scorers. Both versioned with the prompt.

Escape Classes

An escape = a failure that got past every gate and agent. Classify by severity & cause.

ClassMeaningResponse
E0Production incident — user-visible, data loss, or securityHalt factory, root-cause, new gate same day
E1Caught by customer / monitoring post-deployRollback, eval case, control by end of week
E2Caught by deterministic gate after all agents passedNew Red-Team eval case, prompt revision candidate
E3Caught by human reviewer when Red-Team said shipMost valuable — Red-Team prompt & skill update

E3 is the gold. Every E3 caught is a permanent upgrade to the Red-Team agent.

Case Study: A Wednesday E3

What happened

Human reviewer caught a SQL-injection-shaped pattern the Developer produced and Red-Team signed off on. The input was trusted "internal" data that had become externally reachable three months ago.

Why Red-Team missed it

  • Prompt assumed "internal" == trusted
  • No eval case for trust-boundary drift
  • Schema didn't flag the data source change

Closing the loop

  • Eval case added: "data previously internal, now external"
  • Red-Team prompt: explicit trust-boundary question
  • Architect evidence schema: data_source_provenance required
  • Gate: CI check for new external ingress on "internal" types

Next sprint: escape rate on this class drops to zero. That's the job.

Reviewer Checklist → Permanent Control

A reviewer catching the same thing twice means the checklist failed.

The promotion ladder

  1. Reviewer spots a new failure
  2. Add it to the reviewer checklist (humans)
  3. Add it to the Red-Team eval set (agent)
  4. Encode as a deterministic gate if possible
  5. Retire from the checklist once gated

Why it matters

  • Human reviewers are expensive, slow, and biased by fatigue
  • Their best work is finding novel classes, not policing known ones
  • Checklists become gates. Gates free reviewers for new escapes.
  • Friday's proposal: one checklist item ready to promote

Skill-Loading Logic

Bake in vs load on demand

  • System prompt — universal, stable, rarely changes
  • Skill — domain knowledge pack, loaded only when relevant
  • Loading irrelevant skills = context pollution, worse outputs
  • Skills are versioned separately; evals per skill

Routing heuristics

// Architect's skill router
if (spec.mentions("database schema"))
  load("data-contracts");
if (spec.touches(["auth","token","pii"]))
  load("security-architecture");
if (plan_complexity > threshold)
  load("adr-templates");

// Default: no skills — base prompt only

The router itself is an evaluable component. Wrong-skill-loaded is a measurable failure mode.

The SKILL.md format (anthropics/skills) encodes routing hints declaratively: name, description, when_to_use, model, effort, paths. Metadata loads; body loads on demand.

Multi-Agent Coordination

Single-agent chatbot

  • One prompt, one context, one conversation
  • Failure modes: hallucination, drift
  • Evals score final answers

Multi-agent factory

  • Five agents, five contexts, structured handoffs
  • Failure modes: inter-agent — spec drift, plan/code mismatch, review blind spots
  • Evals score each handoff and end-to-end
  • Debugging means tracing the bundle lineage

RAG-chatbot intuitions don't transfer. Coordination is a different problem class.

Anti-Patterns & Red Flags

✗"I use LangChain" — without specifics on what broke
✗Prompts edited in a UI playground, not git
✗No distinction between deterministic gates and probabilistic agents
✗Eval set = "vibes" instead of versioned cases
✗Treating all escapes the same — no severity taxonomy
✗RAG-chatbot experience presented as agent experience
✗Prompt changes shipped with zero eval evidence
✗Hackathon demo experience, not 3+ months of production

Each one predicts a specific class of failure in the factory.

Where This Fits in the OSS Landscape

The Claude Code community has converged on half the factory. The other half is whitespace.

Adopt — OSS has this solved

  • Skill format — anthropics/skills, SKILL.md, 1000+ community skills
  • Hooks — 27 lifecycle events; deterministic enforcement
  • MCP servers — mature tool integration standard
  • Ralph loops — canonical autonomous-loop pattern
  • Worktree-per-task — parallel agents without collision
  • Cross-model handoff — Claude ↔ Codex ↔ Gemini
  • Role-based multi-agent — MetaGPT, BMAD, superpowers, gstack

Build — OSS does not cover

  • Schema-validated evidence bundles — Markdown is the default; nobody gates on structure
  • Eval harnesses with CI regression gates — literally absent in this ecosystem
  • A/B prompt comparison with paired win/loss stats — zero coverage
  • Escape-class taxonomy (E0–E3) — no one has named it
  • Calibration loop — "document learnings" is the ceiling; no automation
  • Red-Team as a seat — cross-model review exists; permanent adversary role does not
  • Checklist → gate promotion ladder — checklists yes, discipline no

The moat is on the right. Everyone ships agents. Almost no one measures them.
