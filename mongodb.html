MongoDB

The document database for modern applications

NoSQLDocumentFlexible SchemaScalable

Why MongoDB?

Overview

  • Document-oriented NoSQL database
  • Stores data as JSON-like BSON documents
  • Flexible schema — no migrations needed
  • Horizontal scaling with sharding
  • Rich query language & aggregation pipeline

When to Use

  • Evolving schemas — rapid iteration
  • Nested data — embed related objects
  • High write volume — logs, IoT, events
  • Horizontal scale — shard across nodes
  • Geo queries — location-based apps

Documents & Collections

// A document (like a JSON object with types)
{
  _id: ObjectId("507f1f77bcf86cd799439011"),
  name: "Alice Johnson",
  email: "alice@example.com",
  age: 28,
  address: {                    // embedded document
    street: "123 Main St",
    city: "Portland",
    state: "OR"
  },
  tags: ["developer", "nodejs"], // array
  createdAt: ISODate("2024-01-15T08:30:00Z")
}

// Database > Collections > Documents
// (like)  Schema > Tables    > Rows
// But documents in a collection can have different fields!

Key: _id is auto-generated (ObjectId) if not provided. Every document must have one.

CRUD Operations

// INSERT
db.users.insertOne({ name: "Alice", age: 28, city: "Portland" })
db.users.insertMany([
  { name: "Bob", age: 32 },
  { name: "Charlie", age: 25 }
])

// FIND (Read)
db.users.find({ age: { $gte: 25 } })           // age >= 25
db.users.findOne({ email: "alice@example.com" })
db.users.find({ city: "Portland" }).sort({ name: 1 }).limit(10)

// UPDATE
db.users.updateOne(
  { name: "Alice" },                             // filter
  { $set: { age: 29 }, $push: { tags: "senior" } } // update
)
db.users.updateMany({ age: { $lt: 18 } }, { $set: { minor: true } })

// DELETE
db.users.deleteOne({ name: "Charlie" })
db.users.deleteMany({ lastLogin: { $lt: ISODate("2023-01-01") } })

Query Operators

Comparison

$eq   — equal
$ne   — not equal
$gt   — greater than
$gte  — greater or equal
$lt   — less than
$lte  — less or equal
$in   — matches any in array
$nin  — matches none in array

// Example
db.products.find({
  price: { $gte: 10, $lte: 100 },
  category: { $in: ["books", "electronics"] }
})

Logical & Element

// Logical
$and, $or, $not, $nor

db.users.find({
  $or: [
    { age: { $lt: 18 } },
    { status: "inactive" }
  ]
})

// Element
$exists — field exists
$type   — field is a BSON type

db.users.find({
  phone: { $exists: true },
  age: { $type: "number" }
})

Indexing

// Single field index
db.users.createIndex({ email: 1 })          // ascending

// Compound index
db.orders.createIndex({ userId: 1, createdAt: -1 })

// Unique index
db.users.createIndex({ email: 1 }, { unique: true })

// Text index (full-text search)
db.articles.createIndex({ title: "text", body: "text" })
db.articles.find({ $text: { $search: "mongodb tutorial" } })

// Geospatial index (2dsphere)
db.places.createIndex({ location: "2dsphere" })
db.places.find({
  location: { $near: { $geometry: { type: "Point", coordinates: [-73.97, 40.77] }, $maxDistance: 5000 } }
})

// TTL index (auto-delete after time)
db.sessions.createIndex({ createdAt: 1 }, { expireAfterSeconds: 3600 })

Tip: Use explain() to verify queries use indexes: db.users.find({email:"a@b.com"}).explain("executionStats")

Aggregation Pipeline

db.orders.aggregate([
  // Stage 1: Filter
  { $match: { status: "completed", createdAt: { $gte: ISODate("2024-01-01") } } },

  // Stage 2: Group & accumulate
  { $group: {
      _id: "$customerId",
      totalSpent: { $sum: "$amount" },
      orderCount: { $sum: 1 },
      avgOrder: { $avg: "$amount" }
  }},

  // Stage 3: Filter groups
  { $match: { totalSpent: { $gte: 1000 } } },

  // Stage 4: Sort
  { $sort: { totalSpent: -1 } },

  // Stage 5: Reshape output
  { $project: {
      customerId: "$_id",
      totalSpent: 1,
      orderCount: 1,
      avgOrder: { $round: ["$avgOrder", 2] }
  }},

  // Stage 6: Limit
  { $limit: 10 }
])

Aggregation Stages

$lookup (Join)

Left outer join with another collection. Like SQL JOIN.

{ $lookup: {
    from: "products",
    localField: "productId",
    foreignField: "_id",
    as: "product"
}}

$unwind

Deconstruct an array field into one document per element.

{ $unwind: "$items" }
// {items: [a,b]} becomes
// {items: a}, {items: b}

$facet

Run multiple pipelines in parallel on the same input.

{ $facet: {
    totalCount: [{ $count: "n" }],
    topResults: [{ $limit: 10 }]
}}

$addFields / $set

Add or overwrite fields in documents.

{ $addFields: {
    fullName: {
      $concat: ["$first", " ", "$last"]
    }
}}

Schema Design Patterns

Embed (Denormalize)

  • Store related data in same document
  • Best for 1:1 and 1:few relationships
  • Single read = all data needed
  • Max document size: 16 MB
// Blog post with comments embedded
{
  title: "MongoDB Guide",
  comments: [
    { user: "bob", text: "Great!" },
    { user: "eve", text: "Thanks!" }
  ]
}

Reference (Normalize)

  • Store references (ObjectIds) to other documents
  • Best for 1:many and many:many
  • Avoids duplication, keeps docs small
  • Requires $lookup for joins
// Order references user
{
  _id: ObjectId("..."),
  userId: ObjectId("user123"),
  items: [ObjectId("prod1"), ObjectId("prod2")]
}
// Use $lookup to join

Transactions

// Multi-document ACID transactions (since MongoDB 4.0)
const session = client.startSession();

try {
  session.startTransaction();

  await db.collection('accounts').updateOne(
    { _id: fromAccount },
    { $inc: { balance: -amount } },
    { session }
  );

  await db.collection('accounts').updateOne(
    { _id: toAccount },
    { $inc: { balance: amount } },
    { session }
  );

  await db.collection('transfers').insertOne(
    { from: fromAccount, to: toAccount, amount, date: new Date() },
    { session }
  );

  await session.commitTransaction();
} catch (error) {
  await session.abortTransaction();
  throw error;
} finally {
  session.endSession();
}

Note: Single-document operations are always atomic. Use transactions only for multi-document consistency.

Mongoose (Node.js ODM)

import mongoose from 'mongoose';

// Define schema with validation
const userSchema = new mongoose.Schema({
  name:  { type: String, required: true, trim: true },
  email: { type: String, required: true, unique: true, lowercase: true },
  age:   { type: Number, min: 0, max: 150 },
  role:  { type: String, enum: ['user', 'admin'], default: 'user' },
  posts: [{ type: mongoose.Schema.Types.ObjectId, ref: 'Post' }],
}, { timestamps: true });  // adds createdAt, updatedAt

// Add methods
userSchema.methods.isAdmin = function() { return this.role === 'admin'; };

// Create model
const User = mongoose.model('User', userSchema);

// Use it
const user = await User.create({ name: 'Alice', email: 'alice@ex.com' });
const admins = await User.find({ role: 'admin' }).sort({ name: 1 });

Performance & Monitoring

explain()

Analyze query performance. Check IXSCAN vs COLLSCAN, execution time, and docs examined.

db.users.find({email:"a@b.com"})
  .explain("executionStats")

Profiler

Log slow queries. Level 0 (off), 1 (slow > threshold), 2 (all).

db.setProfilingLevel(1, { slowms: 100 })
db.system.profile.find().sort({ts:-1})

Key Metrics

  • Op counters (queries, inserts, updates)
  • Connections (current, available)
  • Cache hit ratio (WiredTiger)
  • Replication lag

Best Practices

  • Index all query patterns
  • Avoid $where and regex on unindexed fields
  • Use projections to return only needed fields
  • Monitor with Atlas or Prometheus Exporter

Rule of thumb: Use MongoDB for flexible, nested data. Use SQL for strict relational data with complex joins.

Best Practices

1. Design schema around your query patterns, not entity relationships
2. Embed data you always read together; reference data accessed independently
3. Create indexes for every query pattern — check with explain()
4. Keep documents under 16 MB (use GridFS for large files)
5. Use the aggregation pipeline over client-side data processing
6. Choose shard keys with high cardinality and even distribution
7. Use schema validation rules for data integrity
8. Enable authentication and TLS in production — never expose MongoDB to the internet
9. Use Change Streams instead of polling for real-time features
10. Deploy replica sets (minimum 3) for high availability
