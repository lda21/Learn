Feature Store

Bridging ML training and production serving with Feast

MLFeastFeature EngineeringMLOps

The Problem

Without a Feature Store

  • Training features computed in notebooks
  • Serving features rewritten in production code
  • Training/serving skew — different logic, different results
  • Duplicated feature pipelines across teams
  • No discoverability — "does this feature exist?"

With a Feature Store

  • Single source of truth for feature definitions
  • Same features for training and serving
  • Consistent transformations everywhere
  • Feature catalog — search, reuse, share
  • Point-in-time correct training data

What Is a Feature Store?

A feature store is a centralized platform for defining, storing, and serving ML features consistently across training and inference.

Offline Store

Historical feature values for training. Supports point-in-time joins. Backed by data warehouses (BigQuery, Snowflake, Redshift, files).

Online Store

Latest feature values for real-time serving. Low-latency lookups (< 10ms). Backed by Redis, DynamoDB, or SQLite.

Feature Registry

Catalog of all feature definitions, metadata, ownership, and lineage. The "schema" of your ML data.

Feature Serving

APIs and SDKs for fetching features at training time (batch) and inference time (real-time).

Training/Serving Skew

The #1 reason ML models degrade in production.

CauseWhat Goes WrongFeature Store Fix
Different code pathsPython notebook vs. Java service compute features differentlySingle feature definition used everywhere
Temporal leakageTraining uses future data not available at inference timePoint-in-time joins prevent leakage
Stale featuresOnline features lag behind realityMaterialization pipelines keep online store fresh
Missing valuesTraining fills nulls one way, serving anotherConsistent default values in feature definitions

Feast — Open-Source Feature Store

What Is Feast?

  • Open-source feature store (Linux Foundation / LF AI)
  • Python-first, declarative feature definitions
  • Supports batch, streaming, and on-demand features
  • Pluggable offline/online stores
  • Built-in feature server (REST/gRPC)
  • Used by Gojek, Twitter, Shopify, Robinhood

Key Concepts

  • Entity — primary key (user_id, driver_id)
  • Feature View — group of related features from one source
  • Data Source — where raw data lives (Parquet, BigQuery, Kafka)
  • Feature Service — set of features for a specific model
  • Registry — metadata store for all definitions

Offline Path (Training)

SDK reads directly from the data source (warehouse/files). Performs point-in-time joins for historically accurate feature vectors.

Online Path (Serving)

feast materialize pushes latest values to the online store. Feature server returns values in < 10ms for real-time inference.

Defining Features

# feature_repo/features.py
from feast import Entity, FeatureView, Field, FileSource
from feast.types import Float32, Int64, String
from datetime import timedelta

# 1. Entity — the primary key
driver = Entity(
    name="driver",
    join_keys=["driver_id"],
    description="Unique driver identifier",
)

# 2. Data Source — where raw features live
driver_stats_source = FileSource(
    path="data/driver_stats.parquet",
    timestamp_field="event_timestamp",
)

# 3. Feature View — a group of features from one source
driver_stats_fv = FeatureView(
    name="driver_hourly_stats",
    entities=[driver],
    ttl=timedelta(hours=2),
    schema=[
        Field(name="conv_rate", dtype=Float32),
        Field(name="acc_rate", dtype=Float32),
        Field(name="avg_daily_trips", dtype=Int64),
    ],
    source=driver_stats_source,
)

Feature Services & On-Demand

from feast import FeatureService

On-Demand Features

from feast import on_demand_feature_view

# Computed at request time
@on_demand_feature_view(
    sources=[driver_stats_fv],
    schema=[
        Field(name="trips_today_pct", dtype=Float32),
    ],
)
def driver_trips_pct(inputs):
    df = inputs["driver_hourly_stats"]
    df["trips_today_pct"] = (
        df["avg_daily_trips"] / 500.0
    )
    return df

# No materialization needed
# Runs in real-time during serving

Feast CLI Workflow

# Initialize a new feature repository
feast init my_feature_repo
cd my_feature_repo

# Project structure
my_feature_repo/
  feature_store.yaml     # Config: provider, online/offline stores
  features.py            # Entity, FeatureView, FeatureService definitions
  data/                  # Sample data (Parquet files)

# Apply definitions to the registry
feast apply

# Materialize features to the online store
feast materialize 2024-01-01T00:00:00 2024-12-31T23:59:59

# Incremental materialization (from last run to now)
feast materialize-incremental $(date -u +"%Y-%m-%dT%H:%M:%S")

# Start the feature server
feast serve --host 0.0.0.0 --port 6566

# View registered features
feast feature-views list
feast entities list

Local / Dev

SQLite online store + file offline store. Zero infrastructure. Great for prototyping.

AWS

DynamoDB online + Redshift/S3 offline. Registry on S3. Serverless scaling.

GCP

Datastore online + BigQuery offline. Registry on GCS. Native BigQuery integration.

Self-Hosted

Redis/Postgres online + Spark/Trino offline. Full control. Kubernetes-friendly.

Training: Historical Features

from feast import FeatureStore
import pandas as pd

store = FeatureStore(repo_path="feature_repo/")

# Entity DataFrame — "who" and "when" you want features for
entity_df = pd.DataFrame({
    "driver_id": [1001, 1002, 1003, 1001],
    "event_timestamp": [
        "2024-06-01 10:00:00",
        "2024-06-01 11:00:00",
        "2024-06-02 09:00:00",
        "2024-06-03 14:00:00",   # same driver, different time
    ],
})

# Point-in-time join — no future data leakage!
training_df = store.get_historical_features(
    entity_df=entity_df,
    features=[
        "driver_hourly_stats:conv_rate",
        "driver_hourly_stats:acc_rate",
        "driver_hourly_stats:avg_daily_trips",
    ],
).to_df()

# training_df now has features as they existed AT each timestamp
print(training_df.head())

store = FeatureStore(repo_path=".")

Point-in-Time Joins

The most critical capability of a feature store for correct training data.

Entity DF
driver_id + timestamp
→
Point-in-Time
Join Engine
→
Training DF
Features as-of timestamp

How It Works

  • For each entity row, find feature values that existed at or before that timestamp
  • Prevents using future data in training
  • Respects TTL — stale features return null
  • Handles multiple feature views with different time granularities

Example Timeline

Feature updates:
  09:00 conv_rate = 0.80
  10:30 conv_rate = 0.85
  12:00 conv_rate = 0.90

Entity request at 11:00:
  → Gets conv_rate = 0.85
  (latest value BEFORE 11:00)

Entity request at 09:30:
  → Gets conv_rate = 0.80
  (latest value BEFORE 09:30)

Materialization & Scheduling

What Is Materialization?

  • Copies feature values from offline store to online store
  • Makes batch-computed features available for real-time serving
  • Runs on a schedule (hourly, daily)
  • Incremental — only processes new data since last run
# Full materialization
feast materialize \
  2024-01-01T00:00:00 \
  2024-12-31T23:59:59

# Incremental (last run to now)
feast materialize-incremental \
  $(date -u +"%Y-%m-%dT%H:%M:%S")

Scheduling Options

  • Airflow — DAG runs feast materialize-incremental
  • Kubernetes CronJob — scheduled pod
  • GitHub Actions — cron trigger
  • Prefect / Dagster — workflow orchestrators
# Airflow DAG example
materialize_task = BashOperator(
    task_id="feast_materialize",
    bash_command=(
        "feast -c /repo "
        "materialize-incremental "
        "$(date -u +%Y-%m-%dT%H:%M:%S)"
    ),
    dag=dag,

Feature Engineering Patterns

Aggregation Windows

Compute stats over time windows: avg_trips_7d, total_revenue_30d. Pre-compute in batch, serve online.

Embedding Features

Store pre-computed embeddings (text, image, user) as feature vectors. Serve for similarity search or model input.

Cross-Entity Features

Combine features across entities: user_avg_order_value joined with product_popularity.

Freshness Tiers

Real-time (streaming, seconds), Near-real-time (hourly batch), Daily (nightly ETL).

Feature Versioning

Version feature views: driver_stats_v1, driver_stats_v2. Old models keep reading old features during migration.

On-Demand Transforms

Compute at request time from stored features + request data. No materialization needed. Great for derived ratios.

Best Practices

1. One feature view per data source — keep lineage clean
2. Always set TTL to prevent serving stale features
3. Use Feature Services to bundle features per model
4. Name features descriptively: entity_metric_window
5. Store feature definitions in version control (GitOps)
6. Monitor feature freshness and null rates in production
7. Use point-in-time joins for all training data — never join naively
8. Start with batch features, add streaming only when latency requires it
9. Run feast apply in CI/CD to validate definitions before deploy
10. Test features with great_expectations or Deequ for data quality
