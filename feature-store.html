What Is a Feature?

A feature is a measurable property or attribute of data that serves as input to a machine learning model. It is how raw data becomes something a model can learn from.

Raw Data

  • BTCUSDT last price: $67,432.50 at 14:03:22
  • Funding rate: +0.012% at 08:00 UTC
  • A whale moved 5,000 BTC to Coinbase
  • 24h trading volume: $18.2B
  • RSI(14) on 1h candle: 72.3

Features (model inputs)

  • rsi_14_1h = 72.3
  • funding_rate_trend_24h = +0.008
  • whale_net_flow_4h = -12,500 BTC
  • volume_ratio_24h_7d = 1.45
  • atr_14_pct = 3.2%

Features are derived from raw data through transformations — aggregations, calculations, encodings, and business logic.

Why Features Matter

Model Quality

Better features = better models. XGBoost with well-engineered rsi, atr, and funding_rate features outperforms a deep model fed raw candles.

Reusability

rsi_14_1h powers the signal model, regime classifier, and risk evaluator. Compute once, serve to all.

Consistency

If atr_14 is computed differently in training vs. live trading, the model's position sizing will be wrong — real money lost.

Time Sensitivity

Crypto markets run 24/7. funding_rate_trend_24h right now is different from 8 hours ago. Models need the right value at the right time.

How Features Are Used

Training (Batch)

  • Collect historical features for past trading signals
  • Each training row: features + label (profitable or not)
  • Features must reflect what was known at that time
  • Point-in-time: no peeking at future prices!
# XGBoost signal model training data
| instrument | rsi_14 | atr_pct | funding | profitable |
|------------|--------|---------|---------|------------|
| BTCUSDT    | 72.3   | 3.2%    | +0.012  | 0          |
| ETHUSDT    | 31.5   | 4.1%    | -0.005  | 1          |
| BTCUSDT    | 55.0   | 2.8%    | +0.001  | 1          |

Serving (Real-Time)

  • Fetch latest features for an instrument
  • Must be fast (< 2s) for live trading decisions
  • Model returns signal: direction + confidence
  • Risk gate checks before execution
# Live trading signal flow
New tick: BTCUSDT @ $67,432

1. Fetch features for BTCUSDT:
   rsi_14_1h=72.3, atr_14_pct=3.2%
   funding_trend=+0.008, regime=TRENDING
2. XGBoost model returns:
   direction=SELL, strength=0.7, confidence=0.82
3. Risk gate: position size, drawdown check
4. Execute order (or reject)

The Problem

Without a Feature Store

  • Training: RSI computed in a Python notebook
  • Serving: RSI recomputed in .NET trading engine
  • Training/serving skew — subtle differences = bad trades
  • Signal model, regime classifier, and risk gate all compute atr_14 independently
  • Backtesting uses features that didn't exist at that historical moment

With a Feature Store

  • Single source of truth — one rsi_14 definition shared everywhere
  • XGBoost trains on the exact features it will see in production
  • Point-in-time joins: backtests use features as they existed at that moment
  • Feature catalog: 500+ features searchable by name, entity, or owner
  • Decision replay: exact feature snapshot stored with every trade

Why Not Just a Postgres Table?

You could store features in a regular table. Here's what you'd have to build yourself:

CapabilityPostgres TableFeature Store (Feast)
Store feature valuesYes — just rows and columnsYes — same, but structured
Low-latency serving (<10ms)Slow — 5-50ms per query under load, no built-in cachingBuilt-in — Redis/DynamoDB online store, sub-ms reads
Point-in-time joinsManual — complex SQL with window functions, error-proneBuilt-in — one API call, handles TTL and multiple time granularities
Training/serving consistencyYour problem — must ensure Python training code and .NET serving code compute identicallyGuaranteed — same feature definition serves both paths
Feature catalog & discoveryNone — "which table has RSI? Is it rsi_14 or rsi14_1h?"Registry — feast feature-views list, metadata, descriptions, ownership
Streaming ingestionCustom — build Kafka → Postgres pipeline yourselfNative — KafkaSource with watermark, auto-materialization
Feature versioningSchema migrations — ALTER TABLE, hope nothing breaksDeclarative — technical_v1 and technical_v2 coexist, feast apply validates
Offline + Online splitOne store — can't optimize for both batch reads and real-time lookupsDual store — warehouse for training, Redis for serving, synced by materialization

Postgres is great for storing data. A feature store adds the ML-specific plumbing — point-in-time correctness, dual-store serving, and a contract between training and production.

When Postgres IS Enough

Skip the Feature Store If...

  • You have < 20 features and 1-2 models
  • Batch predictions only (no real-time serving)
  • Single team, single codebase (no sharing needed)
  • Latency budget is relaxed (> 100ms is fine)
  • No need for historical point-in-time joins
  • You're prototyping or in early MVP stage

A features table with good indexes can work well for simple cases.

You Need a Feature Store When...

  • 500+ features across multiple timeframes (like our trading agent)
  • Real-time serving under < 10ms latency
  • Multiple models consuming overlapping features
  • Backtesting requires exact historical feature state
  • Streaming + batch features need to coexist
  • Multiple teams or services produce and consume features

The trading agent hits every item on this list.

What Is a Feature Store?

A feature store is a centralized platform for defining, storing, and serving ML features consistently across training and inference.

Offline Store

Historical feature values for training. Supports point-in-time joins. Backed by data warehouses (BigQuery, Snowflake, Redshift, files).

Online Store

Latest feature values for real-time serving. Low-latency lookups (< 10ms). Backed by Redis, DynamoDB, or SQLite.

Feature Registry

Catalog of all feature definitions, metadata, ownership, and lineage. The "schema" of your ML data.

Feature Serving

APIs and SDKs for fetching features at training time (batch) and inference time (real-time).

Training/Serving Skew

The #1 reason ML models degrade in production.

CauseWhat Goes Wrong (Trading)Feature Store Fix

Feast — Open-Source Feature Store

What Is Feast?

  • Open-source feature store (Linux Foundation / LF AI)
  • Python-first, declarative feature definitions
  • Supports batch, streaming, and on-demand features
  • Pluggable offline/online stores
  • Built-in feature server (REST/gRPC)
  • Used by Gojek, Twitter, Shopify, Robinhood

Key Concepts

  • Entity — primary key (user_id, driver_id)
  • Feature View — group of related features from one source
  • Data Source — where raw data lives (Parquet, BigQuery, Kafka)
  • Feature Service — set of features for a specific model
  • Registry — metadata store for all definitions

Defining Features

# feature_repo/features.py  (AI Crypto Trading Agent)
from feast import Entity, FeatureView, Field, FileSource
from feast.types import Float32, Float64, Int64, String
from datetime import timedelta

# 1. Entity — the trading instrument (BTCUSDT, ETHUSDT, etc.)
instrument = Entity(
    name="instrument",
    join_keys=["instrument_id"],
    description="Trading pair (e.g. BTCUSDT)",
)

# 2. Data Source — technical indicators from TimescaleDB/Parquet
technical_source = FileSource(
    path="data/technical_features.parquet",
    timestamp_field="event_timestamp",
)

# 3. Feature View — technical indicator features per instrument
technical_fv = FeatureView(
    name="instrument_technical_1h",
    entities=[instrument],
    ttl=timedelta(hours=2),
    schema=[
        Field(name="rsi_14", dtype=Float32),
        Field(name="macd_signal", dtype=Float32),
        Field(name="atr_14_pct", dtype=Float32),
        Field(name="bbands_width", dtype=Float32),
        Field(name="volume_ratio_24h_7d", dtype=Float32),
    ],
    source=technical_source,
)

On-Demand Features

from feast import on_demand_feature_view

# Computed at request time (no materialization)
@on_demand_feature_view(
    sources=[technical_fv, funding_fv],
    schema=[
        Field(name="risk_adjusted_signal", dtype=Float32),
    ],
)
def compute_risk_signal(inputs):
    tech = inputs["instrument_technical_1h"]
    fund = inputs["instrument_funding"]
    # Combine ATR volatility with funding pressure
    tech["risk_adjusted_signal"] = (
        (100 - tech["rsi_14"]) / 100.0
        * (1 - abs(fund["funding_rate"]) * 10)
    )
    return tech

# Runs in real-time during serving
# Great for derived ratios and cross-view logic

Training: Historical Features

from feast import FeatureStore
import pandas as pd

store = FeatureStore(repo_path="feature_repo/")

# Entity DataFrame — historical trading signals we want to label
# "what instrument, at what moment did we generate a signal?"
entity_df = pd.DataFrame({
    "instrument_id": ["BTCUSDT", "ETHUSDT", "BTCUSDT", "BTCUSDT"],
    "event_timestamp": [
        "2024-06-01 10:00:00",   # signal during trending regime
        "2024-06-01 11:00:00",   # ETH signal
        "2024-06-02 03:15:00",   # overnight BTC signal
        "2024-06-03 14:00:00",   # same instrument, different market state
    ],
})

# Point-in-time join — no future price leakage!
training_df = store.get_historical_features(
    entity_df=entity_df,
    features=[
        "instrument_technical_1h:rsi_14",
        "instrument_technical_1h:atr_14_pct",
        "instrument_technical_1h:macd_signal",
        "instrument_funding:funding_rate",
        "instrument_funding:funding_rate_trend_24h",
        "market_regime:regime_label",
    ],
).to_df()

# training_df has features exactly as they existed AT each signal timestamp
# Join with trade outcomes (P&L) to create labeled training data

Point-in-Time Joins

The most critical capability of a feature store for correct training data.

Signal Events
instrument + timestamp
→
Point-in-Time
Join Engine
→
Training DF
Features as-of signal time

Why It Matters for Trading

  • For each historical signal, find features at or before that moment
  • Prevents using future candle data — no look-ahead bias
  • TTL=2h: if last rsi_14 update was 3h ago, returns null (stale)
  • Joins technical (1h), funding (8h), and sentiment (5m) features at different granularities

Example: BTCUSDT RSI Timeline

RSI-14 feature updates (1h candles):
  09:00 rsi_14 = 58.2
  10:00 rsi_14 = 64.7
  11:00 rsi_14 = 72.3

Signal generated at 10:45:
  → Gets rsi_14 = 64.7
  (latest value BEFORE 10:45)
  NOT 72.3 (that candle hasn't closed yet!)

Signal generated at 09:30:
  → Gets rsi_14 = 58.2
  (latest value BEFORE 09:30)

Materialization & Scheduling

What Is Materialization?

  • Copies feature values from offline store to online store
  • Makes batch-computed features available for real-time serving
  • Runs on a schedule (hourly, daily)
  • Incremental — only processes new data since last run
# Full materialization
feast materialize \
  2024-01-01T00:00:00 \
  2024-12-31T23:59:59

# Incremental (last run to now)
feast materialize-incremental \
  $(date -u +"%Y-%m-%dT%H:%M:%S")

Scheduling Options

  • Airflow — DAG runs feast materialize-incremental
  • Kubernetes CronJob — scheduled pod
  • GitHub Actions — cron trigger
  • Prefect / Dagster — workflow orchestrators
# Airflow DAG example
materialize_task = BashOperator(
    task_id="feast_materialize",
    bash_command=(
        "feast -c /repo "
        "materialize-incremental "
        "$(date -u +%Y-%m-%dT%H:%M:%S)"
    ),
    dag=dag,
)

Feature Engineering Patterns

Aggregation Windows

Compute stats over time windows: avg_volume_24h, funding_rate_trend_7d, realized_vol_30d. Pre-compute in Kafka Streams, serve from Redis.

Market Regime Features

HMM classifier outputs regime_label (TRENDING, RANGING, VOLATILE, CRISIS) every 15 min. Strategies activate/deactivate based on regime.

Cross-Entity Features

Combine instrument features with market-wide features: eth_rsi_14 + btc_dominance_trend + market_regime for correlated signals.

Freshness Tiers

Hot (Kafka → Redis, seconds), Warm (Feast materialization, minutes), Cold (TimescaleDB, batch nightly).

Feature Versioning

Version feature views: technical_v1, technical_v2. Old XGBoost model keeps reading v1 while new model trains on v2.

On-Demand Transforms

Compute risk_adjusted_signal at request time from stored RSI + funding rate. No materialization needed. Great for position sizing ratios.

Best Practices

1. One feature view per data source — keep lineage clean
2. Always set TTL to prevent serving stale features
3. Use Feature Services to bundle features per model
4. Name features descriptively: entity_metric_window
5. Store feature definitions in version control (GitOps)
6. Monitor feature freshness and null rates in production
7. Use point-in-time joins for all training data — never join naively
8. Start with batch features, add streaming only when latency requires it
9. Run feast apply in CI/CD to validate definitions before deploy
10. Test features with great_expectations or Deequ for data quality
