What is Railway?

  • A modern Platform-as-a-Service (PaaS) for deploying apps, databases, and services
  • Deploy from GitHub, CLI, or Docker with zero configuration needed
  • Automatic builds via Nixpacks — detects language, installs deps, builds, and runs
  • Instant deploys with zero-downtime rollouts and automatic HTTPS
  • Built-in databases: PostgreSQL, MySQL, Redis, MongoDB in one click
# Deploy your app in seconds
npm install -g @railway/cli
railway login
railway init
railway up

# That's it. Railway detects your framework,
# builds it, and gives you a public URL.

Railway Architecture

Project
→
Environments
→
Services
→
Deployments

Projects

Top-level container. Groups all services, databases, and config for one application.

Services

Individual deployable units: web apps, APIs, workers, cron jobs, or databases.

Environments

Isolated copies: production, staging, PR previews. Each has its own variables and domains.

Volumes

Persistent storage attached to services. Survives redeploys. Mounted at any path.

  ├── PostgreSQL           # Service (database plugin)
  └── Redis                # Service (database plugin)

# Each has: production + staging environments

Getting Started

# 1. Install the Railway CLI
npm install -g @railway/cli
# or
brew install railway

# 2. Authenticate
railway login
# Opens browser for OAuth login

# 3. Initialize a new project
cd my-app/

# 4. Deploy
railway up
# Uploads code, builds, and deploys

# 5. Open in browser

Docker & Nixpacks

Nixpacks (Default)

Railway auto-detects your language and builds without a Dockerfile. Supports Node.js, Python, Go, Rust, Java, Ruby, PHP, and more.

# No config needed! Railway detects:
# - package.json -> Node.js
# - requirements.txt -> Python
# - go.mod -> Go
# - Cargo.toml -> Rust
# - pom.xml -> Java

# Override Nixpacks settings:
# railway.toml or nixpacks.toml
[phases.setup]
nixPkgs = ["...", "ffmpeg"]

[phases.build]
cmds = ["npm run build"]

[start]
cmd = "npm run start"

Custom Dockerfile

If a Dockerfile exists in your root, Railway uses it automatically. Full control over the build process.

# Dockerfile
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
EXPOSE 3000
CMD ["node", "dist/index.js"]

# Railway reads EXPOSE and sets PORT
# Or set PORT in env variables
          
# Force builder type in railway.json
{
  "build": {
    "builder": "DOCKERFILE",
    "dockerfilePath": "./Dockerfile.production"
  }
}

# Use a Docker image directly (no build)
# Dashboard: New Service > Docker Image > ghcr.io/org/image:latest

Networking

Public Networking

  • Auto-generated .up.railway.app domains
  • Custom domains with automatic SSL
  • TCP proxy for non-HTTP services
  • HTTP/2 and WebSocket support

Private Networking

  • Internal DNS: service.railway.internal
  • Zero-latency service-to-service
  • No egress charges for internal traffic
  • IPv6 only (handled transparently)
# Generate a public domain
# Dashboard: Service > Settings > Networking > Generate Domain
# Result: my-app-production.up.railway.app

# Add custom domain
# Dashboard: Service > Settings > Networking > Custom Domain
# Add CNAME record: app.example.com -> my-app-production.up.railway.app
# SSL certificate provisioned automatically

# TCP Proxy (for databases, gRPC, custom protocols)
# Dashboard: Service > Settings > Networking > TCP Proxy
# Gives you: roundhouse.proxy.rlwy.net:12345

# Port configuration
# Railway reads PORT env var (auto-set) or EXPOSE in Dockerfile
# Your app MUST listen on 0.0.0.0, not 127.0.0.1
PORT=3000  # Railway sets this automatically

# Private networking between services
# API calling worker service internally:

Volumes & Storage

# Volumes provide persistent storage that survives redeploys
# Dashboard: Service > Settings > Volumes > Add Volume

# Configuration:
# Mount Path: /data          (where it appears in your container)
# Size: 5 GB - 50 GB         (expandable, charged per GB)

# Example: Persistent uploads directory
# Mount Path: /app/uploads
# Your app writes to /app/uploads and data persists across deploys

Volume Use Cases

  • Database data directories
  • File upload storage
  • SQLite databases
  • Cache directories
  • ML model storage

Volume Limits & Tips

  • One volume per service
  • Cannot be shared between services
  • Attached to a single region
  • Backup with railway run pg_dump
  • Use S3/R2 for large file storage
# Backup strategy for PostgreSQL
railway run pg_dump -Fc $DATABASE_URL > backup.dump

# Restore from backup
railway run pg_restore -d $DATABASE_URL backup.dump

# For file uploads, prefer object storage:
# - Cloudflare R2 (S3-compatible, no egress fees)
# - AWS S3
# - Store URLs in your database, files in object storage

Scaling

Horizontal Scaling

Run multiple replicas of a service. Traffic automatically load-balanced.

Replicas: 1 - 10+

Vertical Scaling

Adjust CPU and RAM limits per service. Resources scale automatically up to limits.

Up to 32 vCPU / 32 GB RAM

Autoscaling

Scale replicas based on CPU, memory, or request count thresholds.

Min: 1, Max: 10, Target: 70% CPU

Sleep / Wake

Services can sleep when idle and wake on incoming requests. Saves costs for low-traffic apps.

Idle timeout configurable 
# Configure scaling in railway.json
{
  "deploy": {
    "numReplicas": 3,
    "startCommand": "npm start",
    "healthcheckPath": "/health"
  }
}

# Or via Dashboard:
# Service > Settings > Scaling
# - Set replica count (horizontal)
# - Set vCPU limit (e.g., 2 vCPU)
# - Set memory limit (e.g., 2 GB)

# Important for horizontal scaling:
# - Your app must be STATELESS (use Redis for sessions)
# - Health checks required for load balancing
# - Volumes cannot be used with multiple replicas

CI/CD Integration

# GitHub Actions + Railway
# .github/workflows/deploy.yml
name: Deploy to Railway
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Run Tests
        run: npm ci && npm test

      - name: Install Railway CLI
        run: npm install -g @railway/cli

      - name: Deploy
        run: railway up --service web-app
        env:
          RAILWAY_TOKEN: ${{ secrets.RAILWAY_TOKEN }}

10 Production Best Practices

1 Use environment variables for all config — never hardcode secrets
2 Add a /health endpoint and configure health checks in deploy settings
3 Handle SIGTERM gracefully — finish in-flight requests before shutting down
4 Use private networking for service-to-service and database connections
5 Set CPU and memory limits to prevent runaway costs and OOM crashes
6 Run database migrations as a deploy command, not on app startup
7 Use structured JSON logging for easier filtering and alerting
8 Keep services stateless — store sessions in Redis, files in S3/R2
9 Set up staging environments that mirror production configuration
10 Automate database backups with cron services or external backup tools
# Graceful shutdown example (Node.js)
process.on("SIGTERM", async () => {
  console.log("SIGTERM received, shutting down gracefully...");
  server.close(() => {
    db.disconnect();
    process.exit(0);
  });
  setTimeout(() => process.exit(1), 10000); // Force after 10s
});
