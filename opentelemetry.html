What Is OpenTelemetry?

Overview

  • Vendor-neutral observability framework
  • CNCF incubating project (2nd most active after Kubernetes)
  • Merger of OpenTracing + OpenCensus
  • Single set of APIs, SDKs, and tools
  • Collects, processes, and exports telemetry data

What It Provides

  • APIs — language-specific interfaces for instrumentation
  • SDKs — implementations with processing and export
  • Collector — vendor-agnostic proxy for telemetry
  • OTLP — standard wire protocol
  • Conventions — semantic conventions for naming

The Three Signals

Traces

Follow a request across service boundaries. Each trace is a tree of spans showing the full lifecycle of an operation.

TraceID: abc123 → 6 spans across 4 services

Metrics

Numeric measurements aggregated over time. Counters, histograms, and gauges for throughput, latency, and resource usage.

trade.execution.latency_ms: p99 = 12ms

Logs

Structured event records correlated with traces. Each log entry can carry a TraceID and SpanID for context.

{"traceId":"abc123","msg":"Risk gate passed"}

Unified Observability

All three signals share context. Jump from a slow metric to the trace that caused it, then to the logs within that span.

metric → trace → span → log

Why OpenTelemetry?

The Problem

  • Vendor lock-in with proprietary agents (Datadog, New Relic, Dynatrace)
  • Different SDKs for traces, metrics, and logs
  • Switching backends means re-instrumenting everything
  • Inconsistent attribute naming across services
  • No standard way to propagate context across polyglot services

The OTel Solution

  • Single SDK — instrument once, export anywhere
  • Vendor neutral — swap Tempo for Jaeger in config, not code
  • Community standard — 700+ contributors, every cloud vendor supports it
  • Polyglot — consistent APIs across .NET, Python, Java, Go, JS
  • Future-proof — CNCF backing ensures long-term support

Auto-Instrumentation

.NET Auto-Instrumentation

// Program.cs - add OTel with auto-instrumentation
builder.Services.AddOpenTelemetry()
  .WithTracing(tracing => tracing
    .AddAspNetCoreInstrumentation()
    .AddGrpcClientInstrumentation()
    .AddHttpClientInstrumentation()
    .AddSource("Confluent.Kafka")
    .AddEntityFrameworkCoreInstrumentation()
    .AddOtlpExporter(o =>
      o.Endpoint = new Uri(
        "http://otel-collector:4317"))
  )
  .WithMetrics(metrics => metrics
    .AddAspNetCoreInstrumentation()
    .AddHttpClientInstrumentation()
    .AddRuntimeInstrumentation()
    .AddPrometheusExporter()
  );

What You Get for Free

  • gRPC — spans for every RPC call with method, status
  • HTTP — inbound/outbound request spans with URL, status code
  • Kafka — produce/consume spans with topic, partition, offset
  • EF Core — database query spans with SQL command text
  • Runtime — GC, thread pool, heap metrics
  • ASP.NET Core — request duration, active requests

Zero code changes for 80% of your trace coverage.

Manual Instrumentation

.NET Custom Spans

// Define an ActivitySource (OTel span source)
private static readonly ActivitySource
  Source = new("TradingAgent.RiskGate");

public async Task<bool> EvaluateRisk(
  Order order)
{
  using var activity = Source
    .StartActivity("EvaluateRisk");

  // Add attributes
  activity?.SetTag("order.symbol",
    order.Symbol);
  activity?.SetTag("order.size",
    order.Size);
  activity?.SetTag("order.side",
    order.Side.ToString());

  try {
    var result = await _riskEngine
      .Check(order);
    activity?.SetTag("risk.approved",
      result.Approved);
    activity?.SetStatus(
      ActivityStatusCode.Ok);
    return result.Approved;
  } catch (Exception ex) {
    activity?.SetStatus(
      ActivityStatusCode.Error,
      ex.Message);
    activity?.RecordException(ex);
    throw;
  }
}

from opentelemetry import trace

tracer = trace.get_tracer(
    "trading_agent.ml_model")

def predict_signal(features: dict):
    with tracer.start_as_current_span(
        "predict_signal"
    ) as span:
        span.set_attribute(
            "model.name", "xgboost-v3")
        span.set_attribute(
            "features.count", len(features))

        try:
            prediction = model.predict(
                features)
            span.set_attribute(
                "prediction.signal",
                prediction.signal)
            span.set_attribute(
                "prediction.confidence",
                prediction.confidence)

            # Add event for audit trail
            span.add_event(
                "signal_generated",
                {"signal": prediction.signal,
                 "confidence":
                   prediction.confidence})

            return prediction
        except Exception as e:
            span.set_status(
                trace.Status(
                  trace.StatusCode.ERROR,
                  str(e)))
            span.record_exception(e)
            raise

The OTel Collector

What It Does

  • Receives — accepts telemetry over OTLP, Jaeger, Zipkin, Prometheus
  • Processes — batching, filtering, sampling, attribute enrichment
  • Exports — forwards to one or more backends simultaneously
  • Decouples apps from backends — change destination without redeploying apps
  • Handles retries, buffering, and backpressure

Deployment Patterns

  • Agent (DaemonSet) — one per node, low latency, local collection
  • Sidecar — one per pod, isolated resource limits
  • Gateway — centralized, handles tail sampling and routing
  • Agent + Gateway — best of both: local buffering with centralized processing

Metrics with OTel

Metric Types

  • Counter — monotonically increasing (total trades executed)
  • UpDownCounter — can increase or decrease (active orders)
  • Histogram — distribution of values (trade latency)
  • Gauge — point-in-time value (current portfolio value)

OTel vs Prometheus Native

  • OTel uses push (OTLP); Prometheus uses pull (scrape)
  • OTel SDK can export in Prometheus format
  • Both can coexist: use OTel SDK with Prometheus exporter
  • OTel adds exemplars — link a metric data point to a trace
// .NET - Custom trading metrics
private static readonly Meter meter = new("TradingAgent.Metrics");

// Counter: total trades
private static readonly Counter<long> tradesExecuted =
    meter.CreateCounter<long>("trades.executed.total",
        description: "Total trades executed");

// Histogram: execution latency
private static readonly Histogram<double> execLatency =
    meter.CreateHistogram<double>("trades.execution.latency_ms",
        unit: "ms", description: "Trade execution latency");

// Record metrics
tradesExecuted.Add(1, new("symbol", "BTC-USD"), new("side", "BUY"));
execLatency.Record(12.5, new("symbol", "BTC-USD"));

Sampling Strategies

Head-Based Sampling

Decision made at trace start. Fast, low overhead. Cannot consider downstream events.

TraceIdRatioBased(0.1) = 10% of traces

Tail-Based Sampling

Decision made after trace completes. Keeps errors and slow traces. Requires Collector gateway.

Keep if latency > 500ms or status = ERROR

Rate Limiting

Caps traces per second. Protects backends from burst traffic during market volatility.

rate_limiting: { spans_per_second: 500 }

Always-On for Errors

Sample 100% of error traces, 10% of success. Never miss a failed trade execution.

ParentBased(root=TraceIdRatio(0.1)) 
# Tail-based sampling in Collector config (keep errors + slow trades)
processors:
  tail_sampling:
    decision_wait: 10s
    policies:
      - name: errors-always
        type: status_code
        status_code: { status_codes: [ERROR] }
      - name: slow-trades
        type: latency
        latency: { threshold_ms: 500 }
      - name: percentage-sample
        type: probabilistic
        probabilistic: { sampling_percentage: 10 }

OTel in Python

Instrumenting ML model serving and XGBoost inference

# setup_otel.py - Initialize OpenTelemetry for Python ML service
from opentelemetry import trace, metrics
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.instrumentation.kafka import KafkaInstrumentor
from opentelemetry.instrumentation.requests import RequestsInstrumentor

# Auto-instrument Kafka consumer and HTTP requests
KafkaInstrumentor().instrument()
RequestsInstrumentor().instrument()

# Configure tracer with OTLP export to Collector
provider = TracerProvider()
provider.add_span_processor(
    BatchSpanProcessor(OTLPSpanExporter(endpoint="otel-collector:4317", insecure=True))
)
trace.set_tracer_provider(provider)

tracer = trace.get_tracer("trading_agent.ml")
meter = metrics.get_meter("trading_agent.ml")

# Custom metrics for ML model
inference_latency = meter.create_histogram("ml.inference.latency_ms", unit="ms")
prediction_counter = meter.create_counter("ml.predictions.total")

def run_xgboost_inference(features: dict) -> dict:
    with tracer.start_as_current_span("xgboost_inference") as span:
        span.set_attribute("model.version", "v3.2")
        span.set_attribute("features.count", len(features))

        start = time.perf_counter()
        prediction = model.predict(xgb.DMatrix(features))
        latency = (time.perf_counter() - start) * 1000

        inference_latency.record(latency, {"model": "xgboost-v3"})
        prediction_counter.add(1, {"signal": "BUY" if prediction > 0.5 else "SELL"})

        span.set_attribute("prediction.value", float(prediction))
        span.add_event("inference_complete", {"latency_ms": latency})
        return {"signal": "BUY" if prediction > 0.5 else "SELL", "confidence": float(prediction)}

OTel in Kubernetes

DaemonSet Collector

One collector per node. Apps send to localhost:4317. Low network hop, good for high-throughput trading agents.

Sidecar Collector

One collector per pod. Isolated resource limits per service. Use for services with unique processing needs.

Gateway Collector

Centralized deployment. Required for tail-based sampling. Handles cross-service trace decisions and routing.

OTel Operator

Kubernetes operator that manages Collector deployments, auto-instrumentation injection, and CRD-based config.

Best Practices

1. Use semantic conventions for attribute names (http.method, db.system, messaging.kafka.topic)
2. Always propagate context across async boundaries (Kafka headers, HTTP headers, gRPC metadata)
3. Use the Collector as a buffer between apps and backends — never export directly to storage
4. Implement tail-based sampling for error traces and slow operations to reduce storage costs
5. Set resource attributes (service.name, service.version, deployment.environment) on every signal
6. Record exceptions on spans with RecordException — never silently swallow errors
7. Use batch processors to reduce export overhead (batch size 512-2048, timeout 5s)
8. Add exemplars to metrics to link metric spikes to specific traces
9. Keep span names low-cardinality (use attributes for high-cardinality data like order IDs)
10. Monitor the Collector itself — track dropped spans, queue size, and export failures
