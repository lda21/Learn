What Is GitOps?

Core Principles

  • Declarative — desired state defined in code, not imperative scripts
  • Versioned & Immutable — Git is the single source of truth
  • Pulled Automatically — agents reconcile cluster to match Git
  • Continuously Reconciled — drift is detected and corrected

Why It Matters

  • Every change has an audit trail (git log)
  • Rollback = git revert
  • No more kubectl apply from laptops
  • PR review for infrastructure changes
  • Reproducible environments from a single repo

GitOps vs Traditional CI/CD

Traditional (Push Model)

  • CI pipeline pushes changes to cluster
  • Credentials stored in CI system
  • Cluster state unknown between deploys
  • Drift goes undetected
  • kubectl apply from CI runner

GitOps (Pull Model)

  • Agent inside cluster pulls from Git
  • No external cluster credentials needed
  • Continuous reconciliation loop
  • Drift auto-corrected to match Git
  • Git commit = deployment trigger

What Is ArgoCD?

  • Kubernetes-native GitOps continuous delivery controller
  • Watches Git repositories and syncs desired state to clusters
  • Rich web UI with real-time resource visualization
  • Supports Helm, Kustomize, Jsonnet, plain YAML
  • CNCF graduated project — production-proven at scale
  • Declarative Application CRD defines what to deploy and where

ArgoCD Architecture

API Server

gRPC/REST API powering the web UI and CLI. Handles authentication, RBAC enforcement, and listener/event management.

Repo Server

Clones Git repos, renders Helm/Kustomize templates, and returns plain manifests. Stateless and cacheable.

Application Controller

Watches live cluster state vs desired state in Git. Detects drift, triggers sync, and updates health status.

Redis (Internal Cache)

Caches Git repo metadata and rendered manifests. Reduces load on repo server and speeds up UI responsiveness.

- name: Update image tag in gitops repo

    kustomize edit set image \

Application CRD

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: trading-core
  namespace: argocd
spec:
  project: trading-agent
  source:
    repoURL: https://github.com/lda21/gitops-trading-agent.git
    targetRevision: main
    path: overlays/production/trading-core
  destination:
    server: https://kubernetes.default.svc
    namespace: trading-core
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
      - ApplyOutOfSyncOnly=true
    retry:
      limit: 3
      backoff:
        duration: 5s
        maxDuration: 3m0s
        factor: 2

Sync Policies

Auto-Sync

Automatically apply changes when Git differs from live state. No manual argocd app sync needed.

Self-Heal

Revert manual cluster changes back to Git state. Prevents drift from kubectl edit or direct mutations.

Prune

Delete resources removed from Git. Without pruning, orphaned resources accumulate in the cluster.

Retry

Automatic retry with exponential backoff on sync failures. Handles transient errors gracefully.

# Trading Agent: all namespaces use auto-sync + self-heal
# If someone kubectl-edits a Deployment, ArgoCD reverts it within seconds
syncPolicy:
  automated:
    prune: true      # remove deleted resources
    selfHeal: true   # revert manual changes
  syncOptions:
    - ApplyOutOfSyncOnly=true   # only sync changed resources
    - ServerSideApply=true      # avoid field ownership conflicts

Rollbacks & History

  • ArgoCD stores sync history with commit SHA, timestamp, and status
  • Rollback = sync to a previous Git revision (not just K8s rollback)
  • Automatic rollback on health check failure with Argo Rollouts
  • Full audit trail: who merged the PR, when ArgoCD synced
# View sync history
argocd app history trading-core

# ID  DATE                 REVISION
# 1   2026-03-28 10:15:00  abc1234  Synced   Healthy
# 2   2026-03-29 14:30:00  def5678  Synced   Healthy
# 3   2026-03-30 09:00:00  ghi9012  Synced   Degraded  <-- bad deploy

# Rollback to revision 2
argocd app rollback trading-core 2

# Or better: git revert the bad commit and let ArgoCD auto-sync
git revert ghi9012
git push origin main
# ArgoCD detects the revert and syncs back to healthy state

Best practice: Always git revert instead of argocd app rollback to keep Git as the source of truth.

RBAC & SSO

apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-rbac-cm
  namespace: argocd
data:
  policy.csv: |
    # Role: trading-team can sync trading-core and market-data
    p, role:trading-team, applications, get, trading-agent/*, allow
    p, role:trading-team, applications, sync, trading-agent/trading-core-*, allow
    p, role:trading-team, applications, sync, trading-agent/market-data-*, allow

    # Role: platform-team has full access
    p, role:platform-admin, applications, *, */*, allow
    p, role:platform-admin, clusters, *, *, allow

    # Map OIDC groups to roles
    g, trading-devs, role:trading-team
    g, platform-eng, role:platform-admin

Secrets Management

Sealed Secrets

# Encrypt secret client-side
kubeseal --format yaml \
  --cert pub-cert.pem \
  < secret.yaml > sealed-secret.yaml

# sealed-secret.yaml is safe to commit
apiVersion: bitnami.com/v1alpha1
kind: SealedSecret
metadata:
  name: trading-api-keys
  namespace: trading-core
spec:
  encryptedData:
    BINANCE_KEY: AgBy3i4OJSWK+...
    BINANCE_SECRET: AgCtr8JH2k...

# Only the cluster controller can decrypt

ExternalSecrets + AWS

# ExternalSecret pulls from AWS Secrets Manager
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: trading-api-keys
  namespace: trading-core
spec:
  refreshInterval: 5m
  secretStoreRef:
    name: aws-secrets-manager
    kind: ClusterSecretStore
  target:
    name: trading-api-keys
  data:
    - secretKey: BINANCE_KEY
      remoteRef:
        key: trading-agent/binance
        property: api_key
    - secretKey: BINANCE_SECRET
      remoteRef:
        key: trading-agent/binance
        property: api_secret

Never commit plain secrets to Git. Use Sealed Secrets or ExternalSecrets with AWS Secrets Manager.

Progressive Delivery

Argo Rollouts extends ArgoCD with canary and blue-green strategies for the trading services.

apiVersion: argoproj.io/v1alpha1
kind: Rollout
metadata:
  name: trading-core
  namespace: trading-core
spec:
  replicas: 3
  strategy:
    canary:
      steps:
        - setWeight: 10        # 10% traffic to new version
        - pause: { duration: 5m }
        - analysis:
            templates:
              - templateName: trading-success-rate
            args:
              - name: service-name
                value: trading-core
        - setWeight: 50        # 50% if analysis passes
        - pause: { duration: 10m }
        - setWeight: 100       # full rollout
      canaryService: trading-core-canary
      stableService: trading-core-stable
      trafficRouting:
        istio:
          virtualService:
            name: trading-core-vsvc

# AnalysisTemplate: auto-rollback if error rate > 1%
# Checks Prometheus: rate(http_errors_total[5m]) / rate(http_requests_total[5m])

Notifications

# argocd-notifications-cm ConfigMap
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-notifications-cm
  namespace: argocd
data:
  service.slack: |
    token: $slack-token
  trigger.on-sync-succeeded: |
    - when: app.status.sync.status == 'Synced'
      send: [app-sync-succeeded]
  trigger.on-health-degraded: |
    - when: app.status.health.status == 'Degraded'
      send: [app-health-degraded]
  template.app-sync-succeeded: |
    slack:
      attachments: |
        [{
          "color": "#18be52",
          "title": "{{.app.metadata.name}} synced",
          "text": "Commit: {{.app.status.sync.revision}}\nNamespace: {{.app.spec.destination.namespace}}"
        }]
  template.app-health-degraded: |
    slack:
      attachments: |
        [{
          "color": "#e5443a",
          "title": "ALERT: {{.app.metadata.name}} is Degraded",
          "text": "Check ArgoCD UI immediately.\nNamespace: {{.app.spec.destination.namespace}}"
        }]

ArgoCD at Scale

Multi-Cluster

  • Register external clusters with argocd cluster add
  • Single ArgoCD instance manages dev, staging, prod
  • Different sync policies per environment
  • Hub-and-spoke: one control plane, many clusters

ApplicationSets

# Generate apps from a list of namespaces
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: trading-services
spec:
  generators:
    - list:
        elements:
          - ns: trading-core
          - ns: market-data
          - ns: intelligence
          - ns: monitoring
          - ns: data
  template:
    metadata:
      name: '{{ns}}'
    spec:
      source:
        repoURL: https://github.com/lda21/gitops-trading-agent.git
        path: 'overlays/production/{{ns}}'
      destination:
        server: https://kubernetes.default.svc
        namespace: '{{ns}}'

ApplicationSets replace the App of Apps pattern for dynamic, template-driven app generation.

Best Practices

1. Separate app code repo from gitops config repo
2. Use Kustomize overlays for environment-specific config
3. Enable auto-sync + self-heal in production
4. Never commit plain secrets — use Sealed Secrets or ExternalSecrets
5. Use PreSync hooks for backward-compatible DB migrations (expand-and-contract)
6. Hot-reload ConfigMaps for risk parameters — no pod restart needed
7. Set up Slack notifications for sync, degraded, and rollback events
8. Use ApplicationSets for managing multiple namespaces at scale
9. Always git revert instead of manual argocd rollback
10. Require PR approval before any change reaches production clusters
