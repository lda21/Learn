Memory Safety Without Garbage Collection

C# vs Rust Mental Model

Both are strongly typed with generics. But Rust has no GC, no runtime, no null. The compiler enforces memory safety at compile time.

What's Familiar

  • Enums & pattern matching
  • Generics with constraints
  • Traits (like interfaces)
  • Async / await syntax
  • Strong static typing
  • Closures / lambdas
  • Iterator chains (like LINQ)

What's New

  • Ownership — replaces GC
  • Borrowing — compiler-checked references
  • Lifetimes — reference validity proofs
  • No inheritance — composition only
  • No null — Option<T> instead
  • No exceptions — Result<T,E> instead
  • No runtime — compiles to native code

No Garbage Collector — How?

In C#, the GC tracks references and frees memory when nothing points to it. In Rust, every value has exactly one owner. When the owner goes out of scope, the value is dropped (freed). No GC pauses, deterministic cleanup.

C# — Garbage Collector

// C# - GC handles cleanup
void Process() {
    var data = new List<int> { 1, 2, 3 };
    DoWork(data);
    // data still accessible here
    // GC frees it... eventually
    // You don't know WHEN
}

Rust — Ownership

// Rust - deterministic cleanup
fn process() {
    let data = vec![1, 2, 3];
    do_work(data);
    // data MOVED into do_work
    // dropped when do_work ends
    // freed EXACTLY when scope ends
}

Result: zero GC pauses, predictable memory usage, no "stop the world" collections.

Ownership — The Core Concept

In C#, var b = a; copies the reference — both point to the same object. In Rust, let b = a; moves the value — a is gone.

C# — Reference Copy

// C# - both variables point to same object
var a = new List<int> { 1, 2, 3 };
var b = a;  // copies the REFERENCE
a.Add(4);   // works fine!
// a and b both see [1, 2, 3, 4]
// Two aliases to same memory = bugs

Rust — Move Semantics

// Rust - ownership MOVES
let a = vec![1, 2, 3];
let b = a;  // a is MOVED to b
// println!("{:?}", a);  // COMPILE ERROR!
// "value used after move"
println!("{:?}", b);  // works: [1, 2, 3]
// One owner = no aliasing bugs

This prevents use-after-free, double-free, and data races at compile time. Types that implement Copy (integers, bools) are copied instead of moved.

Borrowing & References

What if you need to read data without taking ownership? Borrow it. The rule: many readers OR one writer, never both at the same time.

C# — ref / in Parameters

// C# - anyone can read/write anytime
void PrintLen(List<int> data) {
    Console.WriteLine(data.Count);
}
void AddItem(List<int> data) {
    data.Add(42); // mutates original!
}
// Both can run with same List
// No compile-time protection

Rust — &T and &mut T

fn print_len(data: &Vec<i32>) {
    println!("{}", data.len()); // read only
}
fn add_item(data: &mut Vec<i32>) {
    data.push(42); // exclusive access
}

let mut v = vec![1, 2, 3];
print_len(&v);       // shared borrow OK
print_len(&v);       // multiple readers OK
add_item(&mut v);    // exclusive borrow OK
// Can't mix & and &mut at same time!

Lifetimes

Lifetimes prevent dangling references. The 'a annotation tells the compiler "this reference lives at least as long as 'a". Usually inferred — you only write them when the compiler asks.

// This won't compile - dangling reference!
fn bad() -> &str {
    let s = String::from("hello");
    &s  // ERROR: s is dropped here, reference would dangle
}

// Lifetime annotations: "output lives as long as input"
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}

// The compiler needs 'a to know: does the return value
// come from x or y? Both must outlive the result.

// In C#, the GC keeps objects alive as long as any reference
// exists. In Rust, YOU prove it to the compiler.
// 90% of the time, lifetime elision handles it automatically.

The scariest syntax in Rust, but the compiler usually figures it out. When it asks for annotations, it is telling you to prove your references are valid.

Structs & Enums — Better Than C# Classes

Structs with Methods

// Like C# class but no inheritance
struct User {
    name: String,
    age: u32,
}

impl User {
    // Constructor (no "new" keyword)
    fn new(name: &str, age: u32) -> Self {
        User {
            name: name.to_string(),
            age,
        }
    }

    // Method (like C# instance method)
    fn greet(&self) -> String {
        format!("Hi, I'm {}", self.name)
    }
}

Enums with Data

// C# has nothing like this!
// Each variant can hold different data
enum Shape {
    Circle(f64),           // radius
    Rectangle(f64, f64),   // w, h
    Triangle { a: f64, b: f64, c: f64 },
}

// Exhaustive pattern matching
fn area(s: &Shape) -> f64 {
    match s {
        Shape::Circle(r) =>
            std::f64::consts::PI * r * r,
        Shape::Rectangle(w, h) => w * h,
        Shape::Triangle { .. } => todo!(),
    } // miss a variant = COMPILE ERROR
}

No Null, No Exceptions

C# has null and exceptions. Rust replaces them with Option<T> and Result<T,E>. The compiler forces you to handle every error path.

C# — Null + Exceptions

// C# - runtime surprises
string? name = GetName();
int len = name.Length;
// NullReferenceException!

try {
    var data = File.ReadAllText("f.txt");
} catch (IOException e) {
    // Easy to forget this
}

Rust — Option + Result

// Rust - compiler enforces handling
let name: Option<&str> = get_name();
// name.len() won't compile!
match name {
    Some(n) => println!("{}", n.len()),
    None => println!("no name"),
}

// ? operator propagates errors
fn read_file() -> Result<String, io::Error> {
    let data = fs::read_to_string("f.txt")?;
    Ok(data) // must be explicit
}

The ? operator is Rust's equivalent of try/catch — it returns early on error. You cannot forget to handle an error.

Traits — Like Interfaces But More

Traits are interfaces + default methods + can be implemented for types you did not write. No inheritance in Rust — composition via traits.

C# Interface

// C# - interface on your own types only
interface ISummable {
    string Summarize();
}

class Article : ISummable {
    public string Title { get; set; }
    public string Summarize()
        => $"Article: {Title}";
}

// Can't implement ISummable for
// System.String (not your type)

Rust Trait

trait Summary {
    fn summarize(&self) -> String;

    // Default implementation!
    fn preview(&self) -> String {
        format!("{}...", &self.summarize()[..20])
    }
}

struct Article { title: String }

impl Summary for Article {
    fn summarize(&self) -> String {
        format!("Article: {}", self.title)
    }
}

// CAN implement for external types!
impl Summary for String {
    fn summarize(&self) -> String {
        self.clone()
    }
}

Generics & Trait Bounds

Same concept as C# generic constraints but with a critical difference: monomorphization. Rust generates specialized code for each type at compile time. Zero overhead.

C# Generic Constraints

// C# - runtime generic dispatch
void Process<T>(T item)
    where T : IDisplay, ICloneable
{
    Console.WriteLine(item.Display());
    var copy = item.Clone();
}

// Uses vtable at RUNTIME
// JIT may or may not specialize

Rust Trait Bounds

// Rust - compile-time specialization
fn process<T: Display + Clone>(item: T) {
    println!("{}", item);
    let copy = item.clone();
}

// Alternative "where" syntax:
fn process2<T>(item: T)
where
    T: Display + Clone,
{
    println!("{}", item);
}

// Compiler generates a SPECIFIC version
// for each type used. Zero-cost.

Monomorphization means generics are as fast as hand-written type-specific code. No boxing, no vtable overhead.

Collections & Iterators

Rust's iterator chains are the equivalent of LINQ — lazy, composable, zero-cost.

C# LINQ

// C# collections & LINQ
var list = new List<int> { 1,2,3,4,5 };
var dict = new Dictionary<string, int>();
var set = new HashSet<int>();

// LINQ pipeline
var result = numbers
    .Where(n => n % 2 == 0)
    .Select(n => n * n)
    .Sum();

var names = users
    .Where(u => u.Age > 18)
    .Select(u => u.Name)
    .ToList();

Rust Iterators

// Rust equivalents
let list: Vec<i32> = vec![1,2,3,4,5];
let dict: HashMap<String, i32> = HashMap::new();
let set: HashSet<i32> = HashSet::new();

// Iterator pipeline (= LINQ)
let result: i32 = numbers.iter()
    .filter(|n| *n % 2 == 0)
    .map(|n| n * n)
    .sum();

let names: Vec<&str> = users.iter()
    .filter(|u| u.age > 18)
    .map(|u| u.name.as_str())
    .collect();

Vec<T> = List, HashMap = Dictionary, HashSet = HashSet, String = string. The .collect() call materializes the lazy iterator (like .ToList()).

Pattern Matching

match is exhaustive — the compiler forces you to handle all cases. Far more powerful than C# switch expressions.

// Exhaustive match (miss a variant = compile error)
match status_code {
    200 => println!("OK"),
    404 => println!("Not Found"),
    500..=599 => println!("Server Error"),  // ranges!
    _ => println!("Other"),                 // catch-all
}

// Destructuring enums
match user.role {
    Role::Admin { level } => format!("Admin L{}", level),
    Role::User => "Regular user".into(),
    Role::Guest => "Guest".into(),
}

// if let - when you only care about one case
if let Some(name) = get_name() {
    println!("Hello, {}", name);
}

// while let - loop until pattern fails
while let Some(item) = queue.pop() {
    process(item);
}

// matches! macro - returns bool
let is_admin = matches!(user.role, Role::Admin { .. });

Error Handling Deep Dive

C# throws exceptions anywhere and you hope someone catches them. Rust forces you to handle errors at every call site.

// The ? operator chains errors elegantly
fn load_config() -> Result<Config, Box<dyn Error>> {
    let text = fs::read_to_string("config.toml")?;  // returns Err early
    let config: Config = toml::from_str(&text)?;     // returns Err early
    Ok(config)
}

// Custom error types
#[derive(Debug)]
enum AppError {
    NotFound(String),
    Database(sqlx::Error),
    Validation(String),
}

// Map between error types
impl From<sqlx::Error> for AppError {
    fn from(e: sqlx::Error) -> Self {
        AppError::Database(e)
    }
}

// Popular crates: thiserror (library errors), anyhow (application errors)
// thiserror: #[derive(thiserror::Error)] for custom error types
// anyhow: anyhow::Result<T> for quick error handling with context

Closures & Fn Traits

Rust closures look like C# lambdas but interact with ownership. Three traits determine how they capture variables.

C# Lambdas

// C# - GC handles captures
Func<int, int> add = x => x + 1;
Action<string> log = s =>
    Console.WriteLine(s);

var factor = 10;
Func<int, int> scale = x => x * factor;
// Captures factor by reference
// GC keeps it alive

Rust Closures

// Rust - three closure traits
let add = |x: i32| x + 1;         // Fn
let log = |s: &str| println!("{}", s);

let factor = 10;
let scale = |x| x * factor; // borrows factor (Fn)

let mut count = 0;
let mut inc = || count += 1; // FnMut

let name = String::from("alice");
let greet = move || {        // FnOnce
    println!("Hi {}", name); // takes ownership
};
// name is MOVED, can't use it here

Fn = borrows immutably (like Func<>). FnMut = borrows mutably. FnOnce = takes ownership (can only call once). The move keyword forces ownership transfer.

Async / Await

Similar syntax to C# but different mechanics. Rust futures are lazy — they don't start until polled. You need a runtime (tokio).

C# async/await

// C# - Task starts immediately
async Task<string> FetchData(string url)
{
    var client = new HttpClient();
    // Starts the HTTP request NOW
    var response = await client
        .GetStringAsync(url);
    return response;
}

// .NET runtime provides thread pool
// Task = hot (already running)

Rust async/await

// Rust - Future is lazy
async fn fetch_data(url: &str)
    -> Result<String, reqwest::Error>
{
    // Nothing happens until .await
    let response = reqwest::get(url)
        .await?
        .text()
        .await?;
    Ok(response)
}

// Need a runtime (tokio)
#[tokio::main]
async fn main() {
    let data = fetch_data("https://...")
        .await.unwrap();
}
// Future = cold (runs when polled)

Smart Pointers

When single ownership is too restrictive, smart pointers give you controlled flexibility.

Box<T>

Heap allocation. Like a C# class instance. Use for recursive types, large data, or trait objects.

let b = Box::new(42);  // 42 on the heap

Rc<T>

Reference counted. Like C# references but explicit. Multiple owners, single-threaded only.

let a = Rc::new(vec![1,2,3]);
let b = Rc::clone(&a); // 2 owners

Arc<T>

Atomic Rc. Thread-safe reference counting. Use for shared data across threads.

let data = Arc::new(Mutex::new(0));
let d2 = Arc::clone(&data); // safe to send

RefCell<T>

Runtime borrow checking. Interior mutability when compiler can't prove safety statically.

let cell = RefCell::new(42);
*cell.borrow_mut() = 99; // checked at runtime

Fearless Concurrency

Rust's ownership system prevents data races at compile time. If it compiles, it won't have data races.

C# Concurrency

// C# - runtime data races possible
var counter = 0;
var tasks = Enumerable.Range(0, 10)
    .Select(_ => Task.Run(() => {
        // DATA RACE! No compiler warning
        counter++;
    }));
await Task.WhenAll(tasks);

// Fix: use lock or Interlocked
lock(lockObj) { counter++; }

Rust Concurrency

// Rust - compiler prevents data races
let counter = Arc::new(Mutex::new(0));
let mut handles = vec![];
for _ in 0..10 {
    let c = Arc::clone(&counter);
    handles.push(thread::spawn(move || {
        let mut num = c.lock().unwrap();
        *num += 1;  // exclusive access
    }));
}
for h in handles { h.join().unwrap(); }
// Without Arc+Mutex = COMPILE ERROR
// Send + Sync traits enforce thread safety

Also available: channels (mpsc::channel), scoped threads, async tasks with tokio. The type system guarantees safety.

Macros — Compile-Time Code Generation

Rust has no runtime reflection. Macros replace it — they generate code at compile time. Think C# source generators but built into the language.

// Built-in macros (note the !)
println!("Hello, {}!", name);        // formatted print
let v = vec![1, 2, 3];              // create Vec
format!("Score: {}", 42);           // string formatting
todo!();                            // placeholder
assert_eq!(a, b);                   // test assertion

// Derive macros = auto-implement traits (like C# attributes)
#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
struct User {
    name: String,
    age: u32,
}
// Debug    = C# ToString() override
// Clone    = C# ICloneable
// Serialize = C# [JsonSerializable] (but compile-time!)

// serde = Rust's serialization framework
// Unlike System.Text.Json, it generates code at compile time
// No reflection overhead, no runtime surprises
let json = serde_json::to_string(&user)?;
let user: User = serde_json::from_str(&json)?;

When to Use Rust

Use Rust For

  • CLI tools — fast startup, single binary
  • Systems programming — OS, drivers, embedded
  • WebAssembly — first-class WASM support
  • Game engines — Bevy, performance-critical
  • Network services — proxies, load balancers
  • Performance-critical libraries — called from any language via FFI
  • The hot path — rewrite bottlenecks in Rust

Stay in C# For

  • Enterprise CRUD apps — EF Core, ASP.NET
  • Rapid prototyping — faster iteration
  • Large team projects — lower learning curve
  • GUI apps — WPF, MAUI, Blazor
  • Game logic — Unity ecosystem
  • Business logic — where GC overhead is fine

Interop via FFI: extern "C" lets Rust and C# call each other. Use Rust for the hot path, C# for business logic.
