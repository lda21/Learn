What Is Claw Code?

Origin Story

  • Clean-room reimplementation of Claude Code's agent harness
  • Captures architectural patterns without copying proprietary code
  • Built using oh-my-codex (OmX) orchestration
  • Repository: instructkr/claw-code

Key Numbers

  • 207 command entries mirrored
  • 184 tool entries mirrored
  • ~20K lines of Rust across 6 crates
  • 100+ Python source files
  • Original TS: 1,902 files across 35+ subsystems

Why Study This Repo?

  • Agent Harness Engineering — Learn how production AI agent loops are built from scratch
  • Dual-Stack Architecture — Compare Python (porting fidelity) vs Rust (production performance)
  • Permission Systems — Three-tier security model gating every tool execution
  • MCP Protocol — Model Context Protocol client with 6 transport types
  • Streaming APIs — Server-Sent Events parsing for real-time AI responses
  • Session Management — Persistence, compaction, replay, and resume patterns
  • Config Hierarchies — Multi-layer resolution (user → project → local)
  • Trait-based Abstraction — Rust generics over ApiClient and ToolExecutor

Permission System (Python)

ToolPermissionContext

@dataclass(frozen=True)
class ToolPermissionContext:
    deny_names: frozenset   # Exact names
    deny_prefixes: tuple    # Name prefixes

    def blocks(self, tool_name: str) -> bool:
        lower = tool_name.lower()
        if lower in self.deny_names:
            return True
        return any(
            lower.startswith(p)
            for p in self.deny_prefixes
        )

How It Works

  • Exact name matching — deny specific tools
  • Prefix matching — deny tool families
  • All comparisons case-insensitive
  • Uses frozenset for immutable deny lists
  • Integrated into filter_tools_by_permission_context()
  • Denials tracked as PermissionDenial objects

ConversationRuntime — The Agentic Loop

// runtime/src/conversation.rs
const DEFAULT_AUTO_COMPACTION_INPUT_TOKENS_THRESHOLD: u64 = 200_000;

// Two core traits for pluggable behavior:
trait ApiClient {
    fn stream(&self, request: ApiRequest) -> Result<Vec<AssistantEvent>>;
}

trait ToolExecutor {
    fn execute(&self, name: &str, input: Value) -> Result<String, ToolError>;
}

// Generic runtime parameterized over both:
struct ConversationRuntime<C: ApiClient, T: ToolExecutor> {
    session: Session,
    api_client: C,
    tool_executor: T,
    permission_policy: PermissionPolicy,
    system_prompt: Vec<String>,
    max_iterations: usize,           // Default: 50
    usage_tracker: UsageTracker,
    hook_runner: Option<HookRunner>,
    auto_compaction_input_tokens_threshold: u64,
}

The Turn Loop

1. User Input → Add to session messages
↓
2. Build ApiRequest (system_prompt + messages)
↓
3. Stream response via ApiClient.stream()
↓
4. Parse AssistantEvents (TextDelta, ToolUse, Usage)
↓
5. For each ToolUse: check PermissionPolicy
↓
6. Execute tool → Append results to messages
↓
7. Loop until MessageStop OR max_iterations (50)

Auto-compaction triggers when input tokens exceed 200K

Key Design Patterns

Registry Pattern

Commands and tools loaded from JSON snapshots into typed registries with lookup, search, and filtering.

Trait-based Generics

ConversationRuntime<C, T> parameterized over ApiClient and ToolExecutor for testability.

Permission-First

Every tool execution gated by three-tier model. No tool runs without permission check.

Session-Centric State

All conversation state organized around a persistent session with compaction and resume.

Config Layering

User → Project → Local precedence with merged output for maximum flexibility.

Parity Auditing

Systematic comparison against reference implementation with coverage metrics.
