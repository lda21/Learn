What Is WebSocket?

Core Concept

  • Full-duplex — both sides send/receive simultaneously
  • Persistent connection — single TCP socket stays open
  • Standardized as RFC 6455 (2011)
  • Protocols: ws:// (unencrypted) and wss:// (TLS)
  • Minimal framing overhead (2-14 bytes per message)

Why It Matters

  • No repeated HTTP headers on every message
  • Server can push data without client polling
  • Sub-millisecond message delivery latency
  • Ideal for real-time: trading, chat, gaming, IoT
  • AI Crypto Agent uses WebSocket for live tick data from Binance, Coinbase, Kraken

WebSocket in the Browser

// JavaScript WebSocket API
const ws = new WebSocket("wss://stream.binance.com:9443/ws/btcusdt@trade");

// Connection opened
ws.onopen = () => {
  console.log("Connected to Binance trade stream");

    method: "SUBSCRIBE",

};

// Receive messages
ws.onmessage = (event) => {
  const trade = JSON.parse(event.data);
  console.log(`${trade.s}: ${trade.p} @ ${trade.q}`);
  // BTC/USDT: 67432.50 @ 0.015
};

// Handle errors and closure

  console.log(`Closed: code=${event.code} reason=${event.reason}`);

};

// Send data anytime
ws.send(JSON.stringify({ method: "PING" }));

// Clean up
ws.close(1000, "Done");  // 1000 = normal closure

WebSocket Servers

Node.js (ws)

import { WebSocketServer } from "ws";

const wss = new WebSocketServer({ port: 8080 });

wss.on("connection", (ws, req) => {
  console.log("Client connected");

  ws.on("message", (data) => {
    // Broadcast to all clients
    wss.clients.forEach((client) => {
      if (client.readyState === 1) {
        client.send(data);
      }
    });
  });

  ws.on("close", () =>
    console.log("Client disconnected"));
});

Python (websockets)

import asyncio
import websockets

connected = set()

async def handler(ws):
    connected.add(ws)
    try:
        async for message in ws:
            # Broadcast to all
            await asyncio.gather(
                *[c.send(message)
                  for c in connected
                  if c.state == 1]
            )
    finally:
        connected.discard(ws)

asyncio.run(
  websockets.serve(handler, "0.0.0.0", 8080)
)

                await self.tick_queue.put(tick)     # Feed to Kafka

Connection Management

Heartbeat / Ping-Pong

WebSocket protocol has built-in ping/pong frames. Server sends ping, client auto-replies pong. Detects dead connections in seconds.

ws.ping() → client pong

Application-Level Keepalive

Exchanges require app-level pings too. Binance disconnects after 10 min without a pong. The agent sends keepalive every 30s.

{"method": "PING"}

Connection Pooling

Binance limits 5 messages/sec per connection. The agent uses multiple connections per exchange, partitioned by symbol groups.

3 exchanges x 4 connections = 12 WS

Health Monitoring

Track last message timestamp per connection. If no data for 60s on an active market, flag connection as stale and reconnect.

if now - last_msg > 60s: reconnect()

Reconnection Strategies

Exponential Backoff + Jitter

class ReconnectPolicy:
    BASE_DELAY = 1.0    # seconds
    MAX_DELAY  = 60.0
    MAX_RETRIES = 20

    async def connect_with_retry(self, url):
        for attempt in range(self.MAX_RETRIES):
            try:
                ws = await websockets.connect(url)
                self.attempt = 0  # reset on success
                return ws
            except Exception as e:
                delay = min(
                    self.BASE_DELAY * (2 ** attempt),
                    self.MAX_DELAY
                )
                # Add jitter to avoid thundering herd
                jitter = random.uniform(0, delay * 0.3)
                await asyncio.sleep(delay + jitter)
        raise MaxRetriesExceeded(url)

Circuit Breaker Pattern

  • CLOSED — normal operation, attempts connect
  • OPEN — too many failures, stop trying for cooldown period
  • HALF-OPEN — allow one test connection, promote to CLOSED on success

The AI Crypto Agent uses circuit breakers per exchange. If Kraken is down, Binance and Coinbase continue unaffected.

Authentication & Security

Token-Based Auth

Pass JWT or API key in the initial HTTP upgrade request. WebSocket has no built-in auth mechanism.

wss://api.exchange.com/ws?token=eyJ...

Always Use wss://

TLS encryption prevents eavesdropping on tick data and trading signals. Never use plain ws:// in production.

wss:// = WebSocket + TLS (port 443)

Origin Validation

Server checks the Origin header during handshake to reject connections from unauthorized domains.

Origin: https://your-app.com

Rate Limiting

Protect against abuse: limit messages per second, max connections per IP, and payload size caps.

Binance: 5 msgs/sec, 1024 connections

The AI agent authenticates to private exchange endpoints using HMAC-signed API keys passed in the upgrade request headers.

Scaling WebSocket Servers

Challenges

  • WebSocket connections are stateful — tied to a specific server
  • Load balancers must use sticky sessions
  • Broadcasting requires cross-server coordination
  • Each connection holds memory (~10KB per socket)
  • 100K connections = ~1GB RAM just for sockets

Solutions

  • Redis Pub/Sub — fan out messages across server instances
  • NATS / Kafka — durable message bus for cross-server broadcast
  • Consistent hashing — route users to stable servers
  • Connection draining — gracefully migrate on deploy
  • AWS ALB / Nginx — WebSocket-aware load balancers

Pod Failover & Data Consistency

What happens when a pod dies and the client reconnects to a different pod?

Client
↔
Pod 1
× dies
→
Pod 2
reconnects here

What the Client Sees

  • YES — connection drops. The TCP socket dies. The client gets an onclose event. This is not transparent.
  • The load balancer does NOT magically hand the connection to another pod — WebSocket is a persistent TCP connection to one specific pod
  • Client must reconnect (new handshake, new TCP socket, new pod assignment by LB)
  • During reconnect: messages are lost. Anything sent while disconnected is gone.

Data Loss Window

Timeline:
  T0: Pod 1 healthy, streaming data
  T1: Pod 1 crashes
      ← messages lost here
  T2: Client detects disconnect (~5-30s)
  T3: Client reconnects to Pod 2
  T4: New stream starts

Gap: T1 → T4 = 5-60 seconds of lost data

Solution 1: Sequence Numbers

Server tags every message with a sequence ID. Client sends last_seen_seq=4582 on reconnect. New pod replays from 4583 onwards. Requires server-side message buffer (Redis, Kafka).

Solution 2: Shared State

Don't keep state in pods. Store all session state in Redis. Any pod can serve any client. Pod 2 reads client state from Redis and resumes. Pod is stateless, Redis is the source of truth.

Solution 3: Event Bus Replay

All messages flow through Kafka with retention. On reconnect, consumer replays from the offset where the client disconnected. Zero data loss if offsets are tracked.

Solution 4: Client-Side Recovery

On reconnect, client calls a REST snapshot endpoint to get current full state, then resumes the WebSocket stream. Simple, no server-side replay needed. Works for trading dashboards.

// Client-side: reconnect with recovery
ws.onclose = () => {
  // 1. Fetch full state snapshot via REST
  const snapshot = await fetch('/api/state?since=' + lastSeqId);
  applySnapshot(snapshot);

  // 2. Reconnect WebSocket with last known sequence
  const ws = new WebSocket(url + '?resume_from=' + lastSeqId);

  // 3. Server replays missed messages, then continues live stream
};

How SSE Last-Event-ID Works

# Server sends events with IDs:
id: 4582
data: {"price": 50123.45}

# Browser sends header:  Last-Event-ID: 4583

# Server (Node.js) — read the header, replay missed events:
app.get('/stream', (req, res) => {
  const lastId = req.headers['last-event-id'];
  if (lastId) {
    const missed = await redis.getEventsAfter(lastId);  // replay from store
    missed.forEach(e => res.write(`id: ${e.id}\ndata: ${e.data}\n\n`));
  }
  // Continue live stream...
});

Key insight: SSE is better than WebSocket for pod failover — recovery is part of the spec. Use SSE for server→client streaming (prices, notifications). Use WebSocket only when you need bidirectional communication.

Data Quality Filtering

Garbage in, garbage out — the agent filters bad ticks before they reach the trading engine

Stale Quote Detection

Reject quotes older than 30 seconds. Exchange WS can replay old data during reconnection. Compare tick.timestamp vs now().

if now - tick.ts > 30s: drop

Crossed Quote Detection

Bid price should never exceed ask price. Crossed quotes indicate stale or corrupted data from one leg of the book.

if bid > ask: flag_crossed()

Flash Move Filter (>5σ)

A price move exceeding 5 standard deviations from the rolling mean is likely a data error or flash crash. Hold for confirmation.

if abs(price - mean) > 5 * std: quarantine

Duplicate Detection

Exchanges can send duplicate trade IDs during reconnection. Track seen IDs in a bounded set to dedup.

if tick.trade_id in seen: skip 
# Data quality pipeline in the AI Crypto Agent
class TickFilter:
    def is_valid(self, tick: Tick) -> bool:
        if time.time() - tick.timestamp > 30:     return False  # stale
        if tick.bid and tick.ask and tick.bid > tick.ask: return False  # crossed
        if self._is_flash_move(tick):              return False  # >5 sigma
        if tick.trade_id in self.seen_ids:         return False  # duplicate
        self.seen_ids.add(tick.trade_id)
        return True

Server-Sent Events (SSE)

When SSE Is Better

  • Server-to-client only — news feeds, notifications, dashboards
  • Automatic reconnection built into the browser API
  • Works over standard HTTP — no proxy issues
  • Simpler server implementation (just text/event-stream)
  • Event IDs enable resumption from last received event

SSE API

// Browser
const es = new EventSource("/api/prices");
es.onmessage = (e) => {

  updateDashboard(data);

// Server (Node.js)
app.get("/api/prices", (req, res) => {
  res.writeHead(200, {
    "Content-Type": "text/event-stream",
    "Cache-Control": "no-cache",
    "Connection": "keep-alive",

  const interval = setInterval(() => {

      getLatestPrice()
    )}\n\n`);

    clearInterval(interval));

Rule of thumb: Use SSE for one-way server push. Use WebSocket when the client also needs to send data.

Socket.IO & Libraries

Socket.IO

Most popular WebSocket library. Auto-reconnect, rooms, namespaces, HTTP long-polling fallback. Node.js + browser.

io.to("room1").emit("tick", data)

SignalR (.NET)

Microsoft's real-time framework. WebSocket, SSE, and long-polling transports. Hub-based RPC model with strong typing.

hub.Clients.All.SendAsync("tick", data)

Phoenix Channels (Elixir)

Built on Erlang/OTP for massive concurrency. 2M+ connections per server. Presence tracking built-in.

channel.push("tick", data)

ws (Node.js)

Lightweight, bare-metal WebSocket. No abstraction overhead. The AI Agent's React dashboard uses ws for raw speed.

npm install ws (zero dependencies)

Choose wisely: Socket.IO for rapid prototyping with fallbacks. Raw ws or native WebSocket for performance-critical systems.

// Custom React hook for WebSocket connection
function useTickStream(symbol: string) {
  const [tick, setTick] = useState<Tick | null>(null);
  const wsRef = useRef<WebSocket | null>(null);

  useEffect(() => {
    const ws = new WebSocket(`wss://api.agent.com/ws/ticks/${symbol}`);
    wsRef.current = ws;

    ws.onmessage = (event) => {
      const data: Tick = JSON.parse(event.data);
      setTick(data);  // Triggers re-render with latest price
    };

    ws.onclose = () => {
      // Auto-reconnect with exponential backoff
      setTimeout(() => wsRef.current = reconnect(symbol), getBackoff());
    };

    return () => ws.close();  // Cleanup on unmount
  }, [symbol]);

  return tick;
}

Best Practices

1. Always use wss:// in production (TLS encrypted)
2. Implement heartbeat/ping-pong to detect dead connections
3. Use exponential backoff + jitter for reconnection
4. Validate and sanitize all incoming WebSocket messages
5. Set max message size limits to prevent memory attacks
6. Use circuit breakers for external WebSocket dependencies
7. Normalize data from multiple sources into a unified format
8. Filter data quality issues (stale, crossed, flash moves)
9. Monitor connection count, message rate, and latency metrics
10. Gracefully drain connections during deployments
